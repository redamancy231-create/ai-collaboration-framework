## 维度1：逻辑自洽性

1. §3.7.0 三条规则之间存在未定义接口，当前不可认为自洽。

   规则1“历史频率抑制”如果只是抑制 prompt 注入、仍保留原始事件日志，则不会阻断规则2“修复循环检测”。但草案没有定义“抑制”的层级：是抑制输入给 AI、抑制事件生成、还是抑制日志记录？如果重复信号在抑制期不再进入 `events.jsonl`，规则2 可能看不到连续 repair，修复循环检测被规则1遮蔽。

   更严重的是，规则2“自动注入 explore 信号”和规则1“自动抑制重复注入”都改变执行策略，不只是观测。当前主框架 §3.7.1 明确要求漂移检测层“只观测，不阻断”，唯一例外是连续红级告警触发提前 HG-1。草案把前置信号过滤器设计成自动干预器，和现有 §3.7 的边界冲突。

2. §9.6 的 500 字符上限与 §3.7.0 的 8 次历史事件不必然矛盾，但草案把两个层级混在一起。

   如果 500 字符限制只约束“注入上下文的经验摘要”，而 `events.jsonl` 保留完整结构化事件，则没有实现矛盾。若 500 字符限制也约束事件本体，则会损害 `intent`、`blast_radius`、原始信号、结果等检测字段。

   真实问题不是 8 × 500 的长度，而是草案没有区分“规范存储层”和“上下文注入层”。REO/Event 应该可以完整存档；进入 prompt 时再生成 ≤500 字符的 compact view。

3. §7 的 REO Schema 与 §9.6 的结构化编码要求不一致，Gene/Capsule/Event 闭环也还只是叙事。

   §9.6 要求经验/信号 JSON 至少包含 `signal`、`confidence`、`context`、`source`。REO Schema 却要求 `hypothesis`、`test`、`result`、`validation_rule`、`epigenetic_mark` 等字段，且 `signals` 只是空数组，没有定义元素 schema。两者可以兼容，但需要明确：REO 是持久化对象，§9.6 是 REO 的注入视图。目前没有这个映射。

   Gene/Capsule/Event 的闭环缺少晋升规则：多少次 Event 成功才固化为 Capsule？Capsule 何时提炼为 Gene？失败如何进入 anti-pattern？冲突 Gene 如何裁决？没有这些规则，闭环不是反馈系统，只是三类文件夹加一张流程图。

4. §3.7.3 表观遗传权重与 §3.7.0 的接口不清晰，而且存在一个直接的符号错误。

   草案写“失败时：boost -= `delta_failure`（默认 -0.15）”。按这个公式，失败会使 boost 增加 0.15。应改为 `boost += delta_failure` 且 `delta_failure = -0.15`，或 `boost -= failure_penalty` 且 `failure_penalty = 0.15`。

   当前主框架的 §3.7.3 是“信号聚合与告警规则”，规则退役在 §3.7.4。草案声称“§3.7.3 原内容是静态规则退役”与本地最新版主文档不符。这不是小编号问题，而是说明草案基于过期结构生成，不能直接合并。

## 维度2：可操作性

1. “2-3 小时实现”只覆盖脚本雏形，不覆盖可用系统。

   如果已有稳定的 `events.jsonl`，写一个规则扫描脚本确实可能 2-3 小时完成。但当前主框架的漂移输出是 `DriftDetectionReport` 模板，并未定义 Evolver 式事件流。要真正可用，还需要事件 schema、日志落点、字段填充责任、冷启动行为、告警接入、人工复核记录。

   “10+ 条事件”在实践中可能是半天，也可能是数周，取决于一个项目的 Loop 粒度。冷启动期间草案只说“宽松阈值”，没有说明是否禁用自动抑制、是否只记录不告警、是否允许人工 override。冷启动退化行为不可评估。

2. §9.6 的 A/B 测试设计太粗，不能支撑原则正式化。

   目前只定义了 Arm A/B、任务和指标，没有样本量、重复次数、模型版本固定、随机种子、数据切分、prompt 控制、评分口径、最小有意义效应、失败判据。按这个设计跑出来的结果很容易只是提示词偶然性。

   如果用户不做 A/B 测试，§9.6 仍可以作为“上下文预算纪律”，但不应写成已验证原则。尤其“结构化编码至少不会比自然语言更差”没有证据。JSON 可能丢失语义、压缩掉反例、让模型过度机械服从错误字段。

3. Phase 1 手动创建 REO 的可操作性不足。

   草案给的是示例对象，不是正式 JSON Schema。没有 `$schema`、required、enum、字段类型、版本号、兼容策略、示例模板、验证命令。手动创建时，人和 AI 都很容易写出看似 JSON、实则不可比较的对象。

   手动 REO 与自动 REO 要兼容，至少需要 `schema_version`、`created_by`、`source_artifact`、`pre_registration_id`、`validation_status`、`compact_view` 等字段或派生规则。当前 schema 不足以支撑 Phase 2/3。

4. `delta_success = +0.05` 和 `delta_failure = -0.15` 的依据不足。

   Evolver 综合报告称这些数值来自 Evolver 的 epigenetic suppression 机制，但 Qwen 审查也确认核心 `epigenetics.js` 是混淆代码。也就是说，这些数值最多是“来自上游分析报告的转述”，不是本次草案可独立验证的公开源码事实。

   即使数值确实来自 Evolver，也不能直接迁移到量化研究。失败两次即触发 -0.3 硬抑制，成功六次才抵消两次失败，这种非对称在金融策略中可能过度惩罚高方差但正期望的探索。应标为可配置初值，并要求敏感性分析。

## 维度3：来源忠实性

1. “所有 Evolver 来源思想已经过去隐喻化处理”是错误声明。

   草案仍保留 `Gene`、`Capsule`、`Event`、`epigenetic_mark`、表观遗传权重、进化周期等术语。这不是去隐喻化，而是继续沿用 Evolver 的生物学叙事。若目标是金融/工程术语，应改为“策略规则 / 成功快照 / 审计事件 / 自适应权重 / 迭代周期”等。

2. 从 Gemini 科学代码求解推广到量化研究，没有经验支持。

   Evolver 的主要证据来自科学代码求解场景和 Gemini 3.1 系列模型。量化研究面对的是非平稳市场、数据泄漏、过拟合、样本外退化、交易成本、因子拥挤等不同问题。把 epigenetic suppression 用到量化研究可以作为假设，但不能作为已验证方法。

3. §9.6 对“4,590 次对照试验”的警示不足。

   草案虽然写了“限于科学代码求解场景和 Gemini 3.1 系列模型”，但遗漏了更关键的限定：Qwen 指出 4,590 更接近 retained analyses，不是 4,590 个独立随机对照实验；原报告指出无置信区间、无 p 值、无多重比较校正、公开仓库缺实验配置和原始数据。

   因此草案不能把“4,590 次对照试验表明”作为强背书。更准确的写法是：“Evolver 论文报告的 retained analyses 暗示紧凑结构化表示可能有优势，但证据受模型、任务、统计和可复现性限制。”

4. 草案夸大了 Evolver 的方法论价值。

   Evolver 综合评分 4.2/10，Qwen 替代综合评分 4.1/10。两者收敛的是“该项目整体质量偏低”，不是“草案新增机制已经被双重验证”。Qwen 还给原分析的方法论可靠性 6.5/10，并指出确认偏误、煽动性语言、动机归因过度。

   草案顶部和结尾反复强调“DeepSeek + Qwen 双重验证”“高度收敛”，会制造不应有的权威感。正确表述应是：Evolver 项目质量低，部分思想有潜在可移植性，但需要本框架内独立试跑验证。

## 维度4：完整性与边界

1. “不采纳项”大体方向正确，但遗漏了可吸收的低风险思想。

   不采纳 EvoMap Hub 网络、默认联网验证、进化循环 daemon 是合理的，理由是隐私、中心化依赖、隐形资源消耗和过度工程化。

   但草案没有充分吸收 Qwen 指出的正面点：token 成本效率、跨运行时集成经验、结构扰动鲁棒性测试、默认验证器的隐私风险教训。尤其“成本效率”比“表观遗传”更容易低风险迁移到框架中。

2. 信号熵检测层与现有 §3.7 漂移检测层边界不清。

   现有 §3.7 已有绩效信号、发现率衰减、规则退役、宪法审计、闭环策展和四色告警。草案再引入 `signal_entropy` 的 0.2/0.5/0.8 阈值，会形成第二套告警体系，与现有绿/黄/橙/红聚合规则并行。

   如果保留，§3.7.0 应只是“事件流健康度”输入源，输出一个或多个普通监测项，交给现有 §3.7.3 聚合。不要新增独立告警等级，更不要自动改变 Loop 行为。

3. REO 与 kill-test-first 的协作关系只写了字段映射，没写权责边界。

   `kill_condition -> validation_rule` 是合理方向，但 kill-test-first 的预注册应是实验前的不可变承诺，REO Event 是实验后的审计记录。草案没有定义谁是 source of truth，可能导致预注册和 REO 各自保存一份判定条件，之后发生漂移。

   建议 Event 型 REO 引用预注册 ID 和不可变快照，而不是复制一套可自由编辑的 `validation_rule`。

4. 草案遗漏了关键实施前验证步骤。

   至少应补充：历史日志 dry-run、只观测模式试跑、误报率和人工耗时统计、§3.7 退出条件对接、JSON Schema validator、数值敏感性分析、版本编号迁移检查、术语去隐喻化检查、A/B 测试最小样本设计、以及“自动抑制/自动 explore”禁用开关。

## 维度5：对抗式审查

1. 最致命的反驳成立：当前草案确实有“洗白低质量项目”的风险。

   低质量项目也可能产生好思想，这一点可以成立。但草案的问题是把“潜在可移植启发”写成“经双重验证的方法论”。Evolver 的 4.1/4.2 综合评分、不可复现、混淆核心代码、统计薄弱，决定了它只能作为假设来源，不能作为方法论背书。

2. 过度工程化风险高。

   §3.7.0 + §3.7.3 + §7 组合后，需要事件日志、信号熵脚本、REO 存储、boost 状态、context-aware 权重、自动抑制、自动 explore、Gene 推荐、冲突检测。主框架已经是重型方法论文档，继续加入这些机制会把“协作指南”推向“专门工具链”。

   最小可接受路径应是：先把信号去重做成只观测 dry-run；§9.6 改成经验注入格式建议；REO 和自适应权重推迟到有真实日志后再设计。

3. 独立审查质量问题确实污染了草案的表述。

   Qwen 并没有无条件背书原报告。它指出原报告存在中等确认偏误、煽动性语言、方法论价值评分与综合评分张力，并给原报告方法论可靠性 6.5/10。草案却选择性引用“综合评分 Δ0.1 高度收敛”和“方法论价值 7.0/7.5”，弱化了这些警告。

   这是一种二次确认偏误：只吸收独立审查中支持采纳的部分，不吸收其对来源分析过程的警告。

4. §9.6 有稻草人和过度包装风险。

   “结构化编码可能优于冗长自然语言”是温和、合理、需验证的主张。草案把它包装成“紧凑表示原则”，再加 500 字符硬上限和“至少不会更差”的判断，证据强度不够。

   更稳妥的名称是“经验注入的上下文预算规则”。它应只约束注入片段，不约束原始记录、解释文档、审查报告和复杂决策理由。

## 总体评估

- 综合评分：4.3/10
- 是否建议合并进框架主文档：驳回当前草案；重写后可重新审查
- 最关键的 1-3 条修改建议：
  1. 先做版本对齐：当前主文档已是 v1.5，已有 §9.6；§3.7.3 也不是规则退役。必须重新编号，并恢复与现有“只观测，不阻断”边界一致。
  2. 把所有自动干预降级为 dry-run 观测和 HG-1 建议；定义事件 schema、`signal_entropy` 公式、冷启动行为、REO compact view、Gene/Capsule/Event 晋升规则。
  3. 去掉 Evolver 权威背书和生物学术语，把所有来源声称按当前主框架 §9.6 的证据分类标注为 [I]/[J]/[Sp]，并要求本框架内试跑验证后再升级。
- 草案中最有价值的部分：
  - “修复循环/重复信号”作为事件流健康问题值得保留。
  - 经验注入应控制长度、结构化、可解析，这个方向有实践价值。
  - 用累积表现替代简单“长期未触发就退役”的直觉是有价值的，但需要重做接口和数值依据。
- 草案中最薄弱的部分：
  - 对现有框架版本和章节结构的适配失败。
  - 把检测层设计成自动策略干预器，违反 §3.7 的核心约束。
  - 对 Evolver 来源证据的校准不足，尤其 4,590 retained analyses、Gemini 单模型家族、核心代码混淆和 Qwen 对原分析偏误的警告没有被充分吸收。
