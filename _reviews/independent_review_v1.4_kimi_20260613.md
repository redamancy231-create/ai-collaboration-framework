## 一、角色 A：方法论纯度审查官

### 评分：5.0 / 10

**核心判断**：v1.4 的新增概念有真实价值，但“诚实降级”做得还不够彻底；部分新内容仍被放在不应在的层级，术语体系整体**略变模糊**。

#### 1. “难度分层漂移”的定位是否诚实？

- **文本层面是诚实的**：§3.7.2.6 标题已明确写成“ML 项目可选 profile，非独立第六类”，并声明与绩效信号存在概念重叠。
- **实现层面仍不一致**：
  - §3.7.2 总标题仍叫“漂移信号（五类核心 + 一个可选 profile）”，但 §3.7.7 监测模板 `signal_summary` 的 schema 只给了五类核心信号的子块，没有为 difficulty profile 预留字段，与文中“五类核心 + 可选 profile 各一个子块”矛盾。
  - §3.7.8 “已知限制”仍写“五类信号的触发阈值……未经验证”，未说明可选 profile 的阈值也未经验证。
- **与绩效信号的关系**：难度分层漂移本质上是“绩效信号的分层视图 + 数据难度分布偏移”。把它与句法/语义/程序/对齐/绩效并列，即使加了“可选 profile”标签，仍然让传感面从通用概念滑向 ML 特例集合。当前定位**半诚实**——口号降级了，结构还没完全降级。

#### 2. “能力获取 vs 行为塑造”是有效二分还是过度泛化？

- 这是一个**单案例（LCPO）驱动的概念标签**。表格的 6 个维度区分清晰，但现实中大量任务处于中间态：既要求行为改变，也要求能力保持；或者不同子任务分别属于两类。
- 框架自己也承认这是“概念标签，非新增任务类型分档”，这是正确的降级。但把它放在 §5.3.1（Loop 收敛条件）下，容易让读者误以为它会影响收敛规则——实际上“现有三种任务类型（探索/执行/修复）的收敛条件不变”。
- **结论**：作为脚注或附录提示合适；放在主流程节下，有把单个案例升格为通用框架术语的风险。

#### 3. 附录 H “候选模式库”是否过早？

- **结构过早**：目前仅 2 个条目，且抽象层级不齐——H.1 是方法模式（优化目标分解/干扰项剥离），H.2 是工具检查（import 完整性）。一个“库”至少需要同一抽象层级的 ≥5 条跨项目模式。
- **重复**：H.2 与 §9.1 的 Import 完整性自检几乎完全重复。
- 框架在附录 H 开头已自注“目前仅 2 条目，待 ≥5 条跨项目模式后升级”，这是诚实的，但**仍然不应该给出一个只有 2 个条目的结构以正式附录编号**。它更适合作为 §14 变更记录中的一条“候选模式待孵化”注记，或合并进 Small_Scale 转化分析文档。

#### 4. 术语体系在 v1.4 中更精确还是更模糊？

- **更精确的部分**：`version_binding`、`evidence_level`、`reproducibility_precheck`、`训练-评估配置对齐`——这些是从真实失败中提炼出的可验证术语。
- **更模糊的部分**：
  - “接口退化识别”（§6.2 模式 7）实际上是一个**代码接口设计规则**，却被挂在 Workflow 层的 Creator-Verifier 编排模式下面，抽象层级错位。
  - “候选模式库”把不同抽象层级的条目混为一谈。
  - “能力获取/行为塑造”作为新二元术语，尚未被外部文献或更多案例验证，容易变成框架内部的“黑话”。
- **净评估**：精确术语的增加被层级错位和过早抽象抵消，**整体略模糊**。

## 二、角色 B：一线工程师

### 评分：6.0 / 10

**核心判断**：v1.4 的新增检查项大多数**真的能防错**，但“能不能落地”取决于它们是否被 profile 化、是否有自动化工具支撑；框架体积增长在可接受范围内，但同步失败会降低信任。

#### 1. 新增检查项是防错利器还是填表负担？

| 新增项 | 价值 | 成本 | 评价 |
|---|---|---|---|
| 训练-评估配置对齐（§9.1） | 高：cutoff_len/max_tokens 不一致是 ML 实验常见且隐蔽的坑 | 中：每次实验前需比对方案/代码；可自动化 | 值得做，建议做成 checklist 或脚本 |
| Import 完整性自检（§9.1 / 附录 H.2） | 中-高：发布前跑一次静态分析能 catch 类似 Small_Scale `utils/utils.py` 缺失的问题 | 低：一次 `ast` 扫描即可 | **但目前只有“工具思路”，没有实际脚本/skill交付**；若没有工具，就是填表负担 |
| 可复现性预检（kill-test 6.5） | 高：在 cheap test 运行前确认代码/参数/版本绑定，避免“不可复现的 cheap test 给出假信号” | 中：启动新项目时多问 4 个问题；但**不阻断运行**，只降级可信度 | 可接受，但应仅限研究/量化/ML/数据管线类新方向默认启用 |
| 配置 schema 验证（§9.1） | 中：对复杂项目启动时验证路径/参数 | 低-中：一次 Pydantic 模型 | 合理，但已明确是“建议”而非强制 |

#### 2. kill-test-first 从 8 步变成 8.5 步——我会逐条过吗？

- **不会**，但 skill 本身也**没有要求你逐条过所有事**。
- skill 明确区分：
  - **应走完整协议**：新研究想法 / 新量化策略 / 新论文选题 / 新模型或架构方案。
  - **不应触发**：修 bug、复现实验、补测试、已批准项目内的常规实验。
- 对后者只保留“轻量提醒”——确认可证伪命题 + 问一句“要不要走完整 kill-test”。
- **风险**：如果用户把 6.5 的“可复现性预检”套到每一个小实验上，确实会变成负担。建议进一步明确：6.5 默认只在涉及**数据/代码/论文可复现性**的新方向上启用，普通代码小改可跳过。

#### 3. 从 2177 行涨到 2313 行——更想读还是更不想读？

- 多出来的 136 行大部分是**可选 profile / 案例 / 附录**，核心流程没有等比例膨胀。
- §1.4 的使用强度分档（最低强制版 / 默认版 / 完整版）已经做了很好的分压——最低强制版仍然很薄。
- **但是**：附录 H、§3.7.2.6、§6.2 接口退化等新内容被放在显眼位置，会给人一种“框架又变厚了”的印象。建议把 ML profile、代码设计规则等内容更明确地标记为“可选 / 仅在 X 场景启用”，并压缩重复案例。

#### 4. “难度分层漂移”需要标注所有测试数据难度——是死规矩吗？

- **不是死规矩**。§3.7.2.6 明确说“仅在项目涉及模型训练/评估且存在按难度分层的测试数据时激活”。
- 对没有难度标签的项目，这条规则**自然不适用**。这是正确的 profile 化写法。
- 改进建议：在 §3.7.2.6 开头再加一句“如果你的项目没有现成的难度分层标签，不要为本条单独造分层——先使用 §3.7.2.5 绩效信号”。

#### 5. 零成本 vs 持续成本

**零成本（读一遍即可，无需每次执行）**：
- §5.3.1 “能力获取 vs 行为塑造”概念标签。
- §6.2 “接口退化识别”概念（知道有这种风险即可）。
- §6.5.1 文件系统作 IPC 的反面模式（知道别这么干）。
- §9.4 嵌入式依赖维护陷阱（知道别直接嵌入大仓库）。

**持续成本（每次都要检查/填/汇报）**：
- §9.1 训练-评估配置对齐（每次 ML 实验前）。
- §9.1 Import 完整性自检（每次发布/归档前）。
- kill-test 6.5 可复现性预检（每次走完整 kill-test 时）。
- §3.7.2.6 难度分层漂移（如果项目有难度分层，每次评估都要看）。
- 版本绑定 / evidence_level 标注（每次独立审查 / 发布）。

**结论**：持续成本项都是高价值防错项，但必须有自动化工具或 checklist 支撑；否则很容易流于形式。

## 三、角色 C：安全/治理审计官

### 评分：4.0 / 10

**核心判断**：v1.4 在**姿态上**很诚实，但在**元审查执行**上严重失分——一个强调 version binding、evidence_level、自反审查的框架，自己的 Markdown/JSON/SOP/清单却没有同步。这是核心承诺未兑现。

#### 1. 局限性和独立性状态是否诚实标注？

- **诚实的地方**：
  - §1.6 明确登记 OPEN-1~5 为“已知待决项”，并说明 OPEN-1 仍缺人类专家 verify。
  - §14 明确声明“框架 v1.4 的所有修改由 DeepSeek-V4-Pro 执行”，并列出 ChatGPT-5.5 两轮独立审查。
- **不诚实 / 不完整的地方**：
  - 未披露**审查提示词由作者（DeepSeek-V4-Pro）撰写**。`codex_review_v1.4_prompt.md` 第一行即写“我是 DeepSeek-V4-Pro……”，且整份提示词由作者设计维度、给出变更摘要、框定讨论范围。
  - 按框架自身 `methodological-review-sop.md` §2.1 / §3.2 及 `meta-audit-checklist.md` I-004 的标准，作者撰写审查 rubric 会引入引导性偏差，审查独立性应降级为 **[SEMI]**（后端不同但 rubric 不纯净），而非 §14 所写的“独立后端+隔离上下文”的 [IND]。
  - 该偏差方向：提示词中作者已预设“v1.4 变更是对 Small_Scale 教训的正确转化”，并主动提供 10 条变更摘要，可能让审查者从作者框架出发而非从零质疑。

#### 2. ChatGPT-5.5 两轮审查的独立性声明是否准确？

- **后端独立**：ChatGPT-5.5（GPT-5）与 DeepSeek-V4-Pro 后端不同——这一点准确。
- **上下文隔离**：Codex CLI 与 Claude Code CLI 不共享 memory/CLAUDE.md——这一点也大致成立。
- **但 rurbic/prompt 不独立**：审查提示词由作者撰写，且包含“变更摘要”“审查维度”等引导性内容。按框架自己的 SOP，这属于 **C8 rubric 纯净** 未通过。
- **因此**：独立性级别应为 **[SEMI]**（半独立，二级证据），而非 [IND]。§14 应补充披露：“两轮审查的提示词由作者设计，审查者在作者给定的维度下展开批评；独立性因此降级为 SEMI。”

#### 3. 框架自身是否遵守了 version binding / evidence_level 要求？

- **没有**。这是 v1.4 最严重的自反性失败：
  - **主文档**：头部版本元数据已写 v1.4，但 §14 标题仍为“变更记录（v1.1 → v1.3）”，目录也写“v1.1 → v1.3”。
  - **主 JSON**：`metadata.version` 是 1.4，但 `framework_self_governance.open_items.OPEN-1.status` 仍为“已覆盖 → §3.7”，落后于 Markdown 的“已操作化，待试跑验证”；JSON 中未同步 §3.7.2.6 难度分层 drift profile。
  - **meta-audit-checklist.md**：元数据仍写“v1.0.1 / 适用范围 v1.3”，虽然正文已改为 67 项；审计日志模板仍写“维度二 11 项 / 维度四 18 项 / 总计 64 项”。
  - **meta-audit-checklist.json**：`metadata.scope` 仍是“v1.3”，`version_note` 仍是 ChatGPT-5.5 对 v1.3 的修订说明（M8/M9），未反映 v1.4 新增 I-012 / B-019 / B-020。
  - **methodological-review-sop.md**：正文已新增 C10 / A9，但 §9.2 操作性附件描述仍写“A1-A8”；附录修订历史只有 v1.0。
  - **methodological-review-sop.json**：`metadata.version_note` 仍是旧版 M6/M7/L10 说明；JSON schema 示例中的 `sop_version` 仍为 `v1.0`。
- **元审查结论**：若用框架自己的 `meta-audit-checklist.md` 审框架自身，至少 P-018（MD/JSON 一致性）、I-010（独立性标注透明度）、B-018（生成文件规范同步）等项会 FAIL。一个高喊 version binding 的框架，自己却未做到 version binding——这是核心信誉损伤。

#### 4. “审查前置”模式是否比 v1.3 的“审查后置”更好？

- **优势**：在编辑前就审查转化建议，可以在概念落地前拦截错误归类（如 T1 不应放入逃生口、T3 不应成为强制规则）。这比写完文档再返工更经济。
- **新盲点**：
  - **选择偏差**：审查者看到的“15 条转化建议”是作者从原始 22 findings 中筛选、重排、定优先级后的版本。审查者没有机会质疑“哪些教训根本不应该进入框架”。
  - **作者可事后绕过**：虽然最终产物也经审查，但中间编辑过程由作者控制，某些第一轮被否掉的建议可能以改写形式悄悄回流。
  - **提示词污染**：最终产物审查的提示词由作者撰写并附带变更摘要，审查者可能已经被锚定到作者的叙事框架。
- **结论**：审查前置**不是**比后置更好，而是**互补**。真正安全的做法应是：先审查原始 findings → 再审查转化建议 → 再审查最终产物，且每一步的 rubric 由独立第三方撰写。

#### 5. OPEN-1（零卷入人类专家 verify）状态如何？

- **缺口没有缩小，反而略有扩大**：
  - OPEN-1 的核心问题是“连续漂移 / Human Gate 频率-覆盖缺口”。v1.3 用 §3.7 漂移检测层做了操作化，但仍是“待试跑验证”。
  - v1.4 又新增了 §3.7.2.6 难度分层漂移 profile——它自己也声明“待试跑验证其噪音/价值比”。这意味着 OPEN-1 的未验证监测面**变大**了。
  - 框架仍未安排零卷入人类专家 verify。§3.6 已明确说“最需要的不是再加一轮同谱系 AI 审查，而是一个零卷入的人类来裁决方向之争”，但 v1.4 没有给出任何获取人类专家审查的计划或时间表。
- **建议**：把 §3.7.2.6 明确标为“OPEN-1 下的一个待验证子项”，并在 §1.6 中为 OPEN-1 增加一条 Action：“在 2026-07 前安排至少一位未参与框架设计的人类专家，对 drift detection 层的方向进行裁决。”
