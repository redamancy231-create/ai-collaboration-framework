## 五、最终修改清单（按优先级）

### P0 — 必须改（3 项，三模型收敛）

**1. 增加混合策略规则**
```
- **混合策略**：同一任务可组合 grep 和 LSP。典型模式：grep 批量收集候选文件/行号 → LSP 对候选精确验证定义/引用/类型。透明度规则中分别列出即可。语义级操作不受文件量阈值约束
```

**2. 解决必须 LSP 与倾向 grep 优先级冲突**
- 在"必须 LSP"中明确：**"必须 LSP"的语义需求优先于"倾向 grep"的文件量条件**
- 在"倾向 grep"中追加：**"当任务同时属于必须 LSP 类别时，此条不适用——语义准确性优先于调用效率"**

**3. 修正透明度规则（歧义 + 位置 + 混合 + 回退）**
```
- **透明度规则**：每次代码理解任务的最终答复开头，保留一行工具选择说明：`[工具] LSP:goToDefinition ×3 + Grep ×1 | 理由: 跳转定义需语义(3处)；候选文件收集用正则一次覆盖`。若因 LSP 不可用而回退 grep，须写明回退原因。非代码理解的纯文本搜索无需标注
```

### P1 — 应该改（5 项，≥1 模型提出且有具体修改方案）

**4. 修正"找引用"→"找语义引用"，区分字面出现**
**5. 修正"零额外代价"→"若 LSP 已就绪可顺带查看诊断"**
**6. 修正"接口实现跳转"→"继承/接口关系理解"，注明工具不支持时的回退**
**7. 修正"语法上几乎不可能"→"误匹配概率低且对任务可接受；若结果驱动修改应抽样核验"**
**8. 正则精度标准增加"若无法快速判断精度，默认走 LSP"**

### P2 — 可选改（2 项，有价值的单模型建议）

**9. 正则标准明确适用边界**："针对 Python 语法（本项目主要脚本语言）"
**10. "倾向 LSP"追加 LSP 不可用回退说明**

## 六、关于 write-claude-md Skill 的方法论观察

本次审查同时实测了 write-claude-md 的 Step 5（多后端验证）：

1. **同 prompt 投三维度的收敛率 ~40%**——说明评审对象本身质量较高时（少明显缺陷），收敛率会低于缺陷多的对象。这不表示方法论失效，而是表示"互补发现"的价值在高质量评审对象上更高。
2. **模型行为模式跨轮次复现**——三次独立审查中 Codex/Kimi/Qwen 各自表现出一致的审查风格。这验证了 write-claude-md Step 5 隐含假设：**不同后端的审查差异不是随机的，而是可预测的系统性差异**。这意味着可以通过"模型-维度"配对来优化审查效率（如 Codex 优先审查实证/可靠性维度，Kimi 优先审查概念/术语维度，Qwen 优先审查结构/一致性维度）。
3. **混合策略缺失由三模型独立发现但各自表述不同**——Kimi 从盲点角度、Qwen 从框架完备性角度、Codex 从最高优先级缺口角度。说明"互补发现"不等于"独特见解"——同一个问题可能被多个模型从不同角度触及，只是措辞和维度归类不同。
