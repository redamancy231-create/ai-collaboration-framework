## 背景

一个 AI 协作方法论文档项目（非软件项目，核心交付物是一份 Markdown 文档）维护了一份 CLAUDE.md 文件，作为项目级 Agent 操作指令。

本次评审聚焦于该 CLAUDE.md 中**新增的"LSP 优先约束"段落**。背景：该项目配置了 `.lsp.json`（pyright Language Server Protocol），Agent 可以调用 LSP 工具（`goToDefinition` / `findReferences` / `hover` / `documentSymbol`）进行代码理解。经实验发现 LSP 并非在所有场景下都优于 grep——纯模式匹配任务（如搜所有函数定义）用 grep 一次调用即可，用 LSP 需要逐文件调用，效率差很多。因此新增了一段分级选择策略来指导 Agent 判断何时用 LSP、何时用 grep。

## 被评估对象

```
- **LSP 优先约束**：本项目已配置 `.lsp.json`（pyright）。代码理解任务必须优先考虑内置 LSP 工具（`goToDefinition`、`findReferences`、`hover`、`documentSymbol`）。选择策略如下：
  - **必须 LSP**（语义理解，grep 无能为力）：找引用（grep 分不清变量/注释/字符串同名文本）、跳转定义（需解析 import/继承/作用域）、类型/签名查询、接口实现跳转
  - **倾向 LSP**（正则精度不够或文件量小 ≤5）：正则易误匹配的场景、单文件/少量文件（LSP 顺带返回 pyright 诊断信息，零额外代价）
  - **倾向 grep**（纯模式匹配 + 正则精度极高 + 文件量大 >5）：`^def ` 搜函数定义、`^import |^from ` 搜导入、`TODO`/`FIXME` 标记搜索等——一次调用覆盖全量文件，远快于逐文件 LSP。正则精度判断标准：行首锚定 + 关键字唯一 + 语法上几乎不可能出现在注释/字符串中
  - 非代码理解任务（纯文本搜索）直接 grep，不触发 LSP 优先
  - **透明度规则**：每次代码理解任务的结果中，必须显式标注实际使用的工具（`[LSP:documentSymbol]` / `[Grep]` 等）及选择理由。格式：`> 工具: LSP:documentSymbol ×21 | 理由: 单文件符号提取，顺带拿诊断` 或 `> 工具: Grep | 理由: ^def 正则精度足够 + 21 文件一次覆盖`。方便事后追溯和审计工具选择是否合理
```

## 评估方法论

本次评审遵循一套名为"脆弱性测试驱动"的 CLAUDE.md 编写方法。核心原则：

- **脆弱性测试**：对每条规则，问"删掉它 → Agent 在该操作上会不会犯错？"会犯错则保留，不会则砍，不确定则标注。
- **五类必含**（每条至少出现其一才合格）：关键命令 / 定制规范 / 架构约束 / 环境要求 / 避坑指南
- **五类必砍**（出现即不合格）：可推导信息（Agent 读文件就能获取的）/ 通用惯例（语言规范级常识）/ 易变信息（随版本/时间变化）/ 文件详述（罗列式文件描述）/ 空泛建议（"遵循最佳实践"类废话）

### 3. 五滥检查
新增段落中是否出现了应砍内容？逐类检查（可推导信息 / 通用惯例 / 易变信息 / 文件详述 / 空泛建议），指出具体问题。

### 4. 透明度规则可执行性

### 5. 自反性检验

### 6. 盲点检测
- 是否需要"同一任务混合使用 LSP 和 grep"的指引？
- 是否遗漏了"需要 LSP 语义能力但文件量很大"这种中间地带？
- 透明度标注格式在实际对话中会不会被截断或丢失？

### 7. 反直觉发现
有没有和主流"LSP 总是更好"建议相反但你认为判断正确的设计？或反过来——有没有你认为判断错误的地方？
