你是受邀对一份方法论文档进行独立评审的审稿人。你的任务不是复述文档内容，而是基于给定的评估原则，对其方法论和产出进行批判性评估。

## 背景

"CLAUDE.md"是 Claude Code（Anthropic 的 AI 编程助手 CLI）的项目级配置文件。它在每次会话启动时自动加载，作为持久化的项目级指令。它的作用是告诉 AI：(1) 不能做什么 (2) 环境/命令是什么 (3) 易误解的概念 (4) 编辑纪律。

## 评估原则（来自方法论专著）

### 三条核心规则
1. **必要性**：每一条都是 LLM 必须知道且无法自行推断的信息。Claude 通过读代码/运行 ls/Glob 能自己获取的内容不应出现在 CLAUDE.md 中。
2. **可操作性**：每一条都是具体、明确且可执行的指令。描述性文字（"项目有X"）不如操作性文字（"做X/别做X"）。
3. **脆弱性测试**：每一条若被删去，都会直接导致 LLM 犯错或偏离预期。

### 十项应包含/应避免
应包含：(1) 关键命令 (2) 定制规范 (3) 架构约束 (4) 环境要求 (5) 避坑指南
应避免：(6) 可推导信息 (7) 通用惯例 (8) 易变信息 (9) 文件详述 (10) 空泛建议

### 核心洞察
- 少即多：冗长的 CLAUDE.md 导致关键规则淹没在噪声中，且每次 API 调用全量加载消耗 token
- 易变信息会导致缓存失效（Anthropic prompt cache 是前缀匹配）
- 专业领域详细知识应迁移至 Skills 模块，实现按需加载

## 评估要求

请从以下维度进行独立评估，给出具体判断和理由（不只说"好"或"不好"，要说"为什么"）：

1. **方法论一致性**：新版是否真正执行了它所声称的原则（三规则+十项）？有没有声称砍掉但在某处偷偷保留的可推导信息？有没有该加但没加的内容？

2. **盲点检测**：从你的视角看，这个 CLAUDE.md 有没有遗漏任何 LLM 不可推导但实际操作中会导致犯错的信息？你有没有想到该项目特有的陷阱是我们遗漏的？

3. **过度削减风险**：有没有哪条被删掉的内容你认为是误删——删了之后 LLM 确实会因此犯错？

4. **可推广性**：这个方法论（脆弱性测试驱动+逐条删留判断）推广到其他类型项目时的适用程度如何？有什么项目类型会比这个更难或更容易？

5. **反直觉发现**：有没有和主流建议相反但你认为正确的判断？（比如"版本历史应该砍掉"这种——很多人会认为版本历史是重要的上下文）

请保持批判性——如果你认为某处做错了，明确指出并给出修正方案。如果你认为有更好的替代方案，也请说明。
