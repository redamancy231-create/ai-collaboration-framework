# Retrospect — 2026-06-27 CLAUDE.md 重构 + 编写方法论建立

## 做了什么
1. 对 AI 协作框架项目的 CLAUDE.md 执行逐条脆弱性测试，163→77 行
2. 同一份审查 prompt 投 Codex/Qwen/Kimi 三个后端
3. 基于审查发现建立了 CLAUDE.md 编写五步协议
4. 创建 `/write-claude-md` Skill
5. 对 Skill 本身也做了三后端审查

## 发现

### 方法论层面
- **自然多样性模式复现**：两轮三后端审查（CLAUDE.md + Skill），分布结构几乎一致（3/3收敛~3条、独特发现~8条、0互相证伪）。同一份 prompt 不刻意设计角度差异，不同模型也会自然形成互补的盲点覆盖
- **审查行为模式分类**：Codex 走"文件验证型"（先读实际文件再下判断），Qwen/Kimi 走"摘要评估型"（基于 prompt 内容评审）。两种模式各有价值
- **三后端视角分布**：Codex=工程/结构完整性，Qwen=操作/诊断，Kimi=元认知/执行可行性。两轮审查中这种分布是稳定的

### 操作层面
- CLAUDE.md 的脆弱性测试本身是通用且有效的——"删了会不会犯错"这个判据不依赖项目类型
- 模型白名单是典型的易变信息，应该指针化而非硬编码
- "最后更新日期"制造无意义的 prompt cache 失效——CLAUDE.md 应该只在指令变更时才改
- Skill 的自反性失败（教别人写好的 CLAUDE.md 但自己不展示好指令的形态）是一个常见的设计陷阱

## 教训
- 多后端审查即使不设计刻意角度差异也有显著价值——基底多样性是免费的
- Skill 写好后应该用自己的协议自审一遍——write-claude-md 初版违反了自己主张的五滥检查
- 追加式日志必须标注读取方向（从末尾读）——Agent 默认从顶部读
