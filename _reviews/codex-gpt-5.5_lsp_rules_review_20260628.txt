### 1. 决策框架完备性

判断：基本覆盖常见场景，但不完备；存在几个会影响 Agent 判断的灰色地带。

理由：四级分流能抓住核心差异：语义理解优先 LSP，纯文本/高精度批量模式优先 grep。这一点是合理的。但当前写法把任务近似处理成“选 LSP 或选 grep”的单选题，而实际代码理解经常是混合任务：先用 grep 批量找候选，再用 LSP 验证定义、引用、类型或作用域。缺少混合策略会让 Agent 在复合任务里过早押注单一工具。

边界条件也有两处不够清晰。第一，“需要 LSP 语义能力但文件量很大”的任务虽然理论上应被“必须 LSP”覆盖，但紧接着又说“大于 5 文件倾向 grep”，容易让 Agent 把文件量错误地置于语义需求之上。第二，“倾向 LSP / 倾向 grep”没有说明 LSP 不可用、超时、索引未就绪时如何回退。

此外，“接口实现跳转”放在“必须 LSP”中有工具能力不匹配风险。段落开头列出的工具是 `goToDefinition` / `findReferences` / `hover` / `documentSymbol`，并没有明确提供 implementation 跳转工具；如果当前 LSP 工具集不能直接跳实现，这条会诱导 Agent 承诺做不到的操作。

建议修改：

Before:
```markdown
- **必须 LSP**（语义理解，grep 无能为力）：找引用（grep 分不清变量/注释/字符串同名文本）、跳转定义（需解析 import/继承/作用域）、类型/签名查询、接口实现跳转
```

After:
```markdown
- **必须 LSP**（语义理解，grep 无法可靠判定）：找语义引用（需排除注释/字符串/同名局部变量）、跳转定义（需解析 import/继承/作用域）、类型/签名查询、继承/接口关系理解。若当前工具集不支持实现跳转，先用 LSP 获取定义/类型，再用 grep 缩小候选并说明限制
- **混合策略**：任务同时包含“全库候选收集”和“语义判定”时，先用 grep 批量收集候选，再对候选使用 LSP 验证；最终按透明度规则同时标注两类工具
```

### 2. 脆弱性测试（逐条）

(a) 必须 LSP

判断：会犯错。

犯错路径：删掉后，Agent 在“找某个函数/变量的真实引用”“判断符号定义来源”“确认类型签名”这类任务中可能直接用 grep 搜同名文本。grep 会把注释、字符串、同名局部变量、不同作用域符号混在一起，导致引用集合污染，进一步导致错误修改、漏改或错误解释调用关系。

(b) 倾向 LSP

判断：会犯错，但严重性低于“必须 LSP”。

犯错路径：删掉后，Agent 在“正则能搜到但精度不稳定”的小范围任务中，可能为了省事直接 grep。例如在少量文件中找类方法、装饰器函数、重名属性或从 import 间接暴露的符号时，grep 结果看似足够，实际可能漏掉跨文件解析信息，或把同名文本当作同一符号。保留该条能把灰区任务推向更稳妥的语义工具。

(c) 倾向 grep

判断：会犯错。

犯错路径：删掉后，Agent 可能把“LSP 优先”机械理解为全场景使用 LSP。对于 `^def `、`^import |^from `、`TODO`/`FIXME` 这类全库纯模式搜索，LSP 需要逐文件调用或绕路提取符号，不但慢，而且容易因为调用次数多、上下文截断或遗漏文件而产生不完整结果。这里的错误主要是效率和覆盖率错误。

(d) 非代码理解直接 grep

判断：会犯错。

犯错路径：删掉后，Agent 可能把 Markdown、配置说明、提示词、任务记录等纯文本搜索也纳入“LSP 优先”。这会浪费工具调用，甚至在非代码文件上得到无意义结果。该条能明确把“代码语义理解”和“纯文本检索”切开，必要。

(e) 透明度规则

判断：会犯错。

犯错路径：删掉后，Agent 即使选错工具，结果中也不会留下可审计线索。后续无法判断错误来自需求理解、搜索策略、工具限制还是执行遗漏；团队也难以迭代这套规则。透明度规则本身不提高一次搜索的准确率，但能显著提高事后追溯和规则修正能力。

### 3. 五滥检查

可推导信息：轻微存在，但不构成应砍问题。“本项目已配置 `.lsp.json`（pyright）”确实可以通过读文件推导，但它在这里承担的是环境要求和触发条件，不是普通文件罗列。建议保留，但如果 CLAUDE.md 其他位置已经重复说明 `.lsp.json`，这里可以压缩为“本项目已配置 LSP”。

通用惯例：未发现明显问题。LSP 与 grep 的分流不是语言规范级常识，而是本项目针对 Agent 工具选择的定制规范和避坑指南，符合应保留类型。

易变信息：存在，需要改。主要是三处：

1. “pyright 诊断信息，零额外代价”过于绝对。LSP 是否返回诊断、是否已完成索引、是否存在启动成本，都取决于工具实现和当前状态。
2. “接口实现跳转”可能依赖未列出的 LSP capability，和当前列出的工具不完全匹配。
3. “文件量小 ≤5 / 文件量大 >5”是经验阈值，适合作为启发式，不宜表现得像稳定规律。

建议修改：

Before:
```markdown
- **倾向 LSP**（正则精度不够或文件量小 ≤5）：正则易误匹配的场景、单文件/少量文件（LSP 顺带返回 pyright 诊断信息，零额外代价）
```

After:
```markdown
- **倾向 LSP**（正则精度不够或范围较小，约 ≤5 文件）：正则易误匹配的场景、单文件/少量文件。若 LSP 已就绪，可顺带查看可用诊断；若 LSP 不可用、未就绪或超时，应记录回退原因
```

文件详述：未发现问题。该段没有罗列项目文件职责，只提到 `.lsp.json` 作为环境前提，合理。

空泛建议：基本没有。虽然“必须优先考虑内置 LSP 工具”单独看可能偏泛，但后面给出了必须、倾向、直接 grep 和透明度格式，已经具备可执行性。

### 4. 透明度规则可执行性

判断：大体可执行，格式示例清晰，但适用位置和混合工具场景需要补强。

一个从未见过此规则的 Agent 能理解“在结果中标注工具和理由”，示例也足够具体：`> 工具: ... | 理由: ...` 这种格式可直接模仿。问题在于“每次代码理解任务的结果中”仍有歧义：是每次中间回复都写，还是最终答复写；是列出每一次调用，还是汇总工具类型；如果任务混用 LSP 和 grep，格式如何表达。

建议修改：

Before:
```markdown
- **透明度规则**：每次代码理解任务的结果中，必须显式标注实际使用的工具（`[LSP:documentSymbol]` / `[Grep]` 等）及选择理由。
```

After:
```markdown
- **透明度规则**：每次代码理解任务的最终答复或交付结果开头，必须保留一行工具选择说明：`> 工具: ... | 理由: ...`。若混用工具，按实际用途汇总，例如 `> 工具: Grep + LSP:goToDefinition ×3 | 理由: grep 批量收集候选，LSP 验证定义归属`。若因 LSP 不可用而回退 grep，也必须写明回退原因。
```

### 5. 自反性检验

判断：基本自洽。

按这套规则，Agent 在判断“该用什么工具来理解 CLAUDE.md 中的指令”时，应该选择 grep 或直接读取文本，而不是 LSP。理由是 CLAUDE.md 是 Markdown 指令文件，不是需要解析 import、作用域、定义、引用或类型签名的代码语义对象；这属于“非代码理解任务（纯文本搜索）直接 grep”或“直接读文件理解”的范围。

这个结论与规则自身倡导的原则一致：不要因为项目配置了 LSP 就把所有理解任务都交给 LSP。该段实际上承认了 LSP 的适用边界，因此自反性上没有根本矛盾。

需要注意的是，如果 CLAUDE.md 中嵌入代码块，理解代码块文字本身仍不等于理解项目源码符号。除非任务明确要求追踪代码块对应的真实源码定义/引用，否则仍不应触发 LSP。

### 6. 盲点检测

是否需要“同一任务混合使用 LSP 和 grep”的指引？

判断：需要，而且是最高优先级缺口之一。

理由：大量真实任务不是纯语义或纯模式。例如“找所有可能调用某接口的地方，并判断哪些是真实调用”“找所有 `@router` 路由并追踪 handler 类型”“全库找候选类名再确认继承关系”。这类任务最合理的路径通常是 grep 批量召回、LSP 精确验证。当前规则没有显式允许混合，可能让 Agent 误以为必须二选一。

是否遗漏了“需要 LSP 语义能力但文件量很大”这种中间地带？

判断：有遗漏，或至少表达不够强。

理由：“必须 LSP”理论上应覆盖它，但“文件量大 >5 倾向 grep”会制造冲突。应明确优先级：只要最终判断依赖定义、引用、类型、作用域，语义需求优先于文件量；文件量大时可以用 grep 缩小候选，但不能用 grep 替代语义验证。

透明度标注格式在实际对话中会不会被截断或丢失？

判断：会，有现实风险。

理由：如果标注放在长结果末尾，或散落在中间过程，容易被上下文截断、摘要压缩或用户只查看最终结论时丢失。应要求放在最终答复或交付文件的开头，并用单行汇总格式，避免逐调用长列表。

### 7. 反直觉发现

判断正确的反直觉设计：明确承认 grep 在高精度批量模式匹配中优于 LSP，这是正确的。主流建议常把“代码理解 = LSP”说得过满，但 LSP 的强项是语义解析，不是全库简单模式扫描。`^def `、`^import |^from `、`TODO`/`FIXME` 这类任务用 grep 一次覆盖全量文件，通常更快、更完整、更容易审计。

判断错误或需要修正的地方：

1. “找引用”表述过绝对。应改成“找语义引用”。如果用户要的是字面出现、日志文本、配置键、文档提及次数，grep 才是正确工具。
2. “接口实现跳转”与列出的工具能力不一致。除非工具集确实提供 implementation 查询，否则不应作为硬性能力写入。
3. “零额外代价”判断过强。LSP 诊断是否顺带可得取决于实现和状态，不能作为稳定承诺。
4. “语法上几乎不可能出现在注释/字符串中”也略绝对。行首锚定能降低误匹配，但多行字符串、文档示例和代码块仍可能匹配。更稳妥的说法是“误匹配概率低且对任务可接受；若结果将驱动修改，应抽样核验”。

建议修改：

Before:
```markdown
找引用（grep 分不清变量/注释/字符串同名文本）
```

After:
```markdown
找语义引用（需要排除变量/注释/字符串同名文本时）；若任务目标是字面出现或文本提及，使用 grep
```

Before:
```markdown
正则精度判断标准：行首锚定 + 关键字唯一 + 语法上几乎不可能出现在注释/字符串中
```

After:
```markdown
正则精度判断标准：行首锚定 + 关键字唯一 + 误匹配概率低且对当前任务可接受；若搜索结果将直接驱动修改，应抽样核验
```

## 总结

1. 最高优先级修改：增加“混合策略”，明确 grep 可用于批量召回，LSP 用于语义验证；大文件语义任务不能被“倾向 grep”覆盖。
2. 修正过度承诺：删除或软化“零额外代价”“接口实现跳转”“语法上几乎不可能”等不稳定表述。
3. 强化透明度规则：要求工具说明放在最终答复或交付结果开头，并支持混用工具和回退原因的单行汇总。
