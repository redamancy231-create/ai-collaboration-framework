# Prompt 偏置审查报告

**审查对象**：`AI协作项目全生命周期框架` 冷读审查 prompt  
**审查目的**：判断该 prompt 是否适合作为 OPEN-1 同构问题（"离散的人类审查机制是否无法发现连续的 AI 协作方向漂移"）的冷读独立发现测试输入。  
**审查原则**：只基于 prompt 文本本身，不判断框架文档内容或 OPEN-1 是否真实存在。

---

## 一、结论

- **偏置等级**：C
- **一句话结论**：该 prompt 没有直接泄漏 OPEN-1 的具体措辞，但其身份设定、缺口类型限定、数量要求与作者意识探查共同构成了中度方向性偏置，会系统性地提高审查者命中"元审查 / 监督验证 / 方向控制"类问题的概率，因此不足以支撑"独立发现"的结论。

---

## 二、逐项审查

### 1. 方向性泄漏

- **判断**：存在中度方向性泄漏，但非强泄漏。
- **证据**：
  - prompt 中未出现"人类审查失效""AI 漂移""连续过程""独立验证"等 OPEN-1 关键词，因此不存在直接的方向性泄漏。
  - 但"方法论审阅者""结构性缺口"的设定，会把审查者导向"框架如何自证有效""监督机制是否闭合"等元层面问题。OPEN-1 正属于这一类元机制问题。
  - 审查对象的标题"AI协作项目全生命周期框架"本身包含"协作""全生命周期"，会自然触发对方向控制、过程监督、审查机制的思考，这部分偏置来自主题本身，不完全是 prompt 的过错。
- **严重程度（1-10）**：6

### 2. 严重性诱导

- **判断**：存在明显的严重性诱导，但主要是一般性的"找重大问题"偏置，不特别指向 OPEN-1。
- **证据**：
  - "设计层面的致命问题""杀伤力排序""不要客气""不要给好话"等措辞，会迫使审查者以高严重性为优先标准，倾向于夸大问题或凑出严重缺口。
  - 这种诱导对任何"元机制类"问题都更有利——因为元机制问题（如审查机制缺失）最容易被描述为"致命"。OPEN-1 恰好属于此类，因此间接受益。
- **严重程度（1-10）**：7

### 3. 搜索空间收窄

- **判断**：存在中度搜索空间收窄，且部分指向 OPEN-1 方向。
- **证据**：
  - "结构性缺口"排除了措辞、错别字、表述问题，把注意力集中在"设计结构"上。
  - "方法论审阅者"的身份进一步把搜索空间收窄到方法论层面的缺陷，如定义、流程、验证、监督、反馈等。
  - "作者是否已意识到"要求审查者主动比对文档内文，这会引导其寻找文档中已有但未被充分解决的"待决项"，而这类待决项往往集中在验证机制、审查机制、风险控制等元问题上。
  - 这种收窄不会直接命中 OPEN-1，但会显著提高进入 OPEN-1 邻近搜索空间的概率。
- **严重程度（1-10）**：6

### 4. 确认偏误放大

- **判断**：存在强确认偏误放大，严重损害阴性结果解释力。
- **证据**：
  - "列出你认为最严重的 3 个"隐含了"至少存在 3 个严重缺口"的前置假设。
  - 审查者即使只发现 1-2 个真正严重的问题，也被迫凑出第 3 个，导致低质量命中和假阳性。
  - 如果审查者最终报告了 OPEN-1 同类问题，我们无法区分这是真实独立发现，还是为了完成"3 个致命问题"任务而构造的结果。
  - 该格式不允许"未发现足够严重缺口"或"零严重缺口"的结论，破坏了冷读实验的阴性对照能力。
- **严重程度（1-10）**：8

### 5. 作者意识诱导

- **判断**：存在中度作者意识诱导，可能提高 OPEN-1 命中概率。
- **证据**：
  - "作者是否已经意识到（看文档是否提到过）"会促使审查者专门寻找文档中已有但未被充分解决的线索。
  - "如 §1.6 的待决项"这一示例非常具体，如果 §1.6 涉及审查机制、验证机制或待决的元问题，审查者会被直接引导到 OPEN-1 的邻近区域。
  - "严重性是否被低估"进一步诱导审查者把已被承认的小问题升级为"致命问题"。
  - 如果 OPEN-1 同构问题在原文档中已有部分痕迹（哪怕是轻微的），该 prompt 会显著放大命中概率。
- **严重程度（1-10）**：6

## 四、是否污染 OPEN-1 独立性

- **如果审查者最终命中 OPEN-1 同构问题，当前 prompt 是否足以让这个命中仍然算"独立发现"？**
  
  **不足以算独立发现。**

- **如果不能，为什么？**
  - prompt 通过"方法论审阅者""结构性缺口""作者是否已意识到"等设定，把审查者的搜索空间系统性地导向元审查、监督验证、方向控制等区域。OPEN-1 正好是这一区域的核心问题。
  - "必须找出 3 个致命问题"的格式会迫使审查者构造或放大问题，无法区分真实发现与任务驱动的凑数。
  - "作者意识诱导"会让审查者主动挖掘文档中已有但未充分解决的线索，进一步提高命中 OPEN-1 的概率。
  - 因此，命中 OPEN-1 更可能是 prompt 结构引导的结果，而非审查者独立、自发地发现。

- **在什么情况下可以勉强算独立发现？**
  - 仅当审查者能够给出超出 prompt 暗示的具体证据，明确说明该缺口为何独立于"方法论""结构性""元机制"等框架，且能解释其发现路径不依赖上述诱导时，才可部分视为独立发现。即便如此，由于 prompt 存在多重偏置，其独立性的置信度仍然较低。

## 五、改写建议

以下是一版更低偏置的冷读 prompt，满足：
- 不提 OPEN-1
- 不暗示"连续漂移 / 人审失效 / AI 方向偏移"
- 允许审查者报告"未发现致命结构缺口"
- 保留发现重大结构问题的能力
- 输出格式仍然便于比较不同模型结果

```markdown
你是一位文档审阅者。请阅读以下方法论文档，并评估其是否存在重要设计缺陷或风险。

【审查对象】
AI协作项目全生命周期框架.md（随附）

【审查任务】
1. 通读文档后，判断是否存在你认为重要的设计缺陷或风险（不是措辞问题、不是错别字）。
2. 如果存在，列出最多 3 个，按重要性排序；如果不存在或不足 3 个，如实报告即可。
3. 对每个缺陷或风险说明：
   - 它是什么；
   - 你为什么认为它重要；
   - 文档中是否已提到或部分涉及该问题。

【审查约束】
- 只基于这份文档本身判断，不引入外部知识。
- 如果文档中已经提到某个问题，请如实标注，并可说明文档的处理是否充分。
- 保持中立、具体，避免泛泛批评或夸大。
- 如果没有发现足够重要的设计缺陷，请明确说明"未发现足够重要的设计缺陷"。

【输出格式】

总体判断：文档是否存在重要设计缺陷或风险？（是 / 否 / 不确定）

缺陷/风险1：[标题]
  - 是什么：
  - 为什么重要：
  - 文档是否已涉及：
  - 重要性评分（1-10）：

（缺陷/风险2、3 同结构；如无，可省略）

未发现的说明：
如果你认为文档整体上没有重要设计缺陷，或只发现不足 3 个，请在此简要说明理由。
```
