# LSP 优先约束段落评审报告

评审对象：CLAUDE.md 中新增的"LSP 优先约束"段落
评审日期：2026-06-28
评审人：Kimi K2.6

---

### 1. 决策框架完备性

**判断：基本完备，但存在两处灰色地带。**

四级分流（必须 LSP → 倾向 LSP → 倾向 grep → 直接 grep）在逻辑上覆盖了从语义理解到纯文本搜索的频谱，结构清晰。但存在以下灰色地带：

**灰色地带 A："正则精度极高"的判断标准可操作性不足。**
规则给出的判断标准是"行首锚定 + 关键字唯一 + 语法上几乎不可能出现在注释/字符串中"。但 Agent 面对一个具体搜索需求时，很难在调用前准确预估正则是否会误匹配注释/字符串。例如搜索 class Foo 时，如果某文件中有 "class Foo" 的字符串字面量，行首锚定可以过滤掉，但如果搜索的是 oo(，注释中完全可能出现 # foo() is deprecated。Agent 缺乏先验知识来判断"语法上几乎不可能出现"，这个标准更像事后审计用语，而非事前决策依据。

**灰色地带 B：文件量阈值 5 的刚性边界。**
"≤5 倾向 LSP，>5 倾向 grep" 的阈值过于绝对。实际场景可能是 6 个文件，每个文件 20 行；也可能是 5 个文件，每个文件 5000 行。规则未考虑文件大小/复杂度，仅按文件数量一刀切，Agent 在边界附近容易做出次优选择。

**灰色地带 C："接口实现跳转"在 Python/pyright 语境下的适用性模糊。**
规则将"接口实现跳转"列为必须 LSP 的场景，但 Python 是鸭子类型语言，"接口实现"概念弱于 Java/Go。pyright 的 goToDefinition 对抽象基类（ABC）的实现跳转支持有限，Agent 可能尝试使用该能力但发现 pyright 返回结果不佳，此时规则未给出降级策略。

**修改建议：**
- before: 正则精度判断标准：行首锚定 + 关键字唯一 + 语法上几乎不可能出现在注释/字符串中
- after: 正则精度判断标准（同时满足）：①行首锚定（^）；②关键字为保留字或语法结构词（def/class/import/from/TODO）；③不含常见变量名（如单字母、驼峰命名）。若无法快速判断精度，默认走 LSP

### 2. 脆弱性测试（逐条）

**(a) 必须 LSP**
**判断：会犯错。**
删掉后，Agent 在"找引用"任务上可能直接使用 grep，导致把注释中的同名词、字符串字面量中的文本都当作有效引用返回。例如搜索变量 config 的引用，grep 会匹配到 "# default config" 注释和 JSON.dumps({"config": path}) 字符串，造成引用列表严重污染。这是语义级错误，非人工后期可简单修正。

**(b) 倾向 LSP**
**判断：会犯错（特定场景）。**
删掉后，Agent 在"正则精度不够"的场景下可能为求效率选择 grep，导致漏匹配或误匹配。例如搜索装饰器 @property 的使用，grep ^\s*@property 在嵌套函数或类方法中可能因缩进层级变化而漏掉（若正则写为 ^@property），或过度匹配注释中的示例代码。LSP 的 documentSymbol 能准确识别装饰器应用位置。在文件量小（≤5）的场景下，删掉此条 Agent 用 grep 通常也能得到正确结果，只是损失了 pyright 诊断信息的附带价值，不算操作错误。

**(c) 倾向 grep**
**判断：会犯错。**
删掉后，Agent 在"纯模式匹配大文件"场景下可能机械地优先 LSP，导致对 21 个文件逐个调用 documentSymbol 或 findReferences，效率极低，甚至可能因调用链过长而触发超时或上下文溢出。这不是结果错误，是性能/可用性错误——在工程实践中同样严重。

**(d) 非代码理解直接 grep**
**判断：会犯错。**
删掉后，Agent 在纯文本搜索任务（如搜索文档中的"发布流程"章节、查找 CHANGELOG 中的版本记录）可能错误地尝试使用 LSP 工具。documentSymbol 对 Markdown 的支持有限，indReferences 对自然语言文本完全无效。Agent 会获得空结果或无关结果，导致任务失败或产生幻觉。

**(e) 透明度规则**
**判断：不会（操作层面），但该规则属于治理约束。**
删掉后，Agent 在选择工具上的决策质量不会下降，仍能正确完成代码理解任务。该规则的价值在于事后审计和持续优化——通过查看历史对话中的工具标注，维护者可以发现 Agent 是否频繁在不应使用 LSP 的场景下调用 LSP（或反之），从而迭代规则。它满足"五类必含"中的"定制规范"，应保留，但脆弱性测试结论为"删掉不会导致操作错误"。

### 4. 透明度规则可执行性

**判断：基本可执行，但存在格式歧义。**

一个从未见过此规则的 Agent 阅读该规则后，大概率能理解"需要在结果中标注工具和理由"这一要求。但格式示例存在以下问题：

**问题 1：×21 的语义不明确。**
示例 > 工具: LSP:documentSymbol ×21 中的 ×21 是指调用了 21 次 documentSymbol？还是处理了 21 个文件？还是返回了 21 个符号？不同解读会导致标注格式不一致。Agent 在没有先验约定的情况下，无法确定如何填写这个数字。

**问题 2：标注位置未明确。**
规则说"每次代码理解任务的结果中"必须标注，但未说明是放在结果开头、结尾还是穿插在中间。Agent 可能把标注塞进代码块内，导致格式混乱。

**问题 3：混合工具场景未示例。**
如果一次任务中先 grep 定位文件，再 LSP 深入分析，标注格式未给出示例。

**修改建议：**
- before: 格式：> 工具: LSP:documentSymbol ×21 | 理由: 单文件符号提取，顺带拿诊断
- after: 格式：放在该次任务输出的末尾独立一行。单工具：> 工具: LSP:documentSymbol(调用次数=21) | 理由: 单文件符号提取，顺带拿诊断。多工具：> 工具: Grep(1次)→LSP:hover(3次) | 理由: 先用Grep定位候选文件，再LSP确认类型

### 5. 自反性检验

**判断：自洽，但存在一个有趣的边界。**

按照规则自身，Agent 在理解 CLAUDE.md 中的这段指令时：
1. CLAUDE.md 是纯 Markdown 文本，非代码文件；
2. 任务目标是"理解文本指令"，不属于"代码理解任务"；
3. 因此应落入"非代码理解任务（纯文本搜索）直接 grep"这一档。

这个结论是自洽的——用 grep 搜索 CLAUDE.md 中的"LSP""grep"等关键词来定位相关段落，再逐段阅读理解，是正确且高效的方式。LSP 工具（documentSymbol）对 Markdown 的支持远不如对代码文件的支持，强行使用 LSP 反而可能得不到有用的符号信息。

**边界情况：** 如果 Agent 把"理解项目指令"误归类为"代码理解任务"（因为指令中涉及代码工具的选择），则可能错误地优先尝试 LSP。这说明规则的"非代码理解直接 grep"这一条本身需要被正确理解——而它的理解又依赖于对"代码理解"与"纯文本理解"的区分。这是一个元级别的自指，好在区分标准足够明确（是否涉及代码语义）。

### 6. 盲点检测

**盲点 1：是否需要"同一任务混合使用 LSP 和 grep"的指引？**
**判断：需要补充。**

当前规则隐含了"二选一"的假设，但实际工作流中混合策略非常常见：
- 场景：在 20 个文件中查找某函数的所有调用，并需要排除注释/字符串中的误匹配。
- 纯 LSP：调用 20 次 findReferences，准确但极慢。
- 纯 grep：一次调用覆盖 20 文件，但可能误匹配。
- 混合策略：先用 grep grep -n "funcName(" 快速定位候选行（20 文件 1 次调用），再对 grep 命中的少量文件（可能仅 3-5 个）用 LSP findReferences 确认真正的语义引用。

规则目前未覆盖这种"先用 grep 粗筛、再用 LSP 精校"的混合模式。Agent 可能被迫在"慢但准"和"快但脏"之间做非此即彼的选择。

**盲点 2：是否遗漏了"需要 LSP 语义能力但文件量很大"这种中间地带？**
**判断：未遗漏，但效率问题未解决。**

"必须 LSP"的场景实际上覆盖了这一类——语义理解的准确性要求优先于效率。但规则未提供任何缓解大文件量下 LSP 调用开销的策略（如批量并行调用、缓存结果、或先 grep 缩小范围）。建议在不改变四级分流结构的前提下，在"必须 LSP"下增加一句效率提示："文件量 >10 时，可先用 grep 粗筛疑似文件，再对候选文件逐个 LSP 验证。"

**盲点 3：透明度标注格式在实际对话中会不会被截断或丢失？**
**判断：会。**

当前格式示例的标注行较长（> 工具: LSP:documentSymbol ×21 | 理由: 单文件符号提取，顺带拿诊断）。在以下场景中会出问题：
- Agent 使用多轮工具调用，最终只输出精简总结，标注行被省略；
- 工具结果被嵌入到 JSON 或代码块中，标注行破坏格式；
- 对话上下文过长时，Agent 为压缩内容可能丢弃"非核心信息"。

规则未规定标注的"不可省略性"，也未处理标注与结果格式的兼容性。

**修改建议：** 在透明度规则后追加："标注行不可省略，不可嵌入代码块内。若输出格式限制（如 JSON），将标注作为同级字段放入。"

### 7. 反直觉发现

**反直觉判断正确的设计：**

1. **"grep 在大文件纯模式匹配场景下优于 LSP"**——这与"LSP 总是更智能"的主流直觉相反，但判断完全正确。LSP 的语义优势建立在 AST 解析上，对于"找所有函数定义"这种纯结构化文本任务，grep 的线性扫描在工程规模下远胜逐文件 RPC 调用。规则敢于在 Agent 指令中明确推翻"LSP 万能论"，体现了对工具成本的清醒认知。

2. **"文件量小（≤5）时倾向 LSP，因为零额外代价拿诊断"**——这是一个容易被忽略的次要收益。通常工具选择只比较"查询结果质量"，规则把"附带诊断信息"纳入成本效益分析，是细致的设计。

**我认为判断可能错误的地方：**

1. **阈值 5 过于经验化，缺乏项目语境支撑。**
在一个只有 10 个 Python 文件的小型项目中，>5 文件占了项目一半，此时"倾向 grep"的阈值可能让 Agent 过度使用 grep；在一个有 1000 个文件的大型项目中，≤5 的场景几乎不存在，阈值失去意义。规则以一个固定数字适用于所有规模的项目，缺乏弹性。建议改为相对阈值或增加项目规模考量，例如："文件量占项目总量 <10% 且 ≤5 时倾向 LSP"。

2. **"类型/签名查询"列为必须 LSP 过于绝对。**
在强类型注解的 Python 项目中，hover 查询类型确实需要 LSP；但如果项目完全没有类型注解，pyright 的 hover 返回的类型信息极为有限（甚至全是 Unknown）。此时 LSP 相对于 grep（搜索函数定义处的默认参数和 docstring）的优势大幅缩小。规则未考虑"项目类型注解覆盖率"这一变量，对无类型项目的指导可能产生反效果（LSP 返回 Unknown，不如 grep 看源码直接）。

## 总结

**最高优先级修改建议（3 条）：**

1. **补充"混合策略"指引**  
   在四级分流后增加一条混合策略规则："同一任务可先 grep 粗筛疑似文件/行号，再对候选结果用 LSP 精准确认。标注格式：> 工具: Grep→LSP:findReferences | 理由: Grep 20 文件 1 次定位候选，LSP 3 文件确认语义引用。"  
   *理由：当前二选一框架在"准确性要求高 + 文件量大"场景下让 Agent 陷入两难，混合策略是工程实践中的常态。*

2. **明确透明度标注的不可省略性与格式细节**  
   将 ×21 改为 (调用次数/文件数=21) 消除歧义；明确标注放在输出末尾、不可嵌入代码块、不可因上下文压缩而省略。  
   *理由：当前格式示例存在语义歧义，且未约束 Agent 在输出精简时保留标注，导致审计链断裂。*

3. **增加"无类型项目"的降级提示**  
   在"必须 LSP：类型/签名查询"后追加限定："若项目类型注解覆盖率极低（pyright 大量返回 Unknown），类型查询可降级为 grep 函数定义处 + docstring 分析。"  
   *理由：Python 项目类型注解差异极大，将类型查询绝对化为"必须 LSP"在无类型项目中会产生空转调用，违背规则自身的效率原则。*
