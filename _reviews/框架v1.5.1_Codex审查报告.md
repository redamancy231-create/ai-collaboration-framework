## 维度1：旧问题修复核查（P0-P9）

### P0：版本对齐 — 确认修复

新草案已基于 v1.5 主文档结构重编号。主文档现有 §3.7.1-§3.7.9、§9.1-§9.6，之后进入 §10；新草案使用 §3.7.0、§3.7.4.1、§9.7、§9.8，未再占用已有 §9.6。

- §3.7.0 放在 §3.7.1 之前，作为漂移检测层前置输入源，位置可用。
- §3.7.4.1 明确是 §3.7.4 规则退役自动化的补充替代，而非再误改 §3.7.3。
- §9.7 避开了 v1.5 已有 §9.6。
- §9.8 编号可用，可插入 §9 跨层关切后段。

合并时需把 Markdown heading 层级调整为主文档风格：草案用 `## §3.7.0`，主文档内应落为 `#### 3.7.0` 或同级标题。这是编辑格式问题，不是设计阻断。

### P1：边界修正 — §3.7.0 已修复，§3.7.4.1 有新边界风险

§3.7.0 的三条规则已从自动干预改为观测与建议：

- 规则1写明“不自动抑制信号注入，不修改事件日志”。
- 规则2写明“不自动注入 explore 信号，不改变当前迭代的 intent”。
- 规则3写明“不自动干预”。
- 草案还写明“不新增独立告警等级，不自动改变 Loop/Workflow 行为”，并把 `event_stream_health` 接入现有 §3.7.3 四级聚合。

因此第一轮指出的“自动抑制/自动 explore 违反只观测不阻断”问题，对 §3.7.0 来说已经实质修复。

但 §3.7.4.1 仍有一个新边界问题：它把 `weight <= -0.3` 或连续 8 次无影响写成“任一触发即淘汰”“移出活跃资产池”。如果这会自动改变后续 prompt 注入或规则可用性，就仍是执行层干预。建议改为“标记 `suppression_candidate` / `inert_candidate`，在下一 HG-1 中由操作者确认后才移出活跃池”，或显式声明该淘汰只改变报告状态，不改变运行行为。

### P2：去隐喻化 + 去背书 — 大体修复，仍有残留措辞

功能性术语已基本替换：

- Gene → Validation Rule / 验证规则
- Capsule → Success Snapshot / 成功快照
- Event → Audit Record / 审计记录
- `epigenetic_mark` → `performance_track`
- 表观遗传权重 → 自适应权重淘汰

来源背书也明显收敛：草案顶部直接标注 Evolver 综合评分 4.1-4.2/10、方法论可靠性 6.5/10，并声明来源项目质量较低，需框架内独立试跑验证后升级。§9.7 把 4,590 改为 retained analyses，且标注 [Sp]，不再写成强背书。

残留问题：草案和 JSON 仍在“旧名称”字段中保留 Gene/Capsule/Event，并在自检清单中同时写“无 Gene/Capsule/Epigenetic 残留”。作为迁移说明可以保留一次，但不应进入运行时 schema 或检查清单。建议把旧名称移到一次性脚注或附录，删除 JSON 中的 `old_name` 字段，或改名为非运行字段 `source_term_for_migration_only`。

### P3：符号修正 — 确认修复

旧版错误 `boost -= delta_failure` 且 `delta_failure = -0.15` 已修复为：

- 成功：`weight += delta_success`
- 失败：`weight += delta_failure`
- `delta_failure = -0.15`

未发现 `boost -= (-0.15)` 的残留。

### P4：REO schema 正式化 + 晋升规则 — 部分修复

Validation Rule 与 Success Snapshot 已给出正式 JSON Schema，包含 `$schema`、`required`、`enum`、`pattern`、`$defs` 等要素，较旧版“示例对象”有实质进步。

不足：

- Audit Record 没有完整 JSON Schema，只是用 prose 说明“同成功快照基础结构，额外包含字段”。这还不能算完整正式化。
- 配套 `框架v1.5.1_新增节草案.json` 是合法 JSON，但只是草案摘要，不包含 Markdown 中的完整 schema。如果它被期待作为机器可读规范，目前不够。
- schema 未约束 `success_count <= total_count`、`updated_at` 与 `created_at` 的关系、`compact_view` 是否在注入层必填等跨字段规则。可接受为 Phase 1，但不能称为完整 validator。
- 晋升规则已补上 Event/Audit Record → Snapshot → Rule、anti-pattern、冲突裁决，但“同一 hypothesis”的匹配标准、跨项目引用的记录方式、逻辑矛盾的检测方法还未定义。

结论：P4 从不可用推进到可试跑，但不是完全闭环。

### P5：存储层/注入层分离 — 确认修复

草案多处明确：

- §9.7 只约束注入层 compact view，不约束存储层。
- 存储层完整保留 REO、事件日志、审查报告。
- §9.8 通过 `compact_view` 连接存储层与注入层。
- 向 AI 上下文注入时只注入 `compact_view`，不注入完整 REO。

第一轮指出的“500 字符上限可能污染事件本体”的问题已解决。

### P6：来源证据校准 — 确认修复

草案已明确：

- 4,590 是 retained analyses，不是 4,590 个独立随机对照试验。
- 原论文缺置信区间、p 值、效应量、多重比较校正。
- 仅限 Gemini 3.1 系列与科学代码求解任务。
- 公开仓库缺实验配置和原始数据，核心算法混淆。
- 按 §9.6 标注为 [Sp]。

这是本轮最明确的实质改进之一。

### P7：补充遗漏 — 基本修复

草案补充了成本效率视角：token 节省作为 §9.7 A/B 测试的次要指标，且“IC 无差异但 token 节省 ≥20%”可作为成本优化建议有条件采纳。

实施前验证清单也明显补齐，包括 dry-run、误报率、JSON Schema validator、无效样例拒绝测试、compact_view 抽检、晋升路径演练、预注册引用完整性、敏感性分析、术语检查等。

剩余缺口是人工成本未显式量化：dry-run 后应记录“每次告警的人类审查耗时”和“误报处理成本”，否则无法判断监测层是否引入过高维护负担。

### P8：A/B 测试设计完善 — 部分修复

相较旧版，A/B 设计已补上样本量、重复次数、固定模型、随机种子、prompt 控制、指标、最小有意义效应、失败判据与部分通过条件，已可作为 pilot 方案。

但它还不足以支撑从 [Sp] 升级到稳健 [J]：

- 未指定统计检验方法、配对设计、置信区间或 bootstrap。
- 未定义任务集抽样、训练/验证/样本外切分、防数据泄漏检查。
- 对量化任务而言，缺交易成本、换手率、样本外 Sharpe、因子相关性/拥挤度等控制指标。
- “30 任务 × 3 重复 × 2 arms = 180 次生成与评估”，草案估时 4-6 小时偏乐观，除非已有全自动评估流水线。

结论：可执行，但只能支持“试跑判断”，不能单独支撑正式方法论升级。

### P9：数值可配置化 + 敏感性分析 — 确认修复

`delta_success`、`delta_failure`、硬抑制阈值、惯性阈值、500 字符上限、滚动窗口 N、晋升阈值均已标注可配置或要求敏感性分析。草案还明确这些数值来自 Evolver 的设计选择且源码混淆，不可当作可验证事实。

## 维度2：逻辑自洽性

1. §3.7.0 三条规则在 dry-run 模式下基本自洽。规则1不再抑制日志，规则2仍能看到连续 repair；规则2也不再自动插入 explore，因此不会改变规则3看到的空周期序列。剩余缺口是 `event_stream_health` 的计算公式未定义，只给了阈值区间。

2. 存储层/注入层分离清晰，且与 §9.7、§9.8 内部一致。`compact_view` 是合理接口。

3. REO 晋升规则已经从叙事闭环变成半正式闭环，但还缺三类关键机制：hypothesis identity resolution、跨项目引用计数来源、规则冲突检测口径。没有这些，Phase 3 的“自动推荐 + 冲突检测”仍会退化成人工判断清单。

4. §3.7.4.1 与 §3.7.0 的接口基本明确，但“抑制期”来源不清。如果 §3.7.0 只建议、不执行，§3.7.4.1 中“被标记的资产在抑制期内的成功率不计入 weight 更新”必须说明抑制期只由人工确认创建。

## 维度3：可操作性

1. dry-run 路径比旧版自动干预路径可行得多。事件日志 schema 仍是前置依赖，草案已把它放到实施顺序第一步，这是正确的。

2. REO Schema 足够支持 Phase 1 的手动试用，但还不够支持完整自动化。至少应补一个 Audit Record JSON Schema，并说明 Markdown schema 与配套 JSON 哪个是 source of truth。

3. A/B 测试已经具备执行骨架，但缺统计判定细节。建议升级前预注册：配对任务、主要检验、CI/Bootstrap、样本外窗口、交易成本、失败优先判据。

4. 实施前验证清单覆盖了第一轮指出的大部分风险。建议再加入三项：人类审查耗时、schema migration 兼容策略、自动淘汰/自动推荐不会改变运行行为的回归检查。

## 维度4：来源忠实性

1. 来源声明基本忠实反映了 Evolver 项目的低质量评价。草案没有再把 4.1-4.2/10 包装成方法论背书。

2. [Sp] 标注恰当。当前证据最多支持“可作为启发和试跑假设”，不支持“已验证方法论”。

3. 草案仍引用 Evolver 作为思想来源，但已经用负面限定包住引用，过度背书风险显著下降。

4. 二次确认偏误仍有轻微残留：header 的“独立审查链”和 §9.7 的“经 Qwen 和 Codex 两轮独立审查确认”容易被误读为“原则被确认”。更准确的写法是“证据限定被两轮审查指出/确认”，不要让审查链成为机制有效性的背书。

## 维度5：总体评估

- 综合评分：7.2/10
- 与第一轮（4.3/10）的对比：提升约 +2.9 分。修正是实质性的，不是表面替换。
- 是否建议合并：修改后通过。

### 剩余问题

1. 把 §3.7.4.1 的“自动淘汰/移出活跃资产池”改成“候选标记 + HG 人工确认”，否则会在新位置重演“检测层改变执行行为”的边界问题。
2. 为 Audit Record 补正式 JSON Schema，并明确配套 JSON 是否应承载完整 schema。
3. 定义 `event_stream_health` 计算公式，以及它映射到 §3.7.3 普通监测项/高权重监测项的精确规则。
4. A/B 测试补统计方案、样本外验证和量化特有风险控制，避免把 pilot 结果过早升级为 [J]。
5. 清理 Gene/Capsule/Event 作为运行时字段的残留；迁移说明可保留一次，但不要与“无残留”检查项冲突。
6. 补 REO 晋升闭环的身份匹配、引用计数、冲突检测和人工确认审计记录。

### 最有价值的改进

1. 版本编号和主文档结构已对齐，旧版最基础的合并阻断消失。
2. §3.7.0 从自动干预改为 dry-run 观测与 HG 建议，符合“只观测不阻断”的核心边界。
3. 存储层/注入层分离清晰，500 字符规则不再污染完整记录。
4. 来源证据校准显著改善，retained analyses、单模型家族、无统计推断、源码混淆等限制被明确写出。
5. REO 从叙事性文件夹设计推进到可验证 schema + 晋升规则，已经具备小规模试跑价值。
