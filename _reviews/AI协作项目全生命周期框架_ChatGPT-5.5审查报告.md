# ChatGPT-5.5 独立审查报告

**审查对象**：AI协作项目全生命周期框架 v1.1  
**审查日期**：2026-06-11  
**审查者**：ChatGPT-5.5（独立第二意见，未参与框架设计）  
**审查用时**：约 45 分钟  

---

## 一、总体评价

这个框架最强的部分是把 AI 协作项目拆成了可讨论、可维护的几个层次：Spec、Prompt、Loop、Workflow、Retrospect、Closure，再用 Human Gate 把人类决策权插入关键节点。作为个人或小团队的工作方法，它已经比“边聊边做、靠记忆交接”的方式成熟很多，尤其是 Spec、Human Gate、会话交接、风险登记和 Closure 清单，能直接减少遗忘、漂移和无休止迭代。

但它目前不应被称为“已定稿的完整生命周期框架”。它更像一个高质量的 v0.7 操作手册：概念骨架明确，模板已经可用，但若直接推广到新项目，会遇到执行成本、触发频率、自动化缺口、概念边界不清和实证验证不足的问题。框架把很多“应该做”的事项列出来了，但还没有证明这些事项会被稳定执行，也没有证明执行成本低于收益。

最弱的部分是两个强断言：第一，外部对标中把 Retrospect、Closure、Dev Log 标为“公开资料未找到形式化对应物”并不成立；公开实践中有 Scrum Retrospective、SRE postmortem、项目关闭流程、ADR、changelog、release notes、runbook、issue tracker 等相近对象。第二，Dev Log 依赖“AI 自动维护”，但当前设计没有强制机制、检测机制或失败补救机制，因此只能作为待验证假设，不能作为可靠基础设施。

我建议继续投入，但投入方向应从“继续扩展层和模板”转为“缩小可执行核心 + 真实项目试跑 + 建立死亡判据”。如果下一轮仍在增加概念，而不是测量漏记率、维护成本、决策质量和交接恢复效率，这个框架会滑向过度工程。

## 二、核心判断逐条评估

### 判断 1：七层架构覆盖完整生命周期 → 部分认同

**信心水平**：高  
**依据**：七层覆盖了项目启动、任务规格、执行迭代、多任务编排、里程碑回顾和闭合，作为 AI 协作项目的主流程是成立的。问题在于它缺少若干项目生命周期中的前置和横向治理环节：机会筛选/需求澄清、利益相关方与外部交付对象、资源预算与成本控制、权限/安全/数据治理、实验验证计划，以及工具自动化与回归测试。部分内容被放在跨层关切里，但没有形成同等强度的流程接口。  
**追问或修正**：不要再增加一个完整“大层”。建议在 L0 Spec 前增加“Intake/Discovery”最小入口，或把它作为 HG-0 的必填子表；同时把权限、安全、预算、验证计划提升为 Spec 必填项，而不是散落在备注中。

### 判断 2：Human Gate 的 4 个闸门位置和触发条件是充分的 → 部分认同

**信心水平**：高  
**依据**：HG-0、HG-1、HG-2、HG-3 覆盖了启动、里程碑、Spec 变更和闭合，主干合理。AI 在闸门处的 5 条职责也基本可执行。但是“任何 Spec 变更都走 HG-2”会制造过高摩擦，小修订、命名约定、路径更新和经验补充不应与红线/目标/评估标准变更同级处理。HG-1 触发频率也偏粗，例如学术项目“每章完成后”不足以覆盖选题、研究问题、文献纳入标准、方法设计、数据解释这些关键价值判断。  
**追问或修正**：把 HG-2 分为低风险批处理、中风险确认、高风险强制闸门。新增“风险升级/权限申请/外部提交”闸门类型。AI 的职责应要求固定输出格式：现状、变更差异、影响范围、可选项、推荐、不可逆后果、需人回答的问题。

### 判断 3：Dev Log 的设计方向正确但成熟度不足，定义为“待实证”是诚实的 → 认同

**信心水平**：高  
**依据**：把变更意图、根因、影响和关联审阅从代码头注释中移出，独立保存为项目级记录，方向正确。成熟度评估中把 AI 自动追加、双视图一致性、FK 健壮性、跨项目泛化和追溯效果列为待实证或零数据，是诚实的。更严格地说，“代码注释不可靠”有合理工程依据，但文档中给出的证据仍偏个案和经验，不足以支撑广泛结论。  
**追问或修正**：Dev Log 的核心假设不是根本不可行，但“仅靠 AI 自觉维护”不可行。应降级为“AI 草稿 + 自动差异检测 + 人类抽查 + 链接校验”的混合机制。至少加入：变更后检查 Git diff 是否有未记录文件、定期校验 dev-logs 链接、Session End 强制检查、漏记率指标。

### 判断 4：Workflow 的 6 种编排模式覆盖实际需求，决策树可用 → 部分认同

**信心水平**：中  
**依据**：六种模式都能映射到真实场景，但作为日常操作手册显得过重。多数项目中真正高频的是三类：pipeline、verification、discovery。barrier parallel、judge panel、loop-until-dry、completeness critic 可以作为子模式，而不是与 pipeline 同级的常用模式。决策树目前也有逻辑问题：先问“子任务之间有依赖吗”再分 pipeline/barrier，容易让人误解 pipeline 只适用于无依赖；实际 pipeline 正是处理阶段依赖的基本方式。  
**追问或修正**：把模式库压缩为“三大模式 + 特殊插件”：执行流水线、独立验证、发现扩展。决策树应先问任务目的：执行、验证、发现、选择方案；再根据成本和风险升级。

### 判断 5：与外部行业实践的对标是充分的 → 反对

**信心水平**：高  
**依据**：外部对标不充分，且“独特贡献”的表述过强。Retrospect 在 Scrum 中是正式事件，SRE/DevOps 中有 blameless postmortem 和 incident review；Closure 在 PMI、PRINCE2 和通用项目管理中有成熟的关闭流程；Dev Log 与 changelog、ADR、release notes、issue tracker、runbook、postmortem、decision log 均有重叠。框架的独特性不在“公开资料没有对应物”，而在“把这些实践组合成个人 AI 协作项目的轻量方法”。  
**追问或修正**：将“公开资料未找到形式化对应物”改为“未找到完全同构于本框架的个人 AI 协作轻量组合”。补充对标：Scrum Retrospective、Google SRE postmortem、Keep a Changelog、Architecture Decision Records、项目关闭流程、AI agent guardrails、LLM eval regression suites。

### 判断 6：框架可以直接用于下一个新项目，不需要重大修改 → 反对

**信心水平**：高  
**依据**：它可以作为新项目的审查清单或骨架，但不能无重大修改地直接用于学术论文、量化策略和工程开发三类项目。附录 A-G 是通用模板，缺少项目类型适配、优先级裁剪和“最低可运行版本”。例如学术论文项目需要研究问题、纳入排除标准、引用真实性、研究伦理、目标期刊、审稿标准；量化项目需要数据泄漏、样本外、交易成本、回测协议；工程项目需要测试策略、发布策略、权限和安全。  
**追问或修正**：为每类项目做一页式启动包：必填 10 项、可选 10 项、禁填项、首个里程碑、最小 Dev Log 规则。先让框架在一个新项目中以 30% 复杂度运行，再逐步启用高级层。

### 判断 7：框架自身的“死亡判据”是明确的 → 反对

**信心水平**：高  
**依据**：框架要求每个项目和 Prompt 有死亡判据，但它没有给自己设置死亡判据，这是内部治理缺口。没有自我否决条件时，框架很容易把失败解释为“执行不到位”而不是“框架过重或无效”。  
**追问或修正**：加入框架级死亡判据：连续 3 个真实项目使用后，交接恢复时间、返工率、漏记率或交付质量没有改善；或维护框架耗时超过项目总工时 15%；或用户连续 2 个项目绕过 50% 以上模板；或 Dev Log 漏记率超过 20% 且无法通过自动化降到 5% 以下，则必须大修或降级。

## 三、结构性遗漏

1. **需求/机会入口**：框架从 Spec 开始，但没有定义如何判断一个项目是否值得进入 Spec。缺少问题定义、用户/读者/评审对象、交付场景、机会成本和替代方案扫描。
2. **资源与成本治理**：没有项目预算、token/时间/人力成本、模型调用成本、认知负担上限。复杂模板本身可能成为主要成本。
3. **权限、安全与数据治理**：未来迭代方向提到权限分级和 guardrail，但当前不是正式层。对文件写入、联网、数据隐私、敏感信息、外部提交、凭据处理没有硬闸门。
4. **验证/回归基础设施**：框架有评估理念，但没有把 golden set、回归测试、可重复 benchmark、自动链接校验、文档一致性检查纳入核心流程。
5. **利益相关方与交付对象**：学术导师、审稿人、竞赛平台、业务方、代码用户等外部评价者没有明确位置。
6. **模型与工具版本管理**：框架强调 memory 和 Spec，但没有规定模型版本、prompt 版本、agent 配置、工具权限和运行环境变化如何记录。

## 四、内部矛盾与张力

1. **“已定稿”与“待实证”冲突**：文档状态说已定稿，Dev Log 又有 7/10 待实证或零数据。更准确的状态应是“框架草案冻结，等待试跑验证”。
2. **Human Gate 不是独立层，却被纳入七层**：文档称 Human Gate 是跨层闸门、非独立层，但又在七层速览中作为 L-H 层呈现。建议改称“六层生命周期 + 一组人类闸门”。
3. **正常 Loop 不记日志，但 Retrospect 要回溯细节**：如果正常迭代不记录，后续 Retrospect 很可能只能依赖聊天记录和记忆，跨会话时尤其脆弱。
4. **AI 自动维护 Dev Log 与人类审查责任边界不清**：文档说人只在 HG-1 抽查，但又承认 HG-1 频率可能不足。若漏记发生，谁负责恢复事实链不明确。
5. **所有 Spec 变更走 HG-2 与高频反向沉淀冲突**：Retrospect 鼓励频繁写回 Spec，但 HG-2 对所有大小变更一视同仁，会压制实际执行。
6. **永久保留 memory 与 30 天时效提醒可能制造噪音**：永久不删会让 memory 老化和膨胀；只靠提醒“需验证”不足以维持质量。

## 五、过度工程

1. **Workflow 六模式并列**：对个人工作流来说过重，应压缩为少数默认路径，复杂模式作为升级选项。
2. **Dev Log 双视图 + FK 导航**：概念有价值，但手工或纯 AI 维护成本高，容易出现链接断裂和索引不一致。
3. **所有闭合都要求多件终期产物**：`.md + .json + docx + 外部审阅 + ≥3 条 memory` 对小项目过重，应按项目等级分档。
4. **Retrospect 每次至少 1 条失效和 1 条意外**：能防走过场，但也可能诱导制造伪发现。建议改为“必须显式搜索，找不到时记录搜索范围和证据”。
5. **所有 Spec 修改强制 HG**：会让低价值变更排队等待人类确认，削弱 AI 协作效率。
6. **30 天 memory 自动提醒**：若没有清理、归档、降权机制，会变成低信噪比噪声。

## 六、概念模糊

1. **项目**：什么规模算项目？一次论文修改、一个章节、一个策略版本是否都是项目？
2. **里程碑**：HG-1 依赖里程碑，但里程碑粒度没有判定标准。
3. **重大/重要/关键**：重要 Bug、重大重构、关键产出、关键决策没有可操作阈值。
4. **有意义的变更**：Dev Log 记录“每个有意义的变更”，但意义由 AI 判断，边界不稳。
5. **独立模型/独立 Agent**：不同模型、不同 prompt、不同上下文隔离各自算不算独立，没有标准。
6. **完成标准 vs 成功标准**：Prompt 的完成标准与 Spec 的项目成功标准容易混用。
7. **Retrospect 候选**：什么从候选升级为 Spec，什么只留在日志，缺少决策规则。
8. **闭合、暂停、封存、重启**：这些状态之间的转换规则不够清楚。

---

## 七、缺失的跨层交互

1. **Retrospect → Spec**：有流程描述，但缺少变更 schema、影响评估模板、版本号和审批记录。
2. **Dev Log → Retrospect**：说 Retrospect 从 Dev Log 提取模式，但没有定义提取查询、统计口径和证据标准。
3. **Risk Register → Human Gate**：只有 H+M 触发规则，缺少风险升级后如何改变 Prompt、Workflow 或 Closure 条件。
4. **Evaluation → Closure**：评估指标如何成为闭合证据不够明确。Closure 应引用具体指标达成情况，而不是只写总结。
5. **Session Handoff → Dev Log/Spec**：会话结束时哪些改动写 project_status，哪些写 Dev Log，哪些写 Spec 候选，三者边界需要表格化。
6. **Workflow → Prompt**：Workflow 失败后如何重写下一轮 Prompt，没有明确回流接口。
7. **Git → Dev Log**：两者互补，但缺少自动对账机制。没有对账，Dev Log 漏记会长期存在。

## 八、可测试性与验证方案

可实证检验的断言包括：

1. 使用框架是否降低跨会话恢复时间。
2. Dev Log 是否降低 Bug 根因追溯时间。
3. Human Gate 是否减少方向性错误和无效迭代。
4. Workflow adversarial verify 是否比单模型审查发现更多高严重问题。
5. Retrospect 写回是否提高下一项目的执行质量。
6. Closure 清单是否减少未归档、未清理、未交付的遗漏。

偏信念或偏好、尚不能直接证明的断言包括：

1. 七层就是完整生命周期。
2. 6 种 Workflow 模式是必要集合。
3. Retrospect 和 Closure 是本框架独特贡献。
4. 30 天 memory 提醒是合适阈值。
5. 所有 Spec 变更都必须人审。

建议的验证实验：

1. **历史回放实验**：选 2 个已完成项目，用框架重建其 Spec、Gate、Dev Log、Retrospect，测量能否准确恢复关键决策、Bug 根因和闭合证据。
2. **新项目 A/B 试跑**：选 2 个相似规模新项目，一个用完整框架，一个用最小版框架，比较完成时间、返工次数、交接恢复时间、产出质量、人类确认次数。
3. **Dev Log 漏记测试**：连续 10 次真实变更后，用 Git diff 和文件时间线对账，统计 AI 漏记率、错误链接率、主文档/分支文档不一致率。
4. **审查有效性测试**：同一交付物分别经单模型审查、adversarial verify、human review，比较高严重问题召回率和误报率。
5. **成本收益记录**：每个项目记录框架维护耗时、报告/模板字数、HG 等待次数，和实际减少的返工时间比较。

## 九、框架死亡判据建议

建议把以下判据写入框架自身的 L0：

1. **无改善判据**：连续 3 个真实项目使用框架后，交接恢复时间、返工次数、漏交付项、Bug 追溯时间没有相对历史基线改善至少 20%，则大修。
2. **维护成本判据**：框架维护和填表耗时超过项目总工时 15%，且没有可量化收益，则降级为最小版。
3. **绕过率判据**：连续 2 个项目中，用户或 AI 实际绕过超过 50% 的模板/闸门，说明框架不适配实际工作，应重构。
4. **Dev Log 失败判据**：10 次真实变更中漏记超过 2 次，或链接错误超过 10%，且无法通过自动化校验改进，则取消“双视图+FK”方案，退回单一时间轴。
5. **Human Gate 过载判据**：HG-2 变更确认中超过 70% 被人直接批准且无实质讨论，说明闸门过密，应分级。
6. **外部审查无增益判据**：连续 3 次独立模型审查没有发现人类或后续执行认为有价值的问题，则降低审查频率或更换审查方式。

## 十、优先级改进建议（Top 3）

| 优先级 | 问题 | 建议方案 | 理由 |
|--------|------|---------|------|
| 1 | 框架太完整但未证明可执行 | 做“最小可运行版”：L0 必填 6 项、HG 3 类、Workflow 3 模式、Dev Log 单时间轴、Closure 分级 | 先降低执行成本，才能真实试跑并收集数据 |
| 2 | 缺少实证验证和自我死亡判据 | 在下一个项目中预登记指标：交接恢复时间、漏记率、返工次数、审查发现率、维护耗时，并写入死亡判据 | 避免框架只能继续自我扩展，不能被证伪 |
| 3 | Dev Log 可靠性不足 | 加自动对账：Git diff 未记录文件检查、链接校验、Session End 强制清单、HG-1 抽查记录 | Dev Log 是新增核心产物，若不可靠，会污染 Retrospect 和 Closure |

## 十一、补充意见

### 对提示词本身的审查

提示词质量较高，明确要求独立审查、信心水平、已证/未证区分和开放问题，这有助于降低自我正当化。但它也有三点引导性：

1. 7 个判断的措辞大多先陈述框架结论，再要求审查，容易让审查者围绕设计者的问题域打转。
2. “不要同情设计者”的措辞可能造成反向偏差，让审查者为了显得独立而过度否定。
3. 提示词称审查者为 ChatGPT-5.5 并要求固定身份，但没有要求披露实际执行环境、可用工具和外部检索范围。

### 外部对标补充依据

以下公开实践足以反驳“Retrospect/Closure/Dev Log 没有形式化对应物”的强表述，但不否定本框架的组合价值：

- Scrum Guide: Sprint Retrospective 是正式 Scrum 事件，见 https://scrumguides.org/scrum-guide.html
- Google SRE: Postmortem Culture 形式化了事故复盘和组织学习，见 https://sre.google/sre-book/postmortem-culture/
- Keep a Changelog: 面向人类可读的变更日志规范，见 https://keepachangelog.com/en/1.1.0/
- Architecture Decision Records: 记录架构决策的轻量文档实践，见 https://adr.github.io/
- PMI 项目管理资料中有项目关闭/阶段关闭实践，见 https://www.pmi.org/learning/library/importance-of-closing-process-group-9949
- PRINCE2 项目关闭流程也包含验收、移交、评估和关闭建议，见 https://www.prince2.com/usa/blog/prince2-project-closure

本框架真正值得保留的定位不是“别人没有”，而是“把既有工程、项目管理和 AI 协作实践裁剪成个人 AI 项目的统一操作手册”。这个定位更诚实，也更抗审查。
