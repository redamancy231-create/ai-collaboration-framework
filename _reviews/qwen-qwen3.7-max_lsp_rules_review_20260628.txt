# LSP 优先约束段落独立评审报告

**审稿人**：Qwen (qwen3.7-max)
**评审日期**：2026-06-28
**被评估对象**：CLAUDE.md 中"LSP 优先约束"段落

---

### 1. 决策框架完备性

四级分流（必须LSP → 倾向LSP → 倾向grep → 直接grep）在逻辑上覆盖了主要场景，但存在以下问题：

**灰色地带一：跨文件语义追踪**。当 Agent 需要"找到所有调用某函数的地方，且项目有 33 个 .py 文件分散在多个子目录"时，这属于"必须 LSP（找引用）"还是"倾向 grep（文件量大 >5）"？两条规则给出了矛盾指引。findReferences 是语义级操作（必须 LSP），但文件量又超过了倾向 grep 的阈值。规则没有说明当"必须"和"倾向 grep"的条件同时满足时谁优先。

**灰色地带二：文件数量阈值 5 的合理性**。项目实际有 33 个 .py 文件，但多数脚本是独立工具（`_workflows/` 下互不依赖），Agent 在大多数任务中实际涉及的代码文件通常 ≤5。阈值 5 是"被调用文件数"还是"项目总文件数"还是"任务涉及文件数"？未明确定义，Agent 可能困惑。

**灰色地带三：混合任务**。"理解一个函数如何被其他脚本调用，然后修改它"——前半段是 findReferences（必须 LSP），后半段可能涉及 grep 搜索所有调用点做批量替换。规则未指引混合场景。

**边界清晰度**：四级名称中"必须"和"倾向"的语义差异是清晰的（hard vs. soft constraint），但两个"倾向"级别之间（倾向 LSP vs. 倾向 grep）的分界线——"正则精度够不够"——本身需要 Agent 做主观判断，这构成了一个递归问题：Agent 需要先理解代码才能判断正则精度，但理解代码正是它要选工具去做的事。

### 2. 脆弱性测试

**(a) 必须 LSP**
- **删掉后 Agent 会不会犯错**：**会**。
- **犯错路径**：删掉后 Agent 会用 grep 搜索变量引用（如搜 `config`），结果把注释中的 `# config 文件路径`、字符串中的 `"config.json"`、以及同名的不相关变量全部命中，导致 Agent 基于错误引用集做出修改决策。在本项目的 `_workflows/i18n/` 目录下多个脚本共享类似命名模式（如 `fix_` 前缀函数），grep 无法区分定义与引用。

**(b) 倾向 LSP**
- **删掉后 Agent 会不会犯错**：**不确定**。
- **理由**：这条本质上是 (a) 的弱化版加上"小文件量"的效率提示。删掉后，Agent 在 (a) 覆盖的语义场景中仍受"必须 LSP"保护；在"正则精度不够但非语义级"的场景中，一个能力足够的 Agent 通常能在 grep 结果中发现误匹配并自行切换。但"顺带返回 pyright 诊断信息"这个附带收益提示会被丢失——Agent 可能不知道 LSP 调用有零成本的诊断附赠。不过这个信息属于效率优化而非防错，不构成犯错路径。

**(c) 倾向 grep**
- **删掉后 Agent 会不会犯错**：**会**。
- **犯错路径**：删掉后 Agent 在"搜所有函数定义"这类任务中会逐文件调用 LSP:documentSymbol，33 个文件 = 33 次工具调用，严重拖慢任务执行。更危险的是，Agent 可能因调用次数过多而触发上下文窗口限制或超时，导致任务中途中断。本项目 CLAUDE.md 本身就提到 LSP 是"逐文件调用"，这条规则的存在直接防止了已知的性能陷阱。

**(d) 非代码理解直接 grep**
- **删掉后 Agent 会不会犯错**：**会**。
- **犯错路径**：删掉后 Agent 可能对"搜索 TODO 标记"或"搜索特定中文术语"这类纯文本任务也触发 LSP 优先逻辑，浪费时间且可能因 pyright 无法解析非代码内容而返回空结果。考虑到本项目核心交付物是 Markdown 文档，Agent 频繁需要在 .md 文件中搜索文本，这条排除规则防止了对非代码文件误用 LSP。

**(e) 透明度规则**
- **删掉后 Agent 会不会犯错**：**不会**（就任务执行质量而言），但**会损失可审计性**。
- **理由**：透明度规则不影响 Agent 的工具选择质量——它是事后标注，不是事前决策。删掉后 Agent 仍能正确选择工具，但人类无法事后追溯"为什么 Agent 在这里用了 grep 而不是 LSP"。在本项目的多模型审查闭环中（§9.2），这种追溯能力对验证 Agent 行为合理性有价值。但严格来说，这属于"可维护性"而非"防错"，按脆弱性测试的严格标准，这条可能被视为"不会犯错"。

### 3. 五滥检查

**可推导信息**：部分存在。`.lsp.json` 的内容（pyright + .py/.pyi 映射）是 Agent 读文件即可获取的信息，CLAUDE.md 中写"本项目已配置 `.lsp.json`（pyright）"属于轻度可推导信息。但考虑到这节省了 Agent 一次文件读取，且放在上下文中帮助理解后续规则的适用范围，代价可接受。**建议**：保留"已配置 pyright"的事实陈述，但不需要展开 `.lsp.json` 的具体内容（当前写法已克制，未犯此问题）。

**通用惯例**："LSP 用于语义理解、grep 用于模式匹配"是开发工具领域的基本知识，接近通用惯例。但本文的价值不在于陈述这个常识，而在于**给出了具体的分流阈值和操作指引**（如文件量 ≤5、行首锚定标准），这些是项目定制内容，不算通用惯例。**结论**：未犯此问题。

**易变信息**：未检测到。规则未绑定特定版本号或时间敏感内容。

**文件详述**：未检测到。没有罗列式文件描述。

**空泛建议**：未检测到。每条规则都给出了具体场景和判断标准，没有"遵循最佳实践"类废话。

**额外发现**：透明度规则的格式示例（`> 工具: LSP:documentSymbol ×21 | 理由: ...`）中"×21"的含义不够直观——它表示"调用了 21 次"还是"涉及 21 个文件"？这是一个微小但值得修正的歧义。

### 4. 透明度规则可执行性

**首次接触的 Agent 能否正确理解并遵守**：**部分能**。

**正面**：规则给出了两个完整的格式示例，覆盖了 LSP 和 grep 两种情况，模板感强，Agent 可以照搬格式。

**问题**：
1. **触发条件模糊**："每次代码理解任务的结果中"——什么算"代码理解任务"？搜索 TODO 标记算不算？搜索函数名算不算？如果 Agent 在一个任务中同时做了代码理解和文本搜索，是否需要对每个工具调用分别标注？
2. **标注粒度不明**：如果一个任务调用了 3 次 LSP:goToDefinition 和 1 次 grep，是写一条汇总还是四条分别标注？示例中的"×21"暗示汇总，但未明说。
3. **位置不明**：标注应该放在工具调用前（计划阶段）、调用后（结果阶段）、还是最终输出的末尾？
4. **格式中的 `>` 符号**：在 Markdown 中是引用块语法，Agent 可能理解为"用 blockquote 格式输出"，但在纯文本对话中这个符号可能显得突兀或与其他 Markdown 结构冲突。

**建议修改**：

**Before**：
```
**透明度规则**：每次代码理解任务的结果中，必须显式标注实际使用的工具（`[LSP:documentSymbol]` / `[Grep]` 等）及选择理由。格式：`> 工具: LSP:documentSymbol ×21 | 理由: 单文件符号提取，顺带拿诊断` 或 `> 工具: Grep | 理由: ^def 正则精度足够 + 21 文件一次覆盖`。方便事后追溯和审计工具选择是否合理
```

**After**：
```
**透明度规则**：代码理解任务完成后，在结果末尾附一行工具选择摘要。格式：`> 工具: LSP:goToDefinition ×3, Grep ×1 | 理由: 跳转定义需语义解析(3处)；TODO搜索用正则一次覆盖`。非代码理解的纯文本搜索无需标注
```

修改要点：明确标注位置（结果末尾）、明确粒度（一次任务一条汇总）、明确排除条件（非代码理解不标注）、修正"×21"歧义。

### 5. 自反性检验

**问题**：Agent 在判断"该用什么工具来理解 CLAUDE.md 中的指令"时，按这套规则应该选什么？

CLAUDE.md 是 Markdown 文件，不是代码文件。Agent 读取 CLAUDE.md 属于"非代码理解任务（纯文本搜索）"，按规则应"直接 grep，不触发 LSP 优先"。这是自洽的——pyright LSP 只配置了 .py/.pyi 扩展名，对 .md 文件本来就无能为力。

**更深层的自反性**：本项目是方法论文档项目，33 个 .py 文件大多是独立工具脚本，不存在深度耦合的类继承链或复杂的 import 图。这意味着在本项目的实际场景中，"必须 LSP"所列的典型场景（找引用区分变量/注释、跳转定义解析继承/作用域、接口实现跳转）极少发生。规则的防御强度与项目的实际风险不匹配——为一个很少出现的场景设置了硬约束。

**这不是错误**，而是合理的预防性设计：即使当前代码库简单，未来可能增长；且 CLAUDE.md 作为操作指令应覆盖可能的最复杂场景。但值得在规则中承认这一现实。

**结论**：逻辑自洽，无自反矛盾。

### 6. 盲点检测

**混合使用指引**：**确实遗漏**。如第 1 维度所述，"先用 LSP:findReferences 找所有语义引用 → 再用 grep 在结果集里过滤"是合理的工作流，但规则未提及。建议增加一条：
```
- **混合使用**：同一任务可组合 LSP 和 grep。典型模式：LSP 定位语义目标 → grep 在已知范围内做批量文本操作。透明度规则中分别列出即可
```

**"需要 LSP 语义能力但文件量很大"的中间地带**：**确实遗漏**。当前规则中"必须 LSP"和"倾向 grep（文件量大 >5）"在文件量大时产生矛盾。建议明确：**语义级操作（必须 LSP 类）不受文件量阈值约束**——即使有 50 个文件，findReferences 仍然比 grep 准确，效率损失是可接受的代价。

**透明度标注截断/丢失风险**：**低风险但存在**。在长对话中，Agent 的工具调用日志可能被上下文压缩截断，透明度标注如果嵌在中间输出里可能丢失。将标注固定在"结果末尾"（如第 4 维建议）可缓解此问题。另外，如果 Agent 使用结构化输出（如 JSON），这种纯文本标注可能不适配——但本项目不涉及结构化输出管道，此风险可忽略。

### 7. 反直觉发现

**反直觉但判断正确的设计**：

"倾向 grep"中对 `^def ` 搜索函数定义的推荐是一个反直觉但正确的设计。主流观点倾向于"LSP:documentSymbol 更权威"，但在本项目场景下——33 个独立脚本、无装饰器嵌套、无动态代码生成——`^def ` 的正则精度接近 100%（行首锚定 + Python 语法中 `def` 只出现在函数定义处），而 LSP 需要逐文件调用 33 次。**效率收益远超精度损失风险**。这个判断体现了对项目实际特征的深入理解。

**判断可能有误的地方**：

"正则精度判断标准：行首锚定 + 关键字唯一 + 语法上几乎不可能出现在注释/字符串中"——这个标准本身很精确，但它要求 Agent 在选工具前先评估正则精度，而评估正则精度需要理解目标代码的语言语法。对于 Python 这可能不是问题（Agent 通常精通 Python 语法），但如果项目未来加入其他语言的脚本（如 .sh、.ps1），这个标准的适用性未明确。考虑到 CLAUDE.md 的环境声明只提到 Python + Node.js（mmdc），当前范围是合理的，但可以在标准中加一句"本标准针对 Python 语法"以明确适用边界。

---

## 总结

**最高优先级修改建议（≤3 条）**：

1. **解决"必须 LSP"与"文件量大 >5"的优先级冲突**。明确声明：当任务属于"必须 LSP"类（语义级操作）时，文件量阈值不适用。当前写法中"倾向 grep"的文件量条件可能覆盖"必须 LSP"的语义条件，导致 Agent 在大规模代码库中错误地用 grep 替代 findReferences。

2. **精简透明度规则并明确执行细节**。当前规则的触发条件、标注粒度、输出位置均存在歧义。建议改为"代码理解任务完成后在结果末尾附一行汇总"，并明确排除非代码理解任务。同时修正"×21"的歧义（次数 vs. 文件数）。

3. **增加混合使用指引**。补充一条关于 LSP + grep 组合使用的简要说明，覆盖"先语义定位后批量操作"的常见工作流。当前四级分流假设每次任务只用一种工具，与实际工作模式不符。

**整体评价**：这是一段高质量的操作性规则，分级逻辑清晰，避坑指向明确，没有犯"五滥"中的任何严重问题。核心缺陷集中在边界条件（优先级冲突、混合使用）和执行细节（透明度格式）上，均为可通过小幅修改解决的问题。规则的设计判断——特别是"倾向 grep"的反直觉推荐——体现了对项目实际特征的务实分析，而非照搬"LSP 总是更好"的教条。
