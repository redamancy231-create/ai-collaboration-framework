**整体结论**：GLM-5.2 的编辑大体遵守冻结期方向，没有新增主框架 [Sp] 节；但 **F3 provenance 事实错误**、**F9/F10 引入了实质性编辑判断却在主披露中说"未引入新实质判断"**，需要修正。

### 未声明实质判断汇总

主要有四处，其中 F9/F10 最重要：

1. **F9**：成熟度分级（每行"已验证/待实证/零数据"）+ 分布估计（~30/40/30）
2. **F10 协议 2**：Qwen 冷读 prompt 设计 + 命中判读表 + 严重性降级规则
3. **F7**："保留但降级"而非"删除监测项"的方案选择
4. **F2**：headroom JSON 的结构化证据等级重建（`rubric_purity=[SEMI]` 等）

### provenance 准确性

- [SEMI-ED] 作为"编辑独立但任务由用户指定"的定性**基本准确**
- 但"第四个后端"计数不准（至少是第五/第六个）、"三 house"遗漏 Kimi
- 核心混淆：将 **model 后端**、**CLI house**、**审查者** 三个概念混用

### 编辑质量

- **质量较好**：F7（降级逻辑清晰）、F8（单一事实源头注简洁）、F6（工具替换理由充分）
- **质量问题**：
  - F3 事实错误（遗漏 Kimi）
  - F4 重复粘贴（provenance 注意出现两次）
  - F9 日期/角色措辞不一致
  - F10 协议 2 的强动作规则未充分标为"待复核"

### 冻结期纪律合规

**没有继续往主框架加 [Sp] 节**，纪律总体比 v1.5.1 前好。真正的边界风险是 **F10**：它以"待执行协议清单"名义新增了可改变 OPEN-1 严重性和 §3.7 命运的判读规则——如果该 prompt 设计有偏，冷读结果会误导框架对 OPEN-1 的定性。

## 修正建议

1. **修正 provenance 声明（F1/F3/F11/JSON）**：
   - 明确列出 Claude / Codex / Kimi / Qwen / ZCode **五个 house**
   - 区分"模型后端"和"CLI house"两个概念
   - "第四个后端"→删除或改准确计数

2. **修改利益披露措辞（F1 版本头 + §14 表头 + JSON edit_history）**：
   - 删除"未引入新的实质审查判断"
   - 改为："引入了已标注的编辑者判断（F9 成熟度评估、F10 协议 2 prompt/判读规则），待独立复核"

3. **F10 协议 2 加安全锁**：
   - 标注为"待复核草案"
   - 在独立审查通过前，**不要执行其降级动作**（OPEN-1 降为低/§3.7 退役为可选）
   - 冷读 prompt 本身应先经独立审查确认不会污染方向

4. **F9 细节修正**：
   - 补精确计数口径（各成熟度级别的具体行数，而非约数）
   - 修正 MD/JSON 日期不一致
   - 统一"GLM 编辑者评估"措辞（不与"作者自评"混用）

5. **§14 清理**：
   - 删除 v1.5 节重复粘贴的 provenance 注意段
   - 重排时间线为严格倒序（v1.5.1 → v1.5 → v1.4 → v1.3 → v1.2 → v1.1）
