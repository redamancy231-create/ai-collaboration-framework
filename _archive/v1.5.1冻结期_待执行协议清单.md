## 协议 1：OPEN-4 试读计时（执行优先级：最高，成本最低）

**原始协议**：`_protocols-and-tools/AI协作项目全生命周期框架_OPEN4试读计时协议.md`（完整，已预冻结判据）

### 为什么是最高优先级
- **成本极低**：1 个测试者、1 次、半天内
- **信息量极大**：直接产出"非设计者上手时间"硬数据，这是框架采用性的核心指标
- **零设计成本**：协议已写好，判据已冻结（T1>2h 不合格 / 误解点≥2 落核心 / 理解检查<2/3）
- **不可被 AI 替代**：协议明确排除 AI 测试——它测的是人类上手成本

### 执行步骤
1. 找一位**没参与框架设计、没读过审查链**的人（贴近目标用户：个人做项目、有基本 AI 协作经验）
2. 给他框架主文档（.md 或 .docx）+ 任务设定："读到你**能照着最低强制版（§1.4）起一个新项目**"
3. **不给**：审查报告、本协议判据、任何口头导读
4. 计时开始；主持人全程不解释，只记录卡点
5. 测试者自认"能开始用了" → 记 T1
6. 做 3 题理解检查（§4）：通过 → 记 T2
7. 对照预冻结判据给 PASS / 改进 / 重做

### 输出写回
- 一页记录（T1/T2/误解点清单/三态裁定）→ 写回 `../框架级成熟度评估表.md` 第 1 节 L1 Prompt 行 + §1.6 OPEN-4 状态
- 若"改进/重做" → 起一次 §7.4 Retrospect

### 待解阻塞
**找谁**：需要一个非设计者。这是唯一的外部依赖。候选：
- 任何不熟悉本框架的 AI 协作实践者

### 执行指令（冷读 prompt，经 Kimi-K2.7-Code 偏置审查修正，v2）

> **原版（GLM-5.2 撰写）偏置问题**：经 Kimi-K2.7-Code 六维审查，原 prompt 存在方向性泄漏（"方法论审阅者""结构性缺口"收窄至元机制空间）、确认偏误放大（强制 3 个破坏阴性解释力）、作者意识诱导（"如 §1.6 的待决项"提供具体锚点）。详见 `_reviews/kimi_prompt_bias_review_report.md`。
> **当前版本（v2）**：Kimi-K2.7-Code 改写，消除上述 5 类偏置。允许阴性结果、移除具体锚点、身份降级为"文档审阅者"。

把以下 prompt 喂给一个**不读作者 memory、不看作者审查 prompt** 的独立后端（推荐 Qwen3.7-Max，已有 R5 渠道）：

```
你是一位文档审阅者。请阅读以下方法论文档，并评估其是否存在重要设计缺陷或风险。

【审查对象】
AI协作项目全生命周期框架.md（随附）

【审查任务】
1. 通读文档后，判断是否存在你认为重要的设计缺陷或风险（不是措辞问题、不是错别字）。
2. 如果存在，列出最多 3 个，按重要性排序；如果不存在或不足 3 个，如实报告即可。
3. 对每个缺陷或风险说明：
   - 它是什么；
   - 你为什么认为它重要；
   - 文档中是否已提到或部分涉及该问题。

【审查约束】
- 只基于这份文档本身判断，不引入外部知识。
- 如果文档中已经提到某个问题，请如实标注，并可说明文档的处理是否充分。
- 保持中立、具体，避免泛泛批评或夸大。
- 如果没有发现足够重要的设计缺陷，请明确说明"未发现足够重要的设计缺陷"。

【输出格式】

总体判断：文档是否存在重要设计缺陷或风险？（是 / 否 / 不确定）

缺陷/风险1：[标题]
  - 是什么：
  - 为什么重要：
  - 文档是否已涉及：
  - 重要性评分（1-10）：

（缺陷/风险2、3 同结构；如无，可省略）

未发现的说明：
如果你认为文档整体上没有重要设计缺陷，或只发现不足 3 个，请在此简要说明理由。
```

### 关键判读规则

审查完成后，看审查者报告的缺陷/风险中**是否包含 OPEN-1 的同构问题**（即"离散审查 vs 连续执行 / 人审频率不足 / 方向漂移未被覆盖"）。注意审查者可能报告 0-3 个缺陷——新 prompt 允许阴性结果。

| 审查结果 | 判读 | 动作 |
|---------|------|------|
| **未发现足够重要缺陷**（总体判断=否/不确定） | 冷读未发现结构性威胁——框架无致命设计缺陷（至少对该审查者而言） | §1.6 OPEN-1 严重性降级为"低"；标注"独立冷读未命中"；OPEN-1 可能是作者自识偏置 |
| 命中 OPEN-1（评分≥7） | OPEN-1 获独立冷读佐证 | §1.6 OPEN-1 严重性保持"中"，标注"获独立冷读佐证"；继续推进 §3.7 试跑 |
| 命中 OPEN-1（评分<7） | OPEN-1 是真问题但严重性可能被高估 | §1.6 OPEN-1 严重性降级为"低-中"；§3.7 试跑时降优先级 |
| 报告了缺陷但未包含 OPEN-1 同构问题 | OPEN-1 不是审查者自然发现的顶级结构问题 | §1.6 OPEN-1 严重性降级为"低"；OPEN-1 可能是作者自识偏置（与 §10.8 同源） |
| 报告不足 3 个（非强制凑数） | **阳性**——说明审查者没有因格式压力制造假缺陷，阴性结果可信度更高 | 上述判读的可信度加分 |

### 输出写回
- 审查报告存为 `独立审查_Qwen_OPEN1降级冷读_{date}.md` + `.json`
- 判读结果写回 §1.6 OPEN-1 行 + §14 治理声明（补充"降级替代执行结果"）
- 写回 `../框架级成熟度评估表.md` §3 漂移检测层各子行（若 OPEN-1 降级）

## 协议 3：首次真实试跑（执行优先级：最高收益，最高成本）

**原始框架**：§1.4（最低强制版）+ §8（Closure）+ 本框架全流程
**执行指南**：见对话中"P0-1 试跑具体聊聊"的四步方案（选项目→预登记→试跑→回写）

### 为什么是最高收益
- 一次性推进 OPEN-1/OPEN-2/OPEN-4/§1.5 死亡判据基线
- 试跑数据是解冻条件——不试跑，框架永远停在"草案冻结"
- 比任何一轮新审查都便宜

### 执行步骤（最小版）
1. **选项目**（三硬标准缺一不可）：
   - 周期 ≤ 2 周
   - 真实需求（非为试跑造的）
   - 有明确完成边界
   - **首次只跑最低强制版（§1.4）**，不跑完整版
   - 优先 A 类（半天探索型），跑通再考虑 B 类（一周项目）
   - **不推荐**用"正在做、未闭合的项目"（框架复杂度和项目复杂度混在一起）

2. **预登记成功判据**（试跑前冻结，不可后改）：
   | 指标 | 目标 | 触发即"框架有问题" |
   |------|------|------------------|
   | L1 Prompt 写作耗时 | ≤15min | >30min |
   | L2 实际迭代轮数 | ≤8 轮（探索型上限） | >8 轮未收敛 |
   | Dev Log 漏记率（B 类） | <30% | ≥30% |
   | HG-1 是否真人看 | 是 | AI 自行代审 |
   | **框架维护耗时占比** | ≤15%（§1.5 死亡判据） | >15% |
   | Retrospect 产出 | ≥1 失效 + ≥1 意外 | 走过场 |

3. **试跑 + 数据收集**：只收结构化可计数数据（时间戳/轮次/漏记/决策/意外）。**失败数据优先于成功数据**（§7.5）。

4. **Retrospect 回写**：用 §7.4 深度版，逐行更新 `../框架级成熟度评估表.md`。**v1.5.2 标题 = "试跑 1 回写"**。
