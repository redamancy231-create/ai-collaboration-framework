## 1. 何时触发

### 1.1 强制触发（必须执行独立审查）

以下任一条件满足时，**必须**执行完整SOP：

| # | 触发条件 | 典型场景 |
|---|---------|---------|
| T1 | **创建或重大修改方法论资产** | 新建/改skill、playbook、复用包、预登记协议、框架文档 |
| T2 | **闭合项目或里程碑** | 项目终期总结、阶段报告定稿、正式交付物发布 |
| T3 | **关键决策点（HG闸门）** | 框架L-H定义的HG-0（启动）/HG-1（里程碑）/HG-2（Spec变更）/HG-3（闭合） |
| T4 | **作者＝审查者的实际或潜在情形** | 同一后端模型同时参与了设计/执行/初步审查，若不加独立审查则全链路无独立声音 |

### 1.2 建议触发（推荐但不强制）

| # | 触发条件 | 说明 |
|---|---------|------|
| S1 | 产出将对外发布或作为学术依据 | 论文、报告、公开发表的分析结论 |
| S2 | 结论依赖于AI生成的关键数据或推理 | 尤其是未经人工独立核验的中间产物 |
| S3 | 上次独立审查距今超过3个里程碑 | 持续项目中的"定期体检" |

### 1.3 触发判定的辅助问句

> 这个产出或决策，如果事后被发现有问题，最贵的修正成本是什么？  
> 如果成本是"重做项目"或"声誉损害"，就走独立审查。

### 1.4 不需要独立审查的情形（负例）

- 修bug、复现实验、补测试
- 已冻结协议内的常规对照臂
- 纯格式/排版调整（不涉及实质内容变更）
- 用户本人已作为独立审查者亲自审过（记入审计记录即可）

## 2. 独立性定义与判定

### 2.1 两条硬轴（缺一不可，见框架§9.2）

**独立审查 = (后端模型不同) AND (上下文隔离)**

| 轴 | 检查项 | 通过条件 | 失败典型 |
|----|-------|---------|---------|
| **轴1: 后端模型不同** | 审查者的后端模型与被审对象作者的后端模型不是同一个 | 已知作者用的模型名，审查者用的模型名不同；两模型是不同供应商或不同模型系列 | 同一后端（如都是DeepSeek-V4-Pro）换CLI壳（一个Claude Code，一个Cherry Studio）再审一遍；同一后端换标签再审（"这次我当审查者"） |
| **轴2: 上下文隔离** | 审查会话不共享作者对话历史、memory、项目Spec | 全新会话、关闭所有项目memory/知识库/技能、审查者看不到作者的设计过程与内部讨论 | 审查者读了作者的MEMORY.md、CLAUDE.md、框架文档中的结论章节后被预锚定 |

### 2.2 独立性层级（三级）

| 层级 | 定义 | 效力 | 标注 |
|------|------|------|------|
| **独立** | 轴1 AND 轴2 均通过 | 可作为独立证据引用，计入"N方独立共识" | 标注"独立审查" |
| **半独立** | 轴1通过但轴2未完全隔离（如审查者知晓项目背景但不知具体设计过程）；或轴2通过但作者端模型不可考 | 可用于内部校验，但不得声称"独立"，引用时须注明局限 | 标注"半独立审查，局限：[具体局限]" |
| **非独立** | 任一轴明确不通过 | 仅可标注为"内部反方检查"或"自审"，不具备独立审查效力 | 标注"非独立/内部检查" |

### 2.3 独立性判定流程

```
作者的后端模型已知？
  ├─ 是 → 审查者后端模型≠作者？
  │       ├─ 是 → 审查上下文隔离于作者？
  │       │       ├─ 是 → 「独立」
  │       │       └─ 否 → 「半独立」(局限：上下文部分共享)
  │       └─ 否 → 「非独立」
  └─ 否（作者后端不可考）
        └─ 审查上下文隔离？
              ├─ 是 → 「半独立」(局限：不能排除作者-审查者同后端)
              └─ 否 → 「非独立」
```

## 3. 角色分工与人员配置

### 3.1 三个角色（必须由不同实体担任）

| 角色 | 职责 | 谁来干 | 不能是谁 |
|------|------|--------|---------|
| **作者 (Author)** | 产出被审对象 | 执行任务的AI或人 | — |
| **操作者 (Operator)** | 搭建裸环境、盲化材料、传递输入输出 | 用户本人或其指定的代理人 | **不能是作者**（作者知道自己写了什么，盲化无效） |
| **审查者 (Reviewer)** | 独立阅读被审对象、评分、挑战、给终判 | 与作者不同的后端模型（裸会话），或独立人类专家 | **不能是作者**（同一后端模型），**不能与作者共享上下文** |

### 3.2 操作者的独立性

操作者是**信息防火墙**——它从作者拿到原始产出，盲化后交给审查者，再把审查结果传回。操作者本人无需满足"与作者不同后端"（它不评分），但必须：

- 不在传递过程中添油加醋或选择性过滤
- 盲化时剥除所有身份标识（模型名、臂别、版本号等可能暗示作者身份的信息）
- 保留还原key（自己留存，不给审查者）

### 3.3 最低配置

- **最低可接受**: 作者（AI-A） + 操作者（用户） + 审查者（AI-B，裸会话）
- **增强配置**: 作者 + 操作者 + 审查者1 + 审查者2（两独立后端，分别裸会话）
- **黄金标准**: 作者 + 操作者 + AI审查者 + 独立人类专家审查（并行，后对照）

```
                    ┌─────────────────┐
                    │  前置准备：裸环境  │
                    │  (见§5)          │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
    步骤一：事实提取    步骤二：独立评分    步骤三：对抗式挑战
    与溯源验证          (见§7)           (见§8)
    (见§6)                                 │
              │              │              │
              └──────────────┼──────────────┘
                             │
                    ┌────────┴────────┐
                    │  步骤四：终判与记录│
                    │  (见§9)          │
                    └────────┬────────┘
                             │
                    ┌────────┴────────┐
                    │  后置核验：       │
                    │  独立性审计(§10)  │
                    └─────────────────┘
```

**关键约束**:
- 步骤一和三**可由同一审查者执行**（都是审查者的工作），但步骤一的"事实提取"必须先于步骤三的"对抗式挑战"——不能让挑战驱动事实提取的方向。
- 步骤二的评分 **必须在步骤三的挑战之前给出**——否则挑战会影响评分，评分变成"先射箭后画靶"。
- 若使用两个不同审查者分别做步骤二和步骤三，两个审查者需各自满足独立于作者的条件。

## 5. 前置准备：裸环境搭建

### 5.1 核心原则

审查者的运行环境必须满足：**无记忆、无工具（除必要读取外）、无项目上下文、无作者方法论框架**。

### 5.2 搭建清单

> 以下checklist提炼自"多模型对照实验"项目的三套裸环境实践。若该项目的裸环境checklist有更新，以项目文件 `裁判运行_CherryStudio_clean_checklist.md` 等为准。

- [ ] **C1. 独立APP/会话壳**: 若审查者通过API运行，使用Cherry Studio或其他独立客户端新建助手（不要复用已有助手）；若通过CLI运行（如Codex CLI），确认该CLI的memory/项目文件路径不与被审项目的MEMORY.md/CLAUDE.md重叠
- [ ] **C2. 模型确认**: 选定审查者的后端模型，记录完整模型ID（非"ChatGPT"这种模糊标签，而是"GPT-5.5 via Codex CLI"或"GLM-5.1 via 智谱BigModel API"）
- [ ] **C3. 系统提示中性**: 不放persona、方法论立场、项目背景。只交代审查任务。若是首次使用，可放一句中性概括（如"你是一个方法论审查者，任务是对一份文档进行独立评估"），但不涉及被审项目的具体内容
- [ ] **C4. 记忆/知识库全关**: 关闭所有项目memory、CLAUDE.md加载、RAG知识库
- [ ] **C5. 工具全关**: 关闭代码解释器/Python/MCP/联网搜索（除非被审对象本身需要联网验证——见步骤一）
- [ ] **C6. 跨会话记忆关**: 若APP支持长期/跨会话记忆，关闭或确认不加载
- [ ] **C7. 采样参数固定**: temperature等参数设固定值并记录，全程不变（若需要重评以评估噪声底，参数一致性是前提）
- [ ] **C8. 全新会话**: 每次审查开全新话题/会话，不在旧会话中接着评
- [ ] **C9. 自检**: 问审查者一个探测问题，如"你了解[被审项目名]吗？"或"你知道[被审对象的作者/设计者]是谁吗？"——预期回答是不了解/不知道。若它"知道"，说明记忆在串，停下排查
- [ ] **C10. Test Distribution Audit（测试分配审计）**: 审查开始前，操作者统计本次被审对象中"LLM推理核心组件"（prompt设计、策略逻辑、模型选择推理）vs"确定性组件"（工具调用、API返回校验、格式检查）的验证投入比例。若 LLM 推理核心占比 < 20%，标注为"测试分配失衡风险"——审查结论须额外提醒"本审查可能系统性欠测了被审对象的推理质量"。**依据**：Agent Harness Engineering 综述实证发现——行业 70%+ 测试投向确定性组件，<5% 投向 LLM 推理核心（Li et al., 2026）。

### 5.3 裸环境的退化检测

每次独立审查开始前，操作者应执行一次自检：

```
向审查者提问：「在开始审查之前，请回答：你是否已经知道以下信息？
1. [被审项目名] 的设计者是谁？
2. [被审对象] 的核心结论是什么？
3. 你之前是否参与过任何与[被审项目]相关的讨论或审查？」

若审查者对以上任一问题回答"知道"或"是"——该环境不裸，更换或清洗后重试。
```

## 6. 步骤一：事实提取与溯源验证

### 6.1 目标

从被审对象中提取**所有可验证的事实声明**，逐个标注置信层级，区分"可独立核实"与"只能来自作者"的断言。

### 6.2 执行

审查者阅读被审对象全文（操作者已盲化），对其中每一项关键事实声明标注置信层级：

| 层级 | 定义 | 典型来源 | 示例 |
|------|------|---------|------|
| **T1 可独立核实** | 有公开可查的原始来源，审查者可自行验证 | 官方文档、公开数据集、DOI/arXiv论文、API文档、开源仓库release | "Python 3.12.7 于2024-10-01发布" |
| **T2 可交叉验证** | 有多个独立来源可佐证，但没有单一官方源 | 多个社区确认的行为、多个平台的一致记录 | "某模型的API定价为X" |
| **T3 作者自报** | 只有作者自己提供的，外部无法独立核实 | 内部实验数据、私有项目记录、作者声称的"我们发现" | "我们在项目中发现X效果优于Y" |
| **T4 不可验证** | 缺乏任何可核实的路径 | 未记录来源的断言、无数据的声称 | "普遍认为X是最好的方法" |

### 6.3 关键动作

1. **逐条列出被审对象的核心事实声明**（至少覆盖：核心结论、关键数字、外部对标声明、方法论的独创性声称）
2. **每条标注T1-T4**，T1/T2标注源文件路径或可验证链接
3. **对T3/T4声明**，审查者应标注"该声明只能来自作者，无法独立核实——审查者对此声明的任何评价均不能视为独立确认"
4. **若审查者具备联网/搜索能力**，对T1/T2声明进行抽查核实（至少抽3条）。若审查者不具备此能力（裸环境无工具），标注"工具受限，未做独立溯源核实——本步骤1仅为声明分类"

### 6.4 输出

一份"事实提取表"（`facts_extracted`），每条包含：
- `claim_id`（序号）
- `claim_text`（原文或精确概括）
- `tier`（T1-T4）
- `verification_source`（可验证源，或"作者自报"）
- `confidence`（审查者对此声明准确性的主观信心，1-5）

## 7. 步骤二：独立评分

### 7.1 目标

在**不受挑战立场干扰**的情况下，给予被审对象一个独立的、基于证据的评分。

### 7.2 评分维度（通用版）

审查者依据被审对象的性质和审查目的，从以下维度中选择适用项。每个维度评1-5分。

| 维度 | 评估重点 | 1分（最差） | 5分（最优） |
|------|---------|------------|------------|
| **D1 事实准确性** | 声明有据，数字正确，无编造 | 多处事实错误或关键数字无来源 | 所有关键声明可溯源，核实无错 |
| **D2 逻辑一致性** | 内部无矛盾，推论链完整 | 前提与结论矛盾，或逻辑跳跃 | 每步推论可追踪，矛盾明确处理 |
| **D3 边界诚实** | 坦承局限、不确定、失败模式 | 过度声称，无局限声明 | 明确标注不确定域、已知局限、N/A情形 |
| **D4 方法论恰当性** | 方法匹配问题，工具选择合理 | 方法不匹配问题，或使用不当 | 方法选择有理由，替代方案有考虑 |
| **D5 可复现性** | 他人能否独立复现 | 关键参数/步骤缺失，无法复现 | 完整可复现，含数据源/参数/代码 |
| **D6 增量价值** | 相比已有方法/基线，新贡献是什么 | 重复已有工作，无增量 | 明确的差异化贡献，有对标证明 |

### 7.3 评分约束

- **评推理质量，不评结论正误或偏好**。若被审对象选边（如选策略A非B），审查者不应因自己偏好B而扣分——只评它选择A的论证是否充分
- **评分必须基于被审对象自身的内容**，不是基于审查者对该领域的先验知识（除非先验知识用于核实事实——见步骤一）
- **评分中若某维无法判断**（如审查者缺乏领域知识），标"N/A"并说明理由，不强行评分

### 7.4 输出

- 每维度的分数 + 逐维评分理由（每条至少一句话）
- 总体加权分（维度权重由操作者根据审查目的预设；若未预设，默认等权）
- 评分者标注自己在该领域的置信度（1-5，5=领域专家）

## 8. 步骤三：对抗式挑战

### 8.1 目标

审查者切换到**魔鬼代言人立场**，对步骤一提取的事实和步骤二给出的评分进行系统性质疑。

### 8.2 挑战框架

审查者对以下每个方面，**必须**给出至少一条具体的挑战（不可空白）：

**(A) 对该评分有利的事实是否存在相反证据？**
- 若步骤一标注为T3/T4的声明在步骤二中支撑了高分，审查者应指出"该高分依赖未经独立核实的声明"

**(B) 评分标准本身是否隐含偏向？**
- 若评分维度对被审对象类型天然有利（如"D6增量价值"对声称"原创"的项目自动高分，而不查是否真有增量），审查者应指出

**(C) 是否存在更简单的解释？**
- 被审对象声称的"发现"或"贡献"，能否用更平淡的因素解释（如基线选择、数据特性、随机波动）

**(D) 如果我是作者，我会希望审查者漏掉什么？**
- 从作者利益出发，列出最想被忽略的弱点、矛盾或边界条件

**(E) 该方法论或决策，若完全失败，最可能的死因是什么？**
- 不是"可能有问题"，而是"如果它错了，错在哪个假设上"

### 8.3 对抗式挑战的输出

- 每个方面(A)-(E)至少一段实质性挑战
- 标注哪些挑战"可独立验证"（需要进一步查证），哪些"属方法论质疑"（不需要外部数据）
- 挑战不得修饰、软化，不得写成"虽然有X，但总体Y"——保持魔鬼代言人的锋利

### 8.4 挑战后评分调整

- 审查者在完成步骤三后，**可以**调整步骤二的评分
- 调整的每一项必须记录：`原分→新分，调整原因（引用步骤三的具体挑战编号）`
- **不可调整的分数也必须注明**"经挑战后维持原分，原因为____"

---

## 9. 步骤四：终判与记录

### 9.1 终判格式

审查者给出一个完整的结构化终判：

| 字段 | 内容 |
|------|------|
| `verdict` | **保留** / **小改后保留** / **重大修改** / **弃用** |
| `verdict_reason` | 一句话核心理由 |
| `score_original` | 步骤二原评分（逐维+总分） |
| `score_adjusted` | 步骤三挑战后调整评分（如有调整，标注改动） |
| `key_strengths` | 被审对象最站得住的部分（≤3条） |
| `key_weaknesses` | 被审对象最该修正/放弃的部分（≤3条，按严重度排序） |
| `critical_risks` | 若终判为"保留"，仍存在的不确定性或风险（如有） |
| `reviewer_confidence` | 审查者对该终判的信心（1-5）及不确定性来源 |
| `independence_declaration` | 见§9.2 |

### 9.2 独立性声明（必填，直接嵌入终判）

```markdown
## 独立性声明

| 项目 | 值 |
|------|-----|
| 作者后端模型 | [模型名，若不可考写"不可考"] |
| 审查者后端模型 | [完整模型ID] |
| 审查者CLI/壳 | [如 Cherry Studio / Codex CLI / 裸API] |
| 轴1: 后端不同？ | 是/否 [若否，自动降为"非独立"] |
| 轴2: 上下文隔离？ | 是/否 [说明隔离措施：全新会话/无memory/无知识库] |
| 操作者与作者不同？ | 是/否 |
| 独立性层级 | 独立 / 半独立 / 非独立 |

若为"半独立"或"非独立"，必须注明：⚠ 本次审查为[半独立/非独立]，局限为[具体原因]。建议补充真正独立的第二次审查。
```

### 9.3 审计痕迹要求

- 步骤一至四的全部原始输出必须保留，不得事后编辑（修正拼写/格式除外）
- 评分调整记录必须保留原分与新分之间的完整轨迹
- 若操作者对审查者的输出做了任何裁切/重排/解释，必须单独标注"操作者中间处理：[具体操作]"

## 10. 后置核验：独立性审计

### 10.1 核验时机

在审查者完成步骤四、全部输出已拿到之后，由操作者或第三方执行。

### 10.2 核验清单

- [ ] **A1. 后端provenance已记录?** 审查者的后端模型ID、CLI壳名、运行环境已完整记录在独立性声明中？
- [ ] **A2. 自检已通过?** 裸环境搭建时的自检问句有记录，且审查者确认"不了解项目"？
- [ ] **A3. 操作者输入检查?** 操作者喂给审查者的材料中，是否包含任何可能泄露作者身份、项目背景、预期结论的信息？（抽查操作者的输入记录）
- [ ] **A4. 审查者输出中有无"知道得太多"的迹象?** 审查者在未被告知的情况下，提到了作者名、项目历史、特定版本号、或只有参与者才知道的内部讨论？若有——标记为红旗，可能上下文未真正隔离
- [ ] **A5. 审查提示词是谁写的?** 若提示词由作者本人写，审查已被潜在引导（框架审查链的教训：提示词作者=框架作者=DeepSeek，审查①ChatGPT-5.5被引导性偏差影响）。标记此偏倚，在终判报告中注明
- [ ] **A6. 评分在挑战前给出?** 步骤二评分的时间戳（或逻辑顺序）早于步骤三挑战？
- [ ] **A7. 调整有记录?** 步骤三后的评分调整，每项都有原分→新分+原因？
- [ ] **A8. 独立于"自我安慰"?** 若审查者是操作者用同一CLI换了个会话标签叫的（如Claude Code里的"现在换个立场审一下"），这不算独立——标记为"内部反方检查"

### 10.3 核验结论

- [ ] 通过：所有检查项通过，本次审查可标记为"独立"，审查结论可作为独立证据
- [ ] 条件通过（标注局限）：部分检查项未完全通过，但仍可用——标记半独立及具体局限
- [ ] 不通过：关键检查项（A1/A2/A3/A4/A8中任一项）失败——本次审查降级为"非独立"，不得声称独立证据

## 12. 红旗与作废条款

以下任一情况出现，**该次审查结果作废**，须重新执行：

| 编号 | 红旗 | 严重度 |
|------|------|--------|
| R1 | 审查者与作者被确认为**同一后端模型** | 致命 |
| R2 | 审查会话中发现了作者的项目memory/Spec/设计文档 | 致命 |
| R3 | 审查者在自检问句中"认得"被审项目或作者 | 致命 |
| R4 | 操作者向审查者传递了含作者身份/项目背景/预期结论的信息 | 高 |
| R5 | 步骤二评分在步骤三挑战**之后**才给出 | 高 |
| R6 | 审查提示词由作者撰写，且审查结果明显沿提示词方向倾斜 | 高 |
| R7 | 审查者的采样参数在评分过程中被改过 | 中 |
| R8 | 用旧会话接着评（非全新会话） | 中 |
| R9 | 独立审查的.md/.json产出被事后修改实质内容而未记amendment | 中 |

## 13. 常见失败模式

### 13.1 伪独立

| 失败模式 | 表面 | 实质 | 预防 |
|---------|------|------|------|
| **换壳不换模** | "这是GPT-5.5审的，不是我" | 两个CLI接的是同一个后端 | 当场记录完整模型ID（非CLI标签） |
| **换会话不洗脑** | "新会话审的，独立了" | Claude Code/Codex CLI的memory全局加载，新会话照样读 | 审计memory加载状态，自检问句 |
| **操作者走后门** | "我就给了它要审的文档" | 操作者还附带了解释性说明或预期 | 抽查操作者输入，记录喂给审查者的完整材料 |
| **评论式独立** | "我也觉得这有问题" | 审查者附和操作者/作者已经提出的批评，而非独立发现 | 对比审查者发现与操作者已知问题列表，查重叠率 |

### 13.2 审查衰减

| 失败模式 | 症状 | 预防 |
|---------|------|------|
| **漂移式放水** | 随着审查次数增多，评分越来越温和，挑战越来越软弱 | 建立审查者轮替规则（同一审查者连续审查同一项目≤3次） |
| **疲劳式评分** | 步骤二分数各维拉不开、全挤在3-4之间 | 设评分锚点对照（"这批有高分有低分才是健康的"） |
| **和睦偏差** | 审查者为显示"公正"而避免给极端分（1或5） | 在审查提示词中明确要求"如果你的判断真的是1或5，就写1或5，不要缩成2或4" |

### 13.3 工具受限

| 失败模式 | 症状 | 预防 |
|---------|------|------|
| **裸环境阻断核实** | 步骤一的事实核实因无工具/联网而只能做声明分类 | 在步骤一输出中诚实地标注"工具受限，仅做分类未做核实"，不把T3/T4声明当事实用 |
| **数据不可用→替代** | 同ETF R5消融事故——关键数据拿不到，捏造或使用替代数据 | 框架§4.3: 工具/数据不可用→**停止并报告**，绝不用替代数据继续 |

### 14.3 与kill-test-first的互补关系

- **kill-test-first**: 在项目"该不该做"的决策点前置否决（门1）+ 预登记冻结（门2）。它管的是**事前**——建设开始之前。
- **本SOP**: 在项目"做得怎么样"的检查点引入独立审查。它管的是**事中和事后**——产出之后的独立评估。
- 两者不重合、不互相替代。一个项目可以同时经历kill-test-first（事前）和独立审查（事中/事后）。

### 14.4 与多模型对照实验的关系

- 本SOP的裸环境要求和独立性两轴定义，直接服务于多模型对照实验的选手/裁判环境要求
- 多模型对照实验的amendment v0.5 C-5（裁判盲化）和C-6（选手裸API统一化）的隔离标准，与本SOP的轴2一致
- 若该实验恢复运行，本SOP可作为操作者（Schwarzes）执行独立审查环节的具体操作手册

## 附录A：审查提示词模板（给操作者用）

> 以下是一个最小可用的审查提示词模板。操作者将其填入审查者裸助手，不附加任何其他上下文。

```markdown
你是一位独立方法论审查官。你的任务是审查一份文档。

**你的立场**：默认怀疑。找盲区、漏洞、过度声称、逻辑不一致。不要客气，不要默认作者是对的。评推理质量，不评立场。

**你的工作流程（严格按顺序）**：

步骤一——事实提取：逐条列出文档中的所有关键事实声明，标注可核实层级（T1=可独立核实/T2=可交叉验证/T3=作者自报/T4=不可验证）。

步骤二——独立评分：对以下维度逐维评分1-5：
- 事实准确性、逻辑一致性、边界诚实、方法论恰当性、可复现性、增量价值

步骤三——对抗式挑战：对以下五个方面，每个至少给出一条实质性质疑：
(A) 支撑高分的T3/T4声明是否可靠？
(B) 评分标准是否有隐含偏向？
(C) 被审对象的"发现"是否有更简单的解释？
(D) 从作者利益出发，最希望审查者漏掉的弱点是什么？
(E) 若该方法论/决策完全失败，最可能的死因是什么？

步骤四——终判：保留/小改/重大修改/弃用，附理由。

输出前，填写独立性声明：你用什么模型？你是否在审查前就了解此项目或作者？
```
