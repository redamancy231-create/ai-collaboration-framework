## 概览

本清单用于对 AI协作项目全生命周期框架 项目进行元审查（meta-audit），即审查框架自身是否遵守其规定的规范和最佳实践。清单分为四个维度，共 75 项检查（v1.4：64 项→v1.5：67 项→v1.6.3：75 项）。

### 评分标准

| 评分 | 通过率 | 含义 |
|------|--------|------|
| **健康** | 90–100% | 框架自身合规性良好 |
| **值得关注** | 75–89% | 存在需改进的薄弱环节 |
| **受损** | 60–74% | 多项核心要求未满足，需系统性修复 |
| **不安全** | <60% | 框架自身规范大面积失效，须暂停新发布并整改 |

### 检查项统计

| 维度 | 检查项数 | 核心关注点 |
|------|---------|-----------|
| 一：Provenance完整性 | 20项 | 模型来源声明的完整性、一致性和可追溯性 |
| 二：审查独立性 | 12项 | 审查链各环节的独立性和无污染性（v1.4 新增 I-012 版本绑定） |
| 三：记忆系统健康 | 15项 | 项目记忆文件的时效性、一致性和完整性 |
| 四：最佳实践实际使用 | 28项 | 框架自身是否按 §2–§7 要求使用工具和实践（v1.4 新增 B-019/B-020；v1.6.3 新增 B-021~B-028） |

## 使用说明

1. **通用项 vs [个人项目] 项**：清单中标有 `[个人项目]` 的检查项仅适用于作者（Acerolaorion）的个人工作环境——它们引用了个人 memory 文件、特定项目状态、硬件记录、skill 清单等。外部独立审查者应**跳过 [个人项目] 项**（标记为 N/A-个人项）。
2. **衍生项目**：可 fork 本清单并替换 [个人项目] 项为被审查项目的对应路径。
3. **预判记录**：本清单的所有"预判"已移至附录：作者预判记录。审查者评分**仅依据实际检查结果**，预判仅供参考，不具约束力。正文各检查项标注 `[待独立审查]`。

- **修复**：逐条审计实验证据的二维标注；M 等级与实际复现状态对齐
- **预判**：[待独立审查] — A2 已从 M0 更新至 M2（Qwen 复现），A3 仍为 M0（未复现）

### B-022：MF 三层模板使用（v1.6.3 新增）

- **条件**：框架中任何方法论片段（MF）的表示
- **通过标准**：MF 使用 §9.10 三层模型表示（第一层：问题识别条件 + 第二层：解决方案适用条件 + 第三层：已知反例/失效模式）；第三层包含静默失效信号字段（如 PT-M1 的"缺少约束→失效不可检测"）
- **失败标准**：MF 仅有一层表示（问题识别）而缺失适用条件和反例；第三层缺少静默失效信号；MF 声称"可迁移"但未经三层模型的适用条件检查
- **修复**：为每个 MF 补全三层表示；静默失效信号强制填写
- **预判**：[待独立审查] — PT-M1 已完成三层示例，其他 MF 待映射

### B-023：对照实验检查清单合规（v1.6.3 新增）

- **条件**：任何 Tier 1 对照实验的设计和执行
- **通过标准**：CK1-CK3（效能分析/GT 完整性/DV×天花板矩阵）作为硬门全部 PASS；CK4-CK6（捷径审计+反编造/INVENTORY 异后端验证/修复-回归审查门）条件触发时执行；检查清单在实验启动前冻结（预登记精神）
- **失败标准**：CK1-CK3 任一项在实验执行前未完成；实验声称"对照实验"但跳过了硬门检查项；检查清单在实验后才填写（事后合理化）
- **修复**：下一实验启动前逐项执行 CK1-CK6；预冻结检查清单
- **预判**：[待独立审查] — A2 和 A3 实验在 CK1-CK6 成文前完成，已做等效检查但非逐项对照。此条目适用于未来实验

### B-024：维护流程合规（v1.6.3 新增）

- **条件**：框架版本升级（Major/Minor/Patch）
- **通过标准**：升级满足 §2.6 规定的审查门（Major≥3 后端+试跑 / Minor≥2 后端 / Patch≥1 后端）；Changelog 记录触发事件+来源+证据等级；三件套（.md/.json/.docx）同步更新；VERSION 文件更新
- **失败标准**：Minor 升级仅单后端审查；三件套版本不一致；VERSION 文件滞后；版本时间线遗漏
- **修复**：补执行审查门；同步三件套和 VERSION
- **预判**：[待独立审查] — v1.6.2 首次满足 Minor 门（Codex+Qwen），v1.6.3 超额满足 Patch 门。v1.6 自身在审查门生效前完成（过渡期）。DOCX 在 v1.6.3 待同步

### B-025：诚实声明更新（v1.6.3 新增）

- **通过标准**：§1.8 已知局限清单在新版本发布前审核——新增局限是否被识别并记录；已有局限的状态是否更新（如"单模型证据"从 1 模型更新为 2 模型）；局限清单编号连续无遗漏
- **失败标准**：新版本引入新局限但 §1.8 未更新；已有局限的状态描述过时（如仍称"仅 GPT-5.5 单模型"而 Qwen 复现已完成）
- **修复**：逐条审核 §1.8 与框架当前状态的一致性

### B-026：外部依赖登记（v1.6.3 新增）

- **条件**：框架新增外部依赖（工具链/模型/平台）或 Minor 升级前

- **失败标准**：新增依赖未登记；登记表长期未检查（>2 个版本）；依赖变化导致框架章节失效但登记表未反映
- **修复**：补登记缺失依赖；执行检查节奏
- **预判**：[待独立审查] — 登记表 v1.0 初次创建（2026-06-21），尚未经过完整检查周期

### B-027：规则退役判定（v1.6.3 新增）

- **条件**：框架中任何规则/检查/反模式条目
- **通过标准**：存在退役判定机制（§2.6 规则退役：T1 工具链覆盖/T2 持续无触发/T3 规则疲劳）；定期（每 3 个项目或 Minor 升级前）扫描是否有满足退役条件的规则；退役过程记录在 DEV_LOG 并经 HG-2 确认
- **失败标准**：框架只有"加入"机制无"退场"机制（v1.6.3 前状态）；已有规则明显失效但未标记退役候选；工具链已原生覆盖手写规则但未触发 T1
- **修复**：执行退役扫描；标记候选规则；HG-2 确认退役
- **预判**：[待独立审查] — 退役判定机制刚写入（v1.6.3），尚未有任何规则实际退役。此条目待首次退役触发后验证

### B-028：核心 vs 外挂边界合规（v1.6.3 新增）

- **条件**：框架新增任何内容（节/反模式/协议/工具）
- **通过标准**：新增内容经 §1.7 三条判定规则（普遍性/强制性/稳定性）评估，决定进核心（主文档）或外挂（配套目录）；边界案例默认外挂（"从外挂升级到核心只需一次编辑+闸门，从核心移除需证明之前的强制要求是错的"）；新增内容不触发反模式 A4（外挂膨胀为核心）
- **失败标准**：新增内容未经核心/外挂判定直接写入主文档；配套目录中的参考实现被后续版本升级为核心但未经过 HG-2；核心内容量增长主要由"外挂膨胀"驱动而非真正的新强制规则
- **修复**：逐条审计核心文档中的内容是否符合核心准入标准；将误归入核心的外挂内容降级至配套目录
- **预判**：[待独立审查] — §1.7 自 v1.5.3 写入，后续新增内容（§7.7/§9.11/§2.6 规则退役）均经核心/外挂判定

## 触发机制

### 触发条件与执行范围

| 触发条件 | 执行范围 | 优先级 | 备注 |
|---------|---------|--------|------|
| **版本发布**（如 Minor 升级） | 完整四维度 75 项 | P0（阻断） | 任何 FAIL 项必须在发布前修复或标注豁免 |
| **月检**（每月1日） | 维度三（记忆系统健康）全部 + 维度四关键项（B-001/B-003/B-004） | P1 | 快速扫描，产出月检报告 |
| **死亡判据警告**（框架 §1.5） | 完整四维度 + 根因分析 | P0（阻断） | 触发条件见框架 §1.5 |
| **审查链新增/变更** | 维度二（审查独立性）全部 | P1 | 每次新增审查者时触发 |
| **新项目启动** | 维度一全部 + 维度四全部 | P2 | 建议在新项目初期执行，预防性合规 |

### 执行规范

1. **执行者资格**：按框架 §9.2，执行者须满足（后端不同 × 上下文隔离）——不可由框架作者（DS-v4-pro）自审；不可与框架项目共享 memory
2. **记录格式**：使用下方审计日志模板，逐项记录结果
3. **豁免标注**：任何 N/A 项须标注原因；任何标记项须说明标记理由
4. **总分计算**：通过率 = PASS项数 / (总项数 − N/A项数 − 个人项目N/A项数)。[个人项目] 项若被审查者跳过（标记为 N/A-个人项），不纳入分母。标记项不计入 PASS 分子——标记=非独立=不计分，仅在报告中单独列出
