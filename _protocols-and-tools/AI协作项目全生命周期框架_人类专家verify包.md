## 最小背景（30 秒就够）

框架把"用 AI 跑项目"分层：**Spec**（不变的项目宪法）→ **Prompt**（单次任务规格）→ **Loop**（AI 在约束内自主反复试错，每轮极快）。在关键节点插 **Human Gate（人类闸门）**：HG-0 启动 / HG-1 里程碑 / HG-2 改 Spec / HG-3 闭合。

**矛盾的根**：人的审查是**离散**的（按里程碑/事件触发），AI 的 Loop 执行是**连续**的（每轮都在动）。

## 要你裁的那一个问题

当前 Human Gate 设计的真问题，更接近哪一个？**两个判断方向相反**：

| | **A** | **B** |
|---|---|---|
| 谁提的 | 独立 AI——GPT-5.5（via Codex CLI） | 框架作者（DeepSeek-V4-Pro）+ 同栈 AI（Opus 4.8） |
| 诊断 | 闸门**太多**、摩擦过重；"任何 Spec 改动都要人确认"会拖垮效率 | 闸门有**覆盖盲区**：AI 在 Loop 内连续小调整（放宽筛选→改权重→调阈值）每次都不触发闸门（因为没改 Spec），却累积成方向偏转，到下个里程碑才被发现 |
| 方向 | **减少 / 分级**闸门 | **新增**一个 Loop 内"方向一致性自检" |
| 最强理由 | 个人项目里闸门越多越没人真用，最后全是橡皮图章；过度工程是个人方法论的头号死法 | 失败常是缓慢漂移而非大爆炸；框架自己也承认"人抽查频率 < AI 犯错频率"；过去真出过"机制存在但没被及时触发"的事故 |
