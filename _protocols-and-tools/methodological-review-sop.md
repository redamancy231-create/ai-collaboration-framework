## §1 何时触发

### 1.1 强制触发条件 (T1-T4)

以下任一条件满足时，必须启动独立审查流程：

| 编号 | 触发条件 | 说明 |
|------|---------|------|
| **T1** | 方法论资产产出 | 框架、SOP、分类体系、评估rubric、模板——任何会被后续项目复用的结构性产出 |
| **T2** | 结论将进入用户论文/正式交付物 | 包括毕业论文、学术报告、对外发布的分析结论 |
| **T3** | 决策涉及资源分配或不可逆操作 | 数据删除、架构推倒重来、公开发布、提交竞赛结果 |
| **T4** | 两个以上模型/agent给出冲突结论 | 需要独立第三方裁决时 |

### 1.2 建议触发条件 (S1-S3)

| 编号 | 触发条件 | 说明 |
|------|---------|------|
| **S1** | 单模型产出但复杂度高 | 超过3个推理步骤的链式结论 |
| **S2** | 使用了新颖/未经验证的方法 | 首次在项目中使用的方法或工具链 |
| **S3** | 作者模型已知有系统性盲点 | 基于历史记录，该模型在某领域反复出错 |

### 1.3 不触发的情况 (负例)

- 纯格式转换（无内容判断）
- 已有独立审查结论且输入未变
- 用户明确声明「不需要审查」（需记录声明时间和上下文）。**限制**：对 **T2（正式交付物）和 T3（不可逆操作）**，不得静默跳过。须记录：风险接受人、接受原因、时间戳，写入项目 Risk Register。审计时可追溯。对 T1/S1-S3，声明即豁免。

## §2 独立性定义与判定

### 2.1 两轴判定模型

独立性沿两个轴判定，缺一不可：

**轴1: 后端模型独立性**
- 审查者使用的后端模型必须与作者使用的后端模型**不同**
- 「另一个agent」「另一个会话标签」「同一模型的不同温度参数」均不构成独立
- 不同后端的具体判定：不同公司（Anthropic/OpenAI/DeepSeek/Google）→独立；同一公司不同型号→半独立（需在报告中声明并降权）

**轴2: 上下文隔离性**
- 审查者不得接触作者的中间推理、评分、结论
- 审查者只能接触：源材料（被审查对象）+ 审查rubric + 裸环境（见§5）

### 2.2 独立性三级分类

| 级别 | 轴1 | 轴2 | 标签 | 如何使用 |
|------|-----|-----|------|---------|
| **独立 (Independent)** | 不同后端 | 完全隔离 | `[IND]` | 可作为一级证据 |
| **半独立 (Semi-Independent)** | 同公司不同型号 或 轴2部分隔离 | `[SEMI]` | 可作为二级证据，需声明局限性 |
| **非独立 (Non-Independent)** | 同一后端 或 轴2未隔离 | `[NON]` | 不可作为独立审查使用，仅作内部交叉检查 |

### 2.3 独立性判定决策树

```
审查者后端 ≠ 作者后端？
  ├─ 是 → 上下文完全隔离？
  │        ├─ 是 → [IND] 独立
  │        └─ 否 → [SEMI] 半独立（声明泄露点）
  └─ 否 → 同公司不同型号？
           ├─ 是 → 上下文完全隔离？
           │        ├─ 是 → [SEMI] 半独立
           │        └─ 否 → [NON] 非独立
           └─ 否 → [NON] 非独立
```

## §3 角色分工

### 3.1 三角色模型

| 角色 | 谁干 | 不能是谁 | 职责 |
|------|------|---------|------|
| **作者 (Author)** | 产出被审查内容的模型/人 | — | 提供源材料，回答事实性问题（但不得透露自身评分/推理） |
| **操作者 (Operator)** | 执行审查流程的人（通常是用户或指定的协调agent） | 不能是作者模型，不能是审查者模型 | 搭建裸环境、传递材料、收集结果、确保信息防火墙 |
| **审查者 (Reviewer)** | 执行独立评分的模型 | 不能与作者同后端，不能接触作者中间产出 | 独立阅读源材料→评分→挑战→终判 |

### 3.2 操作者的信息防火墙职责

操作者是独立性的守门人：
- 从作者处提取源材料时，剥离所有评分、自评、结论性判断
- 向审查者传递前，验证裸环境完整性（§5 checklist）
- 不得向审查者透露「作者认为哪里好/哪里有问题」
- 不得向作者透露审查者的评分直到终判完成

## §4 执行流程

### 4.1 四步一核全景图

```
源材料 → [步骤一: 事实提取] → [步骤二: 独立评分] → [步骤三: 对抗式挑战] → [步骤四: 终判]
              ↑                      ↑                    ↑                    ↑
         操作者传递              评分在挑战前！         挑战后调整评分         独立性声明嵌入
         (裸环境验证)           (防污染规则)           (adversarial turn)     (provenance锁定)
```

### 4.2 关键约束

1. **步骤二（评分）必须在步骤三（挑战）之前**：评分须基于审查者对材料的初次理解，而非已被对抗式挑战塑造后的理解。此顺序不可颠倒——颠倒将导致「审查者被自己的挑战带偏」的污染。
2. **步骤三必须包含「魔鬼代言人不得软化」约束**：挑战方须尽最大努力找出问题，不得以「但总的来说还是不错的」收尾。
3. **每步产出必须标注provenance**：模型后端+会话标识+时间戳。

## §5 裸环境搭建

### 5.1 九项Checklist

审查者在开始审查前，操作者须逐项确认以下隔离措施：

| 编号 | 检查项 | 通过标准 | 验证方法 |
|------|--------|---------|---------|
| **C1** | 新会话/新对话 | 审查者会话不包含任何项目历史 | 检查会话开头无上下文 |
| **C2** | 无memory注入 | 审查者不加载用户的memory/偏好文件 | 确认未传递MEMORY.md或等效文件 |
| **C3** | 无项目文档 | 审查者不接触项目的其他文档 | 只传递本SOP + 源材料 + rubric |
| **C4** | 无作者评分 | 源材料中剥离所有自评/结论 | 操作者预处理源材料 |
| **C5** | 无中间产出 | 审查者不看到作者草稿/推理链 | 只传递最终产出 |
| **C6** | 不同后端 | 审查者使用与作者不同的模型后端 | 记录双方后端provenance |
| **C7** | 无先验标签 | 不告知审查者「这是谁写的」「什么水平」 | 匿名化源材料 |
| **C8** | rubric 纯净 | 审查rubric不含导向性语言 | 操作者审核rubric措辞 |
| **C9** | 时间戳记录 | 每步操作记录时间戳 | 用于后置审计 |
| **C10** | 版本绑定（v1.4 新增） | 被审材料版本标识已记录且不可变（arXiv 版本号+日期 / OpenReview camera-ready 日期 / git commit hash / 文件 hash 至少一项） | 审查启动前记录；审查过程中发现版本不一致→在报告中显式标注差异 |

### 5.2 退化检测

如果某项checklist不可行（例如只有一个后端可用），须在报告中显式声明退化级别：
- **完全裸环境**：C1-C9全部通过 → [IND]可达
- **部分退化**：C6不可行（同后端）→ 最多 [NON]
- **严重退化**：C1-C4任一不可行 → 流程降级为「内部交叉检查」，不适用本SOP

## §6 步骤一：事实提取

### 6.1 置信层级

审查者从源材料中提取事实性声明，逐条标注置信度：

| 层级 | 标签 | 定义 | 示例 |
|------|------|------|------|
| **T1** | `[VERIFIED]` | 可被公开可查来源验证 | 引用自权威文献、官方文档、可复现的实验结果 |
| **T2** | `[CONSISTENT]` | 与审查者自身知识一致 | 与模型训练数据中的共识一致 |
| **T3** | `[PLAUSIBLE]` | 逻辑自洽但无法验证 | 推理链内部一致，但缺乏外部锚点 |
| **T4** | `[UNVERIFIABLE]` | 无法判断真伪 | 超出审查者知识范围且无法查证 |

### 6.2 声明分类

每条提取的声明标注类型：
- `[FACT]` — 可验证的事实断言
- `[INFERENCE]` — 从事实推导的结论
- `[ASSUMPTION]` — 未声明的假设
- `[VALUE]` — 价值判断/偏好

### 6.3 工具受限标注

如果审查者因工具限制无法验证某声明，须标注 `[TOOL-LIMITED]` 而非降级置信度。例如：无法执行代码验证计算结果、无法访问付费数据库。

## §7 步骤二：独立评分

### 7.1 六维评分框架 (D1-D6)

| 维度 | 名称 | 评分范围 | 说明 |
|------|------|---------|------|
| **D1** | 逻辑有效性 | 1-5 | 推理链是否无逻辑跳跃/循环/矛盾 |
| **D2** | 事实准确性 | 1-5 | 可验证事实是否正确（参考§6 T1-T2） |
| **D3** | 方法适当性 | 1-5 | 所选方法是否适合研究问题 |
| **D4** | 完整性 | 1-5 | 是否覆盖了必要的维度/边界条件/替代解释 |
| **D5** | 清晰度 | 1-5 | 表达是否清晰、结构是否合理 |
| **D6** | 新颖性/洞察深度 | 1-5 | 是否有超出常规的洞察（对方法论资产尤其重要） |

### 7.2 评分约束

- **评推理不评结论**：D1/D3-D6 评「推理如何到达结论」，不评「结论对不对」。**D2（事实准确性）例外**：先验证事实断言和结论可靠性，再评推理质量。D2 的核心是事实是否正确——这与结论可靠性直接相关，不适用"不评结论"约束。
- **N/A 允许**：如果某维度不适用于该材料类型，标记N/A并说明理由
- **锚定说明**：每项评分须附一句话锚定（为什么给这个分）
- **不知为不知**：审查者缺乏某维度领域知识时，标记 `[OUT-OF-DOMAIN]` 而非强行评分

### 7.3 综合评分

综合评分 = 各维度评分的算术平均（N/A维度排除）。不做加权——加权本身是价值判断，应由人类用户决定。

## §8 步骤三：对抗式挑战

### 8.1 五方面必填挑战 (A-E)

审查者须从以下五个方面对源材料进行系统性挑战。每个方面至少提出一个具体挑战：

| 编号 | 挑战方面 | 触发问题 |
|------|---------|---------|
| **(A)** | 替代解释 | 「有没有其他解释能同样好地说明这些事实？」 |
| **(B)** | 隐藏假设 | 「哪些假设被当成了既定事实？」 |
| **(C)** | 边界条件 | 「在什么情况下这个结论会失效？」 |
| **(D)** | 反例/反证 | 「有没有与结论矛盾的事实或案例？」 |
| **(E)** | 方法论替代 | 「用另一种方法会不会得出不同结论？」 |

### 8.2 挑战约束

1. **魔鬼代言人不得软化**：每个挑战须以最尖锐的形式提出，不得附加缓和语（「不过可能也没那么严重」「但总体来说还是合理的」——禁止）
2. **具体而非抽象**：挑战须指向材料中的具体段落/声明/推理步骤，不得泛泛而谈
3. **挑战后评分调整**：完成(A)-(E)所有挑战后，审查者重新审视步骤二的评分。如果挑战说服了自己，须调整评分并记录调整幅度和理由

### 8.3 挑战后评分调整规则

```
IF 挑战发现了步骤二未注意到的问题：
  → 下调相关维度评分，记录「从X调整为Y，原因：挑战(Z)揭示...」
IF 挑战未发现新问题（材料经受住挑战）：
  → 维持原评分，记录「经受住(A)-(E)挑战，评分不变」
IF 挑战发现了致命问题：
  → 触发红旗R1-R3（见§12），可能直接建议「弃用」
```

## §9 步骤四：终判

### 9.1 四级终判

| 判决 | 含义 | 后续行动 |
|------|------|---------|
| **保留 (Keep)** | 材料质量合格，可直接使用 | 标注审查结论，归档 |
| **小改 (Minor Revision)** | 有可改进之处但不影响核心结论 | 列出修改建议，作者修改后无需重新审查 |
| **重大修改 (Major Revision)** | 存在影响核心结论的问题 | 列出问题，作者修改后须重新审查 |
| **弃用 (Discard)** | 材料不可用，核心结论不可靠 | 记录弃用原因，触发根因分析 |

### 9.2 终判须包含

1. 综合评分（D1-D6平均）+ 各维度明细
2. 独立性级别（[IND]/[SEMI]/[NON]）
3. 关键挑战摘要（A-E中最重要的2-3个挑战）
4. 终判结论 + 理由
5. 独立性声明（见9.3）

### 9.3 独立性声明格式（强制嵌入）

```yaml
independence_declaration:
  reviewer_backend: "<模型名称，如 deepseek-v4-pro>"
  author_backend: "<作者模型名称>"
  axis1_different_backend: true/false
  axis2_context_isolated: true/false
  independence_level: "IND | SEMI | NON"
  degradation_notes: "<退化说明，如无可省略>"
  session_id: "<审查者会话标识>"
  timestamp: "<ISO 8601>"
```

## §10 后置核验

### 10.1 九项审计 (A1-A9, v1.4 新增 A9)

在审查完成后，由操作者（或用户指定的审计者）对审查过程进行审计：

| 编号 | 审计项 | 通过标准 |
|------|--------|---------|
| **A1** | 后端验证 | 审查者后端 ≠ 作者后端 |
| **A2** | 会话隔离 | 审查者会话无项目上下文泄露 |
| **A3** | 评分在前 | 步骤二的评分时间戳早于步骤三的挑战时间戳 |
| **A4** | 挑战不软化 | 步骤三的(A)-(E)不包含缓和语 |
| **A5** | provenance完整 | 每步有后端+会话+时间戳记录 |
| **A6** | 声明嵌入 | 终判包含完整的independence_declaration |
| **A7** | 调整有据 | 如有评分调整，记录了调整幅度和理由 |
| **A8** | N/A有理 | 如有N/A标记，有合理说明 |
| **A9** | 版本绑定（v1.4 新增） | 审查对象版本标识已记录且可追溯；如审查过程中发现版本不一致，已显式标注差异并讨论对结论的影响 |

### 10.2 审计结论三级

| 审计结论 | 条件 | 后续 |
|---------|------|------|
| **审计通过** | A1-A9全部通过 | 审查结论可作为正式证据使用 |
| **审计有条件通过** | 非关键项不通过（A5/A7/A8/A9），独立性核心项（A1-A4/A6）通过 | 标注条件，可使用但需说明局限性 |
| **审计不通过** | A1-A4或A6任一不通过 | 审查结论不可使用，需重新审查 |

## §11 输出格式

### 11.1 双件规范

每次独立审查产出两份文件：

1. **`.md` 文件**（人读）：完整审查报告，包含所有步骤、评分、挑战、终判、审计结论
2. **`.json` 文件**（机读）：结构化数据，字段与md章节一一对应

### 11.2 JSON结构规范

```json
{
  "metadata": {
    "title": "独立审查报告",
    "version": "1.0",
    "date": "2026-06-13",
    "model": "DeepSeek-v4-pro (via Claude Code CLI shell)",
    "sop_version": "v1.0"
  },
  "trigger": {
    "type": "T1|T2|T3|T4|S1|S2|S3",
    "description": "..."
  },
  "source_material": {
    "title": "...",
    "author_backend": "...",
    "type": "methodology|analysis|conclusion|other",
```

## §12 红旗条款 (R1-R9)

以下任一红旗出现，审查流程须立即暂停，由操作者决定后续处理：

| 编号 | 红旗 | 触发条件 | 建议处理 |
|------|------|---------|---------|
| **R1** | 核心事实错误 | 步骤一发现T2级以上的事实错误且影响核心结论 | 直接建议「弃用」，不进入步骤二 |
| **R2** | 循环推理 | 结论被用作推导自身的前提 | 建议「重大修改」 |
| **R3** | 范畴错误 | 方法不适用于研究问题类型 | 建议「弃用」或「重大修改」 |
| **R4** | 独立性伪证 | 审计发现审查者实际接触了作者中间产出 | 审查结论作废，重新审查 |
| **R5** | 换壳不换模 | 「审查者」和「作者」实际是同一后端的不同包装 | 审查结论作废 |
| **R6** | 审查衰减 | 审查者的评分/挑战显著低于该模型在类似任务上的历史表现 | 标记为「低置信度」，考虑补充审查 |
| **R7** | 数据不可用替代 | 审查者因数据不可用而使用了替代数据，但未声明替代的局限性 | 标记为「数据污染风险」 |
| **R8** | 时间错位 | 审查者使用了已过时的信息验证「当前」结论 | 标记时间局限性 |
| **R9** | 裸环境泄露 | 审计发现C1-C9任一未通过但未在报告中声明 | 审查结论降级 |

## §13 常见失败模式

### 13.1 伪独立 (Pseudo-Independence)

**现象**：「另一台机器上的同一个模型」被当作「独立审查者」
**检测**：轴1后端相同 = 非独立，无论包装成什么
**案例**：Claude Code的不同agent做对抗——同一后端，非独立
**预防**：轴1强制不同后端（§2.1），provenance当场记录

### 13.2 审查衰减 (Review Decay)

**现象**：审查者的对抗性随时间递减——第5次审查比第1次显著温和
**检测**：对比同一审查者在不同时间点的(A)-(E)挑战尖锐度
**预防**：魔鬼代言人不得软化约束（§8.2），审计A4

### 13.3 工具受限伪装 (Tool-Limitation Camouflage)

**现象**：审查者因工具限制无法验证，但未标注[TOOL-LIMITED]，而是给出了模糊的评分
**检测**：逐条检查步骤一声明的验证方式
**预防**：§6.3强制标注规则，§7.2的N/A和[OUT-OF-DOMAIN]机制
