> **生成模型**: DeepSeek-V4-Pro
> **分析者**: Claude Code (方法论对标分析)
> **对标对象**: headroom CacheAligner (detector-only 缓存前缀稳定性检测器) vs AI 协作框架 v1.2 §3.6 OPEN-1 (连续漂移/离散审查缺口)
> 
> **方法论前提声明（P0-2，2026-06-13）**：本对标建立在 headroom（确定性工程域）与 AI 协作框架（语义治理域）的类比之上。以下维度可能不可迁移：(1) **答案确定性**——headroom 操作在字节层（检测=hash比较、错误=关键词匹配），框架操作在语义层（方向漂移=上下文判断）；(2) **模式可穷举性**——headroom 检测对象有先验定义（UUID/ISO8601/JWT/HexHash 是封闭集合），框架漂移形态是开放集合；(3) **反馈信号客观性**——headroom 反馈自动可量化（检索率、token数），框架反馈依赖人类语义判断；(4) **代价结构可比性**——headroom 错误代价=多保留几行 token，框架错误代价=项目方向偏转。本对标的核心迁移假设未经独立验证，应被视为**方法论启发**而非操作建议。详见配套《深度审阅报告》§2 F2。

## 1. 背景

### 1.1 CacheAligner：一个"死后才被正骨"的 detector-only 检测器

CacheAligner 是 headroom SDK 中 `headroom/transforms/cache_aligner.py` 的核心组件。它的当前形态是 **PR-A2 / P2-23 修复后的修正版**——此前它有一条 rewrite 路径，会从系统提示词中剥离动态内容再重新注入，结果违反了 invariant I2（"缓存热区绝不可被修改"）。修复后，rewrite 路径全部移除，CacheAligner 变成一个纯粹的检测器：

- 用**结构化解析器**（非正则）检测四类易变内容：UUID（`uuid.UUID()`）、ISO 8601 时间戳（`datetime.fromisoformat()`）、JWT（三段 base64url 形状检查）、Hex 哈希（MD5=32/SHA1=40/SHA256=64 字符长度+纯十六进制检查）
- 发出客户可见的警告日志："CacheAligner: detected volatile content in system prompt (uuid=3, iso8601=1); cache prefix unstable."
- 计算系统消息的稳定 SHA-256 hash，跨请求追踪 `prefix_changed`
- `get_alignment_score()`：100 - 10 * 检测到的易变模式数
- **关键约束**：`apply()` 方法的 `result.messages` 与输入字节相等——提示词永不改写。动态内容应路由到"live zone"（最新用户轮次），而非系统提示词。

### 1.2 OPEN-1：框架最致命的结构性缺口

AI 协作全生命周期框架 v1.2 §3.6 将 OPEN-1 登记为 §1.6 已知待决项的**最高杀伤发现**。核心诊断：

- 框架的 4 个 Human Gate（HG-0/1/2/3）全部是**离散事件触发**（里程碑、Spec 变更、项目闭合）
- AI 在 Loop 内的执行是**连续**的——每次迭代都可能进行微小的方向调整
- 这些微调每个都在当前 Prompt 约束内、不触发任何闸门，但累积效应可能导致**方向漂移**
- 最可能的失败模式不是大爆炸，而是缓慢漂移：AI 漏记 Dev Log（~15% 估计漏记率）→ Retrospect 基于不完整记录 → Spec 被不完全更新 → 下轮 Prompt 基于微偏的 Spec → 偏差累积
- 候选机制：Loop 内插入"轻量级方向一致性自我报告"——AI 每 N 轮自报偏差
- **独立性状态**：ChatGPT-5.5（唯一独立于框架创作的外部评审）未接战；OPEN-1 来自作者自识 + 同谱系 Opus 背书，佐证偏弱

### 1.3 为什么将两者放在一起看

CacheAligner 和 OPEN-1 关注的是**同一个底层问题在不同领域的投影**：离散检查无法覆盖连续变化。CacheAligner 面对的"连续变化"是缓存前缀字节级漂移（每次 API 调用间系统提示词悄悄变了），OPEN-1 面对的"连续变化"是 AI Loop 内的语义方向漂移（每轮迭代间目标悄悄偏了）。两者的共同结构是：

```
离散检查点 → 检查点之间发生连续微小变化 → 离散检查点在下一触发时才发现问题 → 损失已累积
```

但一个已经有工作实现和真实教训，一个还在候选机制阶段。CacheAligner 的设计决策——特别是 detector-only 哲学——对 OPEN-1 有直接的方法论启发。

## 2. 结构映射

| 维度 | CacheAligner | OPEN-1（当前候选） | 映射关系 |
|------|-------------|-------------------|---------|
| **检测对象** | 系统提示词中的易变内容（UUID / 时间戳 / JWT / 哈希） | Loop 迭代间的语义方向变化（筛选条件放宽 / 权重替换 / 阈值调整） | 都是"可检测的微小变化"，但 CacheAligner 是**字节级、结构化**的，OPEN-1 是**语义级、非结构化**的 |
| **检测方法** | 结构化解析器（stdlib 确定性函数，无 regex，无 ML） | 候选：AI 自我报告（"本轮产出与上个里程碑偏差多少？"） | CacheAligner 是**完全确定的**（给定输入，输出恒等）；OPEN-1 候选是**模糊自评**（AI 自己判断自己偏了多少） |
| **失败模式** | 动态内容误入系统提示词 → 缓存前缀不稳定 → 每次请求 cache miss | Loop 内方向微调累积 → Spec 被微偏更新 → 下轮迭代延续偏差 → 到里程碑才发现 | 同构：都是"微变累积→延迟发现→损失已发生" |
| **设计哲学** | **检测但不改写**（detector-only）：发现问题→发出警告→由调用方决定如何修复（路由到 live zone），自己不动手 | **自报但不阻断**（候选）：不要求人确认（避免 HG-2 式高摩擦），AI 自己报告偏差→作为 HG-1 的前置输入 | 都是"新增信号，不新增闸门"——检测信号和处置动作解耦 |
| **处置策略** | 警告 + hash 追踪 + 对齐评分 + 建议："Move dynamic values out of the system prompt to recover cache hits." | 自我报告 → HG-1 时人阅览 → 人判断是否修正 | CacheAligner 的处置链完整（检测→信号→建议→外部处置），OPEN-1 候选只有检测→信号，缺乏中间链路 |

## 3. CacheAligner 的可迁移设计决策

### 3.1 结构化模式检测（非模糊自评）

CacheAligner 最核心的设计选择是：**用确定性解析器检测具体模式，而非让 AI 判断"这段文本是否包含易变内容"**。

它的四个检测函数（`_is_uuid`、`_is_iso8601`、`_is_jwt_shape`、`_is_hex_hash`）全部使用 Python stdlib 的结构化解析器，没有 regex，没有 ML，没有启发式阈值。规则是硬编码的：

```python
_HEX_HASH_LENGTHS = frozenset({32, 40, 64})  # MD5/SHA1/SHA256
_UUID_CANONICAL_LEN = 36                       # 拒绝 32 字符无破折号形式（会与 MD5 混淆）
_JWT_SEGMENT_COUNT = 3
_JWT_MIN_SEGMENT_BYTES = 4
```

这意味着：**相同输入必然产生相同检测结果**，没有解释弹性，没有"可能/大概"。

映射到 OPEN-1：当前候选机制的"方向一致性自我报告"本质上是 AI 对自身行为的语义判断——同一轮迭代让不同 AI 模型（甚至同一模型不同调用）报告偏差，结果可能完全不同。CacheAligner 的方法论启示是：**如果能把"方向漂移"的形式化为可检测的结构化信号，就可以绕开模糊自评的不确定性**。

具体来说，如果能在 Loop 执行中捕捉以下结构化信号：
- Spec 中某个数值型约束（比如"筛选条件：PE < 15"）在本轮被替换为不同数值
- Prompt 中的参数网格与上一轮产生了差异
- 产出文件中关键数字（阈值、权重、样本量）的变动序列

那么方向漂移就可以变成**确定性检测**而非 AI 自述。

### 3.2 Hash-based 变化追踪

CacheAligner 的第二层设计是 SHA-256 hash 追踪：

```python
system_text = "\n---\n".join(
    (m.get("content") or "")
    for m in result_messages
    if m.get("role") == "system" and isinstance(m.get("content"), str)
)
stable_hash = compute_short_hash(system_text)
prefix_changed = (
    self._previous_prefix_hash is not None
    and self._previous_prefix_hash != stable_hash
)
```

它追踪的**不是"哪些具体内容变了"**，而是"前缀是否与上次不同"。这是一个用最小信息量回答最关键问题的设计：**"东西变了吗？"**——而不试图回答"变成了什么？"

映射到 OPEN-1：当前框架完全没有一个机制追踪**上一轮 Prompt 和这一轮 Prompt 之间的差异**。Key Spec 字段（目标、约束、评估指标）如果在 Loop 间被微调，这个变化不会被记录下来——它没有触发 HG-2（因为 Spec 文件没改），也不触发 HG-2 分级（变更是隐式的）。如果对每个 Loop 产出的 Key Spec 相关字段做 hash，Open-1 的方向漂移就从"当事人自我报告"变成"观测者独立检出"：

```
Loop N 开始时的 Spec hash  →  Loop N 结束时的 Spec hash
                ↓                        ↓
              比较 → hash 变了？→ 什么东西变了？→ 是否符合预期方向？
```

**这个 hash 不是替代 HG-1 人的裁决，而是给 HG-1 提供一条无需依赖 AI 自我报告的独立信号线。**

### 3.3 Detector-only 哲学（检测但不改写）

这是 CacheAligner 最深刻的架构教训，也是它对 OPEN-1 最有价值的方法论贡献。

**被移除的 rewrite 路径做了什么**：早期 CacheAligner 检测到系统提示词中的动态内容后，会主动将其剥离并重新注入为上下文块——它想"修好"缓存前缀。但这个行为违反了 invariant I2（缓存热区绝不可被修改）。修复是彻底的：整条 rewrite 路径删除，动态内容必须由调用方自行路由到 live zone。

**教训**：检测器不能既当裁判又当球员。检测和处置必须在不同层完成。检测告诉调用方"这里有 X 类易变内容"，但修复动作——把它从系统提示词中移走——是调用方的责任。

映射到 OPEN-1：当前候选机制的"AI 每 N 轮自报偏差"隐含了一个类似的风险——**AI 既是漂移的制造者，又是漂移的报告者**。如果 OPEN-1 落地为"AI 自报→AI 自己根据报告调整方向"，那就重复了 CacheAligner 被删除 rewrite 路径的同构错误：同一实体既检测问题又尝试修复问题，失去了独立校验。

> **关键区分（P0-1 修正，2026-06-13）**：AI 自我报告并非铁板一块。"我偏了多少"（主观自评）和"我动了什么参数"（客观事实陈述）是两类完全不同的报告。前者与 CacheAligner 被删除的 rewrite 路径同构——AI 自判自修，失去独立校验。后者则与 CCR 的压缩标记同构——AI 报告客观变动事实，裁决权留给 HG-1 的人。**本节批评的对象是前者（主观自评），后者（客观变动记录）是 §4.2 层 1 确定性检测的合理补充。** 详见配套文档 `SmartCrusher方法论提取.md` §6.4（已通过交叉引用调和立场差异）。

正确的落地方式应该是：
1. **检测层**（detector）：确定性信号检测（hash 追踪 + 结构化模式匹配）+ AI 辅助语义标注（"这轮变了对阈值的理解"）
2. **报告层**（reporter）：将检测信号格式化为 HG-1 的前置输入——"以来上一个里程碑，检测到以下方向变动：[列表]"
3. **处置层**（decider）：HG-1 时人类裁决——"这些变动在正确方向上吗？需要回退吗？"

**检测信号和处置动作的解耦，是 CacheAligner 用一条被删除的代码路径换来的教训。**

## 4. 对 OPEN-1 候选机制的启发式观察与假设

> **定位修正（P1-1，2026-06-13）**：本章内容从"修正建议"降级为"启发式观察"。确定性检测信号在OPEN-1场景中的有效性取决于噪声比——若90%+ hash变动为正常迭代进展，本方案信号价值趋近于零。三层架构应被视为待验证的假设，而非可立即采纳的建议。该假设需在下个真实项目中试跑验证后方可考虑纳入框架正文。

### 4.1 当前候选的问题

当前框架 §3.6 的候选机制：

> "在 Loop 中插入轻量级方向一致性自我报告——不要求人确认，但要求 AI 每 N 轮自我报告一次：'本轮产出与上个里程碑设定的方向偏差了多少？哪些约束被隐式放宽了？'"

这个候选有三个问题：

1. **自我指涉**：AI 既是漂移的制造者又是漂移的评估者。等同于让运动员为自己的动作打分。
2. **模糊性**："偏差了多少"没有锚定标准——同一轮迭代的不同 AI 调用可能给出完全不同的自评分数。
3. **缺少触发信号**：自我报告只在"每 N 轮"触发。但如果方向在第 N-2 轮就已经偏了，等到第 N 轮才自报，已经损失了两轮的效率。这和现有的 HG-1 离散触发是同构的问题——只是把触发间隔从"里程碑间"缩短到"N 轮间"，没有从根上改变"离散检查覆盖连续变化"的结构性缺陷。

### 4.2 修正方案：结构化漂移信号检测 + 确定性 hash 追踪

基于 CacheAligner 的 detector-only 哲学，一个待验证的假设是将 OPEN-1 从"AI 自我报告"升级为**双层检测架构**：

**层 1：确定性结构信号检测（自动化，可复现）**

在每轮 Loop 结束时，运行一个无 AI 参与的确定性检查器，抓取以下信号：

| 信号 | 检测方式 | 含义 |
|------|---------|------|
| Spec hash 变动 | 对当前 S4（成功判据）和 S9（约束条件）字段做 SHA-256 | 本轮产出蕴含的 Spec 解释是否与上一轮不同 |
| Prompt 参数漂移 | 提取 Prompt 中的数值参数（阈值、权重、筛选条件），与上轮 Prompt 做逐字段 diff | 哪个约束被隐式放宽或收紧 |
| 输出结构变动 | 产出文件的 schema hash（字段增减、类型变化） | 是否在迭代中不知不觉改了输出格式 |
| Loop 轮次内的决策数 | 本轮 AI 做了多少个"选择"（选参数、选方法、选数据源） | 选择密度高的轮次更可能引入方向偏移 |

这些信号**不需要 AI 参与判断**——它们是对文本/文件做确定性计算的产物。信号本身是客观事实（"S4 的 hash 变了"），不是主观判断（"我觉得偏了"）。

**层 2：AI 辅助语义标注（在确定性信号之上加一层解释）**

确定性信号告诉人"什么东西变了"，但不告诉人"这个变化是否合理"。这里才是 AI 的用武之地——**但不是让 AI 自评，而是让 AI 对确定性信号做语义标注**：

```
本轮检测到的方向信号：
  - Spec hash 变动: S4 字段 hash 从 abc123 → def456
  - 参数漂移: 筛选条件 PE<15 → PE<20（放宽 33%）
  - 变更理由(由 AI 标注): "数据量不足，需扩大样本池以满足 S6 最小样本量要求"

上一轮与本轮的方向变动摘要(由 AI 生成):
  - 方向向量: [PE 筛选, -5] [信号权重, stock_selection +0.05] [评估阈值, sharpe -0.1]
  - 综合判断: 两轮累计向"扩大样本、容忍低 Sharpe"方向偏移
  - 信心水平: 高（三个变动信号均有明确的 Prompt 内参数对应）
```

注意这里的 AI 角色不同——它不是"评估自己偏了多少"，而是"解读确定性检测器已经发现的事实"。解释可以错（AI 对理由的猜测可能不对），但事实信号不会错（PE 从 15 变成 20 是客观的）。**人拿到的是"事实 + AI 解读"，可以独立判断后者是否可信。**

### 4.3 落地节奏

```
阶段 1（低成本，可立即上）:
  - Loop 结束时对 Spec 关键字段做 hash，写入 Dev Log
  - 下一轮 Loop 开始时比对 hash
  - 无需 AI 参与，纯计算

阶段 2（需要 AI 集成）:
  - 在 Prompt 中嵌入"结束时请报告本轮对以下 Spec 字段的实质变动"指令
  - AI 输出结构化变动摘要（不是"偏了多少"，而是"动了什么、动了多少、为什么"）
  - 变动摘要作为 HG-1 的前置输入

阶段 3（需要工具链）:
  - 将变动摘要纳入 Retrospect 的 Spec 写回流程
  - 如果连续多轮变动向量高度一致（比如始终向"放宽约束"方向偏），自动触发预警
```

## 5. 关键差距与局限性

将 CacheAligner 的设计直接移植到 OPEN-1 存在本质性的困难，需要诚实面对。

### 5.1 已知模式 vs 无先验模式

这是两者之间最深的鸿沟。CacheAligner 检测的四类易变内容（UUID、ISO 8601、JWT、Hex Hash）是**已知的、定义明确的、边界硬**的模式。长度、字符集、结构约束都是精确的——一个 36 字符且有 4 个破折号的字符串就是 UUID，不是别的。检测逻辑完全是 if-else 的分支判断，零模糊空间。

OPEN-1 的方向漂移则可能有**无限多种形态**——今天放筛选条件，明天改模型参数，后天换评估口径。没有一个"漂移模式的封闭列表"，因为它恰好是"我们没预料到的变化"才构成真正的漂移风险。

**这意味着 CacheAligner 的结构化模式匹配不可直接平移**——我们不能定义 `_is_direction_drift(token)` 函数，因为方向漂移没有 token 级别的粒度，也没有先验的模式定义。

### 5.2 字节级 vs 语义级

CacheAligner 工作在**字节层**：它拆分文本为 token，逐个检查每个 token 是否匹配某个易变模式。检测结果是非黑即白的——"这个字符串是 JWT"或"不是"。

OPEN-1 工作在**语义层**：PE < 15 变成 PE < 20，在字节层面是 `"15"` 变成了 `"20"`（检测到，简单），但在语义层面是"筛选条件放宽了 33%"。**"放宽"是不是"漂移"取决于上下文**——如果上一轮就因为样本量不足而卡住，放宽筛选条件恰好在正确方向上。如果只是 AI 偷懒想跑更快出结果，那就是漂移。

**语义标注无法全自动化**——这也是为什么 CacheAligner 的结构化检测可以完全不需要 AI，而 OPEN-1 的漂移检测必须有（人在回路的）语义判断。这不是设计选择的差异，而是问题本体的差异。

### 5.3 补救思路：从封闭模式到开放锚点

虽然无法定义"漂移的封闭模式列表"，但可以定义**漂移的开放锚点集**：

1. **Spec 锚点**：S4（成功判据）、S9（约束条件）中的数值约束、方法约束、数据约束——这些是"不该变"的东西。检测它们**是否变了**（而非判断变得是否合理）。
2. **Prompt 锚点**：每轮 Prompt 中显式声明的约束条件——提取结构化字段做逐轮 diff。
3. **产出锚点**：产出文件中的关键数字（样本量、收益率、显著性水平）——追踪它们是否在 Loop 间沿一个方向连续移动。

这些锚点不是"漂移模式"，而是**漂移会留下痕迹的位置**。检测器不需要知道"什么是漂移"，只需要知道"这里如果变了就值得报告"。

## 8. 结论

CacheAligner 对 OPEN-1 的核心贡献不是一段可复用的代码或算法，而是**一条用一条被删除的代码路径换来的架构原则**：检测和处置必须解耦，检测信号必须是确定性的，同一实体不能既制造问题又检修问题。

OPEN-1 的当前候选机制（"AI 自我报告方向偏差"）恰好落入了这个陷阱——让 AI 自判自报告，就像让 CacheAligner 的 rewrite 路径既检测又修复系统提示词。修正方向不是否定 AI 的参与，而是**重新定义 AI 的角色**：从"自我评估者"变成"确定性信号的语义标注者"。事实由检测器说话（hash diff 不会说谎），解释由 AI 提供（人可以独立判断），决策由人做出（HG-1 裁决）。

以上主张建立在一个未经真实项目验证的类比假设上（headroom工程域→框架治理域），应被视为方法论启发而非操作建议。

最终，这不是 CacheAligner 拯救 OPEN-1 的故事——两者是同一个元问题（离散检查覆盖不了连续变化）在两个领域（缓存工程 vs AI 协作治理）的独立投影。CacheAligner 已经完成了它的教训提取（rewrite 路径被删了，detector-only 留下了），OPEN-1 的教训提取还在路上。但方向已经清楚：**不新增闸门，新增检测信号；不依赖自评，依赖确定性事实；检测层和处置层永远不在同一个实体手里。**
