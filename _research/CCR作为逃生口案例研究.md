## 1. 概述

CCR（Compress-Cache-Retrieve）是 headroom 平台中实现**可逆压缩**的核心架构组件。它的基本逻辑简单到用四行描述：

1. 压缩工具输出时，把原始数据存入缓存，用一个哈希指纹标记压缩结果。
2. 向 LLM 注入 `headroom_retrieve` 工具，让模型在需要时按哈希检索原始数据。
3. 拦截 LLM 的检索调用，从缓存中取回原始数据，透明地继续对话。
4. 跨对话轮次追踪哪些压缩数据可能和新查询相关，主动扩展上下文。

但 CCR 的意义远不止一个"缓存+检索"工程实现。从安全架构和逃生口设计的角度看，它代表了一种**范式升级**——把逃生口从"被动停止（stop-and-report）"升级为"主动检索恢复（retrieve-and-continue）"。

### 为什么是逃生口设计模式的重要案例

传统逃生口的设计直觉是"当信息不可靠时，停止"。AI 协作项目全生命周期框架的 §4.3 逃生口（本文以下简称"框架 §4.3"）的三条规则——改 Spec → 触发 HG-2 / 前提假设不成立 → 停止并报告 / 工具/数据不可用 → 停止并报告、绝不用替代数据——都遵循这个范式：**出问题时，阻断执行路径**。

CCR 基于一个不同的洞察：压缩本身创造了"信息可能不完整"的状态，但这种不完整是**有意为之且有恢复路径**的。如果每个压缩操作都在不可逆的前提下进行（"我决定丢掉哪些行，LLM 永远看不到那些行"），那么压缩就会变成一个高风险的赌博。CCR 把这个问题重构为：压缩可以激进（为节省 tokens），因为检索路径兜底（LLM 可以判断是否需要更多信息并主动取回）。

这种设计模式把逃生口从**围栏（barrier）**变成了**安全阀（relief valve）**：不是阻断执行，而是提供一条受控的恢复通道。

### 对框架的直接价值

框架目前只有一种逃生口反应模式——停止并报告。CCR 展示了一种中间可能性：信息可能不完整，但存在已知的检索路径时，不必立即停止。这种分级思维可以直接丰富框架 §4.3 的能力层次。

## 2. CCR 四阶段架构

CCR 的流程可以分解为四个独立阶段，每个阶段都有明确的安全边界和恢复机制。

### 2.1 Compression Store：压缩时存储原始数据 + 生成 hash

**这是 CCR 的基石。** 当 SmartCrusher 或其他压缩器处理工具输出时，在生成压缩表示的同时，原始完整数据被存入 `CompressionStore`，并生成一个哈希键作为检索索引。

**关键实现细节**：

- **哈希函数**: SHA-256 取前 24 hex 字符（96 位）。96 位在生日边界下约 2^48（约 280 万亿）个条目时有 50% 碰撞概率——对于进程内 LRU 缓存（默认 1000 条目、TTL 300 秒）而言绰绰有余。Rust 侧使用 BLAKE3 而非 SHA-256，但同样截断为 24 个 hex 字符以匹配 Python 侧的正则匹配模式。
- **标记格式**: 标准化为 `[N items compressed to M. Retrieve more: hash=abc123...]`（24 位 hex）——支持多种压缩器的统一检测（SmartCrusher / Kompress / LogCompressor / SearchCompressor 均使用同一标记格式）
- **存储后端可插拔**: InMemory（默认）/ SQLite（生产默认，持久化跨 worker 重启）/ Redis（多 worker 无需 sticky session），通过 `HEADROOM_CCR_BACKEND` 环境变量或 entry point 选择
- **TTL 机制**: 默认 300 秒（5 分钟），通过 `HEADROOM_CCR_TTL_SECONDS` 可配置。过期的条目在读取时惰性清除
- **Rust-Python 桥接**: SmartCrusher（Rust 侧）的行删除和模糊化路径生成 `<<ccr:HASH>>` 标记后将原始数据存入 Rust 的 `CcrStore`，Python 侧通过 `_mirror_ccr_to_python_store()` 将标记镜像回 Python `CompressionStore`，使 `/v1/retrieve` API 能解析所有标记来源。镜像失败仅在 debug 级别记录，永不中断压缩——因为压缩已成功，桥接只是让检索可用

**安全意义**: 压缩不再是"决定丢掉什么"，而是"决定先展示什么，完整的还在后面"。这消除了传统压缩中 LLM 永远无法知道"被丢掉了什么"的根本信息不对称。

### 2.2 Tool Injection：向 LLM 注入 headroom_retrieve 工具

当对话中出现压缩标记时，CCR 向 LLM 的工具列表中注入 `headroom_retrieve` 工具定义。该工具接受两个参数：`hash`（必填，检索键）和 `query`（可选，用于 BM25 搜索过滤）。

**三种注入模式**：

1. **工具定义注入**：将函数工具添加到请求的 tools 数组中。支持 Anthropic / OpenAI / Google 三种格式。
2. **系统消息注入**：在系统提示中追加检索说明，包含可用的 hash 列表。
3. **PR-B7 粘滞注册（sticky-on）**：一旦会话曾触发过 CCR 压缩，`headroom_retrieve` 工具在后续所有轮次保持注册——即使当前轮次没有新的压缩标记。这避免了工具列表在会话中途出现/消失从而导致 prompt cache 失效。

**安全意义**: Tool Injection 是让检索通道为 LLM 所知的机制。如果 LLM 不知道可以检索，CCR 就只是一次普通的不可逆压缩。工具定义必须在正确的时机（压缩已发生）和正确的位置（在 tools 列表中）出现，这正是逃生口设计中"提供界面，而非替决策"原则的体现——逃生口给了 LLM 一个选择，而不是替 LLM 决定是否需要。

### 2.3 Response Handler：拦截检索调用，透明返回原始数据

当 LLM 在响应中调用了 `headroom_retrieve`，Response Handler 拦截此工具调用，执行实际的检索操作，然后通过连续性 API 调用将工具结果反馈给 LLM。

**运行流程**：

1. 检测 LLM 响应中是否包含 `headroom_retrieve` 工具调用
2. 分离出 CCR 调用和其他工具调用——如果 LLM 同时调用了非 CCR 工具则跳过 CCR 处理（无法构造不完整的工具结果延续），由客户端处理
3. 对每个 CCR 调用执行检索：有 `query` 参数则执行 BM25 搜索，否则完整返回原始数据
4. 将检索结果作为工具结果消息附加到对话中
5. 发起连续性 API 调用（保持原 tools 列表），让 LLM 基于完整数据重新回复
6. 最多迭代 3 轮（`max_retrieval_rounds = 3`），防止无限循环

**安全意义**: Response Handler 是一个**非侵入式中间层**——它不修改 LLM 的生成逻辑，只是在 LLM 请求检索时透明地提供数据。这保持了 LLM 作为决策者的角色（LLM 判断需要什么），同时保证检索操作的可靠执行（代理层处理存储细节）。从逃生口角度看，这是一个"自主触发 + 系统执行"的模式：LLM 触发检索（就像飞行员触发弹射座椅），系统执行检索并提供数据（就像弹射机制展开降落伞）。

### 2.4 Context Tracker：跨轮次追踪，主动扩展相关数据

Context Tracker 解决了压缩带来的"上下文遗忘"问题：第 1 轮压缩了 500 个文件列表，到第 5 轮用户问"auth middleware 在哪"时，如果 LLM 已经忘记第 1 轮压缩过什么，就永远不会调用 `headroom_retrieve`。

**关键机制**：

- **追踪**: 记录每次压缩的 hash、轮次、时间戳、工具名、原始/压缩条目数、查询上下文、内容样本
- **工作空间隔离**: `workspace_key` 是每个 CompressedContext 的必填字段。`analyze_query` 只用匹配当前工作空间的上下文——这是 2026-05-26 一次跨项目泄露事故（Python 文件在 Ruby 会话中浮出）后加入的防护
- **时间折扣**: 越旧的压缩相关性分数越高衰减（5 分钟内线性衰减至 50%）
- **主动扩展**: 当新查询的 keywords 与追踪的压缩样本内容匹配度（BM25-based heuristic）超过阈值（默认 0.3）时，主动将相关原始内容注入上下文
- **每轮限制**: 最多主动扩展 2 个压缩条目，防止注入过多导致新一轮压缩的循环

**安全意义**: Context Tracker 是 CCR 从"被动逃生口"升级到"主动逃生口"的关键。它不在问题出现后才提供逃生路径，而是**预判问题并预先铺设逃生路径**。这是一种"逃生口预加载"——在 LLM 还不知道需要之前，就把相关数据准备好。

## 3. 与框架 §4.3 逃生口的对照分析

### 对照表

| 维度 | 框架 §4.3 逃生口 | CCR 逃生口 |
|------|-----------------|-----------|
| **触发条件** | 工具/数据不可用，或前提假设不成立 | 压缩导致信息可能不完整 |
| **响应动作** | 停止执行，向人类报告 | 提供检索工具，允许 LLM 按需取回原始数据 |
| **决策者** | AI 自身判断"不可用"→ 停止 | LLM 判断"需要更多信息"→ 检索 |
| **信息流方向** | 单向：AI → 人类（报告） | 双向：LLM ↔ 存储（检索 + 返回） |
| **恢复机制** | 人类介入修正后重新启动 | 自动恢复——检索成功后继续原任务 |
| **适用场景** | 硬失效（bash 不可用、文件系统损坏、数据缺失） | 软信息损失（压缩删减、摘要化、采样） |
| **哲学** | 阻断执行，防止错误继续 | 提供通道，让判断者自主恢复 |
| **失败模式** | 误停止（假性认为工具不可用） | 未检索（LLM 不调用工具，信息永久丢失） |
| **安全前提** | AI 能正确判断"不可用"状态 | LLM 能正确判断"需要更多信息" |

### 根本差异

框架 §4.3 本质上是**passive block design（被动阻断设计）**——当条件不满足时，系统停止执行，等待人类介入。这个设计由 ETF R5 消融事故（工具失效时用了错数据导致虚假结论）直接驱动，处理的是**数据完整性问题**而非**数据可用性问题**。

CCR 本质上是**active recovery design（主动恢复设计）**——当信息可能不完整时，系统提供一条检索路径，让 LLM 自主判断是否需要更多信息并主动取回。它处理的是**有意为之的信息压缩**，是一种可预期的、结构化的信息不足，而非突发的工具故障。

两者的关键区别在于：**框架 §4.3 的逃生口是在执行路径上设路障（stop），而 CCR 的逃生口是在执行路径上设加油站（retrieve）**。

## 4. 分级逃生口设计模式

> **定位修正（P1-3, 2026-06-13）**: 本章的分级逃生口模式是对 CCR 设计决策的**描述性分析（descriptive）**——它描述了 headroom 如何在压缩场景中处理信息不完整性。将其应用于框架 §4.3 修订属于**独立的设计考量（prescriptive）**，需要独立论证。以下各节的"建议"标签仅指涉前者；后者见 §5（给框架 §4.3 的具体建议），且有独立的条件约束。软信息压缩恢复（CCR 场景）与硬工具故障停止（框架 §4.3 场景）是两类不同的问题，不应共用同一升级链。

从 CCR 的实现中可以提炼出一个通用的三级逃生口模式。这个模式不替代框架 §4.3 的"停止报告"，而是将其作为最严格的一级嵌入一个更精细的分级体系中。

### 4.1 Level 1：检索恢复

**适用场景**: 信息可能不完整，但存在**已知的、结构化的检索或验证路径**。

**核心动作**:
1. 向执行者（LLM）告知信息不完整的事实和程度
2. 向执行者提供检索/验证界面（工具、查询端点、文件路径等）
3. 执行者自行判断是否需要更多信息并主动触发恢复

**CCR 中的体现**:
- 压缩标记（`[N items compressed... hash=abc123]`）告知 LLM 信息被压缩了
- `headroom_retrieve` 工具提供检索界面
- Context Tracker 甚至在 LLM 提问前就主动扩展相关数据

**推广到通用场景**:
- 数据库查询返回了采样结果 → 提供分页/完整导出路径
- API 响应被摘要化 → 提供完整响应的缓存键
- 文件内容被截断 → 提供 `Read` 偏移量和文件路径
- 搜索结果被截断 → 提供完整搜索的查询参数供重跑

**关键约束**: Level 1 的前提是**检索路径已知且可靠**。如果检索后端本身不可用，Level 1 退化为不可用，需要升级到更高层级。

### 4.2 Level 2：降级继续

**适用场景**: 信息质量下降但**并非完全不可用**——信息仍在，但精度、覆盖度或时效性低于理想水平。

**核心动作**:
1. 标注信息的不确定性来源和程度
2. 以降级信息继续执行（不停止）
3. 在输出中明确标注置信度和信息缺口
4. 继续执行但在关键决策点附上"条件依赖"标记

**CCR 中的体现（间接）**:
- BM25 搜索返回部分匹配（而非全部原始数据）时，标签为"基于搜索子集，可能不完整"
- `response_handler` 在检索失败的第三轮停止前，仍然将前两轮的部分结果返回给 LLM，让其基于不完整信息给出有条件判断
- `compression_feedback` 的学习循环——当检索率 > 50% 时降低压缩激进程度（从 Level 2 的"可能不完整"向 Level 1 的"大概率完整"校准）

**推广到通用场景**:
- 实时数据源不可用 → 使用最近一次缓存数据，标注时间戳和 staleness
- API 返回部分字段失败 → 使用可用字段继续，标注缺失维度
- 模型推理超时 → 使用上一次快照或启发式结果，标注"未经完整验证"
- 权限不足无法读取某文件 → 基于目录结构推测内容和用途，标注"推测"

**关键约束**: Level 2 的前提是**部分信息仍可用**且**不确定性能被标注**。不可标注的不确定性（"我不知道我不知道"）不是 Level 2 能处理的。

### 4.3 Level 3：停止报告

**适用场景**: 信息完全不可用，或恢复/降级路径均不存在或已耗尽。

**核心动作**:
1. 立即停止执行
2. 清晰报告不可用的具体资源/工具
3. **绝不用替代数据、假设或记忆中的数据继续**
4. 如果可能，提供恢复建议或等待条件

**当前框架 §4.3 的状态**: 框架的三条逃生口规则全部落在 Level 3。这由 ETF R5 消融事故（工具失效时用了错数据导致虚假结论）直接驱动，是完全合理的硬安全要求。

**关键约束**: Level 3 是最强安全保证，但也是最昂贵的恢复路径——它需要人类介入。过度使用 Level 3 会制造不必要的摩擦（ChatGPT-5.5 对框架的审查中已经指出 HG 闸门过多的问题）。

### 4.4 升级条件：从 L1 到 L2 到 L3 的触发规则

```
Level 1（检索恢复）
   │
   ├── 检索路径本身不可用 → 升级到 Level 2
   │  （如：compression_store 宕机，无法提供原始数据；
   │   但仍可基于压缩摘要继续工作）
   │
   └── 信息缺口过大，即使最大化检索也不足以支撑当前任务
       → 升级到 Level 2 或直接到 Level 3
       （如：缺失信息是验证决策逻辑的核心输入）
       
Level 2（降级继续）
   │
   ├── 不确定性不可标注（"我不知道什么缺失"）
   │  → 升级到 Level 3
   │  （如：数据源的 schema 变了但不知道变在哪；
   │   无法标注"哪些字段的置信度低"）
   │
   └── 降级信息已被证明会导致错误决策
       → 升级到 Level 3
       （如：用缓存数据已产生两次错误判断，
       再继续就是明知故犯）

Level 3（停止报告）
   │
   └── 条件恢复后（工具重新可用/人类介入修正前提）
       → 降级到 Level 1 继续
       （如：修复文件系统权限后，可继续检索恢复执行）
```

**升级原则**：
1. **每次升级都需要明确理由**：为什么当前层级不足以处理？具体缺失什么？
2. **升级后的状态必须显式标记**：从 L1 升级到 L2 时，输出中必须标注"信息不完整，以下基于降级信息，置信度低"
3. **Level 3 的触发必须满足至少一个硬条件**：检索路径不可用（hits L1 约束）、不确定性不可标注（hits L2 约束）、降级信息已导致错误（hits L2 约束）、或框架 §4.3 原始条件（工具/数据完全不可用）

## 5. 给框架 §4.3 的具体建议

### 当前状态

框架 §4.3 的逃生口规则（三条）全部落在 Level 3。这是一个**正确但不完整**的设计——硬失效场景的防护是完备的，但缺少对软信息损失场景的分级响应。

### 建议 1：增加 Level 1（检索/验证路径）

**建议措辞**（在 §4.3 第 6 点"逃生口"中增加一项）：

```
6. 逃生口（超出范围时怎么办）
   - [保留现有三条]
   - 如果信息可能不完整但存在已知检索/验证路径 →
     (a) 向执行者告知不完整的程度和检索路径
     (b) 提供检索界面（工具/查询端点/文件偏移量）
     (c) 执行者自主判断是否需要更多信息
     (d) 如果检索路径本身不可用 → 升级到降级继续（Level 2）
     (e) 检索返回的数据源必须与被压缩数据同源——不同源的数据视为 Level 3 触发条件
     适用实例：搜索结果被截断→提供完整查询参数；
            文件内容被轮次压缩→提供 headroom_retrieve hash；
            数据库采样返回→提供分页/导出端点
```

### 建议 2：增加 Level 2（降级继续）

**建议措辞**（在 §4.3 第 6 点中再增加一项）：

```
   - 如果信息质量下降但非完全不可用 →
     (a) 标注不确定性来源、程度和影响范围
         标注中必须区分"同源降级"（时效/精度/覆盖度下降）和"异源替换"（使用了不同来源的替代数据）。后者直接触发 Level 3。
     (b) 以降级信息继续执行（不停止）
     (c) 在关键决策输出中标记"条件依赖"
     (d) 如果不确定性不可标注或降级信息已导致错误 → 升级到停止报告（Level 3）
     适用实例：实时数据源不可用→用最近缓存标注 staleness；
            API 部分字段失败→用可用字段标注缺失维度；
            模型推理超时→用上一次快照标注"未经完整验证"
```

### 建议 3：当前 Level 3 的措辞微调

**建议措辞**（保持当前措辞，在末尾增加升级关系说明）：

```
   - 如果所需工具/数据不可用（bash 失效、读不到/建不了文件、数据缺失）
     → 立即停止并报告，**绝不用替代/假设/记忆中的数据继续**
     → Level 3 是最终兜底：Level 1 检索路径不可用 且 
       Level 2 降级不安全/不可标注 时，到达此层级
     （ETF R5 消融教训：工具失效时用了错数据 → "两版本一致"虚假结论）
```

### 建议 4：新增升级规则小结

在三条规则后增加一个分级总结框：

```
**分级逃生口升级规则**：
L1(检索恢复) → L2(降级继续) → L3(停止报告)
- 从低层向高层升级需要明确理由
- L3 触发的硬条件：检索路径不可用 | 不确定性不可标注 | 
  降级信息已导致错误 | 工具/数据完全不可用
- 条件恢复后允许高层降回低层继续执行
```

### 5.5 Kill-test: ETF R5 消融事故在引入 L2 后是否仍被阻止？

> **测试场景**: 假设框架 v1.3 已纳入三级逃生口。复现 ETF R5 事故条件——bash 不可用、文件无法创建、回测数据缺失。AI 按新规则判断:
>
> 1. 检索路径是否可用? **否**——bash 失效意味着所有工具调用（包括检索）均不可执行。→ L1 不可用，升级。
> 2. 是否可以降级继续? AI 有上一轮回测的缓存结果。按 L2 规则——信息质量下降但非完全不可用——AI 可能判断为"可用缓存数据标注 staleness 后继续"。
> 3. 这是否正确? **错误**——缓存数据是完全不同的输入（上一版本的回测结果），不是"同源但降级"的信息。使用它就是"用替代数据继续"。
>
> **结论**: L2 在当前措辞下无法阻止 ETF R5 事故——因为它将"不同源替代"误分类为"同源降级"。L2 必须增加硬条件: **替代数据必须与缺失数据来自同一源、同一查询、仅时效或精度降级**。不同源的数据 = Level 3。
>
> **教训**: 此 kill-test 表明，L2 的"降级继续"条件必须从"信息质量下降"收紧为"同源信息的时效/精度/覆盖度降级"。任何涉及数据源替换的场景直接升级到 Level 3。此条件现已纳入 §5 建议措辞。

### 设计考虑

这些建议的目的不是削弱框架 §4.3 的硬安全要求（Level 3 的"绝不用替代数据"仍然是最强防护），而是在安全的前提下增加**可操作的低成本恢复路径**。考虑到框架审查链中 ChatGPT-5.5 的判断（"闸门太多应减少摩擦"）和 DeepSeek-v4-pro 的判断（"覆盖不够应新增检查"）之间存在方向性分歧，此处建议保持审慎：**Level 1 和 Level 2 的引入不应被视为对现有 Level 3 的削弱或替代，而应被视为对同一安全诉求的不同粒度响应**。

## 6. CCR 的关键安全决策

CCR 的设计中包含多项安全相关的显式决策，每条都为逃生口设计提供了具体启示。

### 6.1 哈希碰撞抗性（BLAKE3 → 96 位）

**决策**: 使用 BLAKE3（Rust 侧）或 SHA-256（Python 侧）截断为 24 hex 字符（96 位），而非使用更短的 8 或 16 字符。

**依据**: 96 位在生日边界下约 2^48（约 280 万亿）条目时达到 50% 碰撞概率。对于默认 1000 条目、TTL 300 秒的有界 LRU 缓存，这个碰撞空间远超实际使用范围。

**避免的攻击面**: 较短哈希（如 8 字符 = 32 位）的生日边界约为 2^16（约 65000）条目——在一个高吞吐的生产环境中是可达到的，碰撞概率非零。CCR 选择 96 位使攻击者无法通过选择特定输入来碰撞现有缓存条目的哈希（实际上不可能在 300 秒 TTL 窗口内计算 2^48 个碰撞候选）。

**给框架逃生口的启示**: 检索键的设计应足以抵抗所在使用场景中的意外碰撞。对于进程内工具调用标记系统，96 位是足够的安全余量。对于跨系统的持久化逃生口 ID，可能需要更长的碰撞空间。

### 6.2 TTL 过期（默认 300 秒，可配置）

**决策**: 所有缓存条目在 300 秒后自动过期。可通过 `HEADROOM_CCR_TTL_SECONDS` 环境变量配置。过期条目在读取时惰性清除（而非主动定时扫描，节省后台开销）。

**设计理由**: 5 分钟覆盖了大多数用户对话的"上一轮输出"窗口——用户在几轮内追问相关问题的概率高，超过 5 分钟后相关性急剧下降。同时防止缓存无限增长导致的存储泄漏。

**给框架逃生口的启示**: 逃生口的"恢复窗口"需要有界。一个 30 分钟前压缩的文件列表，在当前对话中已经不太可能被主动检索——与其无限期保留所有逃生通道，不如明确定义"合理追溯期"并在此之后关闭通道。

### 6.3 工作空间隔离（跨项目上下文泄露的主动防护）

**决策**: Context Tracker 的每个 `CompressedContext` 必须携带 `workspace_key`。`analyze_query` 时只匹配与当前请求相同工作空间的压缩条目。空 `workspace_key` 时 fail-closed（返回空列表而非匹配所有）。

**直接原因**: 2026-05-26 一次实际发生的跨项目泄露——共享的进程内 Context Tracker 没有项目来源标识，导致 Project A 的压缩样本在 Project B 的查询中浮出。

**给框架逃生口的启示**: 逃生通道必须携带**来源标识**。多个项目共享一个执行环境时，没有来源隔离的检索接口会产生信息泄漏风险。框架如果将来增加 Level 1 检索恢复机制，工作空间隔离是必须的设计要素。

### 6.4 密钥脱敏（检索事件日志中脱敏 API 密钥/凭证）

**决策**: 在检索事件日志的 `payload_preview` 中，对 API 密钥、Token、密码、Authorization header 等敏感模式执行正则脱敏（替换为 `[REDACTED]`），只保留脱敏后的预览（最大 4096 字符）。

**设计理由**: 检索事件包含原始数据的具体内容。如果原始数据中包含 API 密钥（如用户在 `.env` 文件中存储了密钥，该文件被压缩后在检索时会被完整返回），则日志的 payload 预览会泄漏这些密钥。

**给框架逃生口的启示**: 任何信息恢复/检索路径的审计日志都需要考虑**敏感数据的二次暴露**。逃生口如果让信息重新进入执行上下文（这是它的本职），就需要确保辅助路径（如日志）不会因此泄漏敏感信息。

### 6.5 最大检索轮次限制（默认 3 轮，防无限循环）

**决策**: Response Handler 在检测到 `headroom_retrieve` 调用后最多执行 3 次检索-延续循环。超过 3 轮后，返回最后一轮响应（即使仍包含未处理的 CCR 调用）。

**设计理由**: 理论上 LLM 可以反复检索（检索到数据后→认为不够→再次检索→仍然不够→...），形成无限循环导致 API 费用不受控。3 轮是一个工程折中——绝大多数情况下一轮检索就足够，3 轮覆盖了需要多次过滤的极端情况。

**给框架逃生口的启示**: 逃生通道不是无限的——允许不受限的"继续尝试"本质上是无界的恢复循环。框架的 Level 1 检索恢复也应该有轮次上限。如果 3 次检索都不足以让 LLM 获得足够信息完成任务，那么信息缺口大概率不是能通过更多检索解决的——此时应该升级到 Level 2 或 Level 3。

### 6.6 这些决策给框架逃生口设计提供的启示

综合以上 5 个决策，可以提炼出一条设计原则：

**逃生口不是免费的**——它需要碰撞抗性（防止错误的检索结果）、时效边界（防止过时信息残留）、来源隔离（防止跨项目泄露）、日志脱敏（防止审计泄漏）和资源上限（防止无限循环）。一个设计良好的逃生口在提供恢复能力的同时，绝不放松对上述五个维度的要求。

## 7. 局限性：CCR 不能替代现有逃生口

### 硬失效场景 CCR 无用

CCR 的所有能力都建立在一个前提上：**压缩发生了，原始数据在存储中，LLM 可以调用工具**。以下场景中 CCR 完全无效：

- **bash 不可用**：LLM 的工具调用完全无法执行，`headroom_retrieve` 也不例外
- **文件系统不可读**：压缩操作本身无法成功——如果输入数据都读不到，Compression Store 的 store() 就没有输入
- **数据源本身缺失**：如果查询 API 返回 500 或数据文件被删除，压缩操作没有输入，检索无从谈起

在这些场景中，框架 §4.3 的 Level 3（停止报告）是唯一正确的响应。CCR 做的不是替代它，而是在这些硬失效之上增加一层**软信息恢复能力**。

### 资源假设不总是成立

CCR 假定：
- 有存储空间来缓存原始数据（每个压缩条目可达数 MB）
- 有检索带宽来执行 BM25 搜索或完整数据返回
- 压缩操作发生的频率足以触发 TTL 窗口内的追踪

在**内存受限环境**（如边缘设备、容器化 CI runner）或**极端高频压缩场景**（如秒级批量处理）中，这些假设可能不成立。此时 CCR 的核心循环会更快触发 eviction 和 expiration，降低检索命中率，实际上退化回"不可逆压缩"。

### LLM 可能不调用检索工具

CCR 最本质的局限是：**检索是 LLM 的选择，不是系统的强制**。如果 LLM 基于压缩后的摘要做出了错误的判断但不自知（"我不知道我缺少什么"），它就不会调用 `headroom_retrieve`。这是一种与 Level 3 失败模式对称但相反的风险：

- Level 3 的失败模式：误停止（假性认为工具不可用）
- CCR 的失败模式：未检索（假性认为信息充足）

Context Tracker 的主动扩展机制在一定程度上缓解了这个问题（在 LLM 提问前就预加载相关数据），但它只能匹配**字面相关性**——如果 LLM 真正需要的信息和当前查询的关键词没有表面匹配，扩展也不会触发。

### CCR 是补充，不是替代

表格式总结：

| 场景 | 框架 §4.3（停止报告） | CCR（检索恢复） | 推荐 |
|------|----------------------|----------------|------|
| bash 完全不可用 | 正确响应 | 无效 | §4.3 |
| 文件系统不可读 | 正确响应 | 无效 | §4.3 |
| 数据源返回空/错误 | 正确响应 | 无效（无输入可压缩） | §4.3 |
| 工具输出被压缩截断 | 过度反应（可恢复） | 正确响应 | CCR + Level 1 |
| 搜索结果被采样 | 过度反应 | 正确响应（完整搜索可通过 hash 取回） | CCR + Level 1 |
| 日志被摘要化 | 过度反应 | 正确响应（LLM 可按需检索完整日志） | CCR + Level 1 |
| 缓存数据异源替换（如 ETF R5: 用旧版回测结果替代当前缺失数据） | 正确响应（停止） | 危险（可能误判为 Level 2 降级继续） | §4.3 Level 3 |
