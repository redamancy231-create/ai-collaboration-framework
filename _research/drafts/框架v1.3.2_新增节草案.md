## §3.7.0 信号熵检测层（新增）

### 定位
位于§3.7（漂移检测层）之前，作为漂移检测的**前置信号过滤器**。漂移检测关注"AI行为与预期的偏差"，信号熵检测关注"信号本身的健康度"——信号太密集（过度响应）或太稀疏（响应枯竭）都预示着AI协作质量下降。

### 三条核心规则

#### 规则1：历史频率抑制（Frequency Suppression）
- **触发条件**：同一信号在最近8次进化事件中出现 ≥3 次
- **动作**：自动标记为 `over_processed`，抑制后续3个周期内的重复注入
- **原理**：Evolver论文发现，修复循环（repair loop）——不断对同一问题触发修复——是AI自进化系统最常见的失效模式。频繁出现的信号往往不是"需要更多关注"，而是"已被充分处理但反馈回路卡住"

#### 规则2：修复循环检测（Repair Loop Detection）
- **触发条件**：连续 ≥3 次进化事件的 `intent = repair`
- **动作**：自动注入一个 `intent = explore` 信号，强制切换策略方向
- **冷启动豁免**：系统初始化后前5个周期内默认阈值翻倍（≥6次连续repair才触发）
- **原理**：纯修复模式会收敛到局部最优。周期性强制探索防止系统陷入"修好A→B崩→修B→A又崩"的死循环

#### 规则3：空周期检测（Empty Cycle Detection）
- **触发条件**：连续 ≥5 次进化事件的 `blast_radius = 0`（无实际变更）
- **动作**：触发 `steady_state` 告警，建议人工审查是否方法论需要更新
- **原理**：长期无变更可能是"系统已收敛"（正面），也可能是"检测机制失效"（负面）。规则3不自动干预，仅告警——因为区分这两种情况需要人工判断

### 信号熵指标
将上述三条规则的检测结果聚合为单一 `signal_entropy` 指标（0.0-1.0）：
- `signal_entropy < 0.2`：信号健康（有信号但不过密，偶尔repair但不过度，偶尔空周期但不久）
- `0.2 ≤ signal_entropy < 0.5`：轻度不健康——记录到监控日志
- `0.5 ≤ signal_entropy < 0.8`：中度不健康——触发§3.7.2（告警聚合）的低优先级告警
- `signal_entropy ≥ 0.8`：重度不健康——触发§3.7.2的高优先级告警，并建议暂停自主进化切换到人工审查模式

### 实现
- 一个Python脚本 `signal_entropy.py`，读取 `events.jsonl`（或等效事件日志），输出 `signal_entropy` 值 + 触发规则列表
- 每个进化周期结束时运行，结果写入监控日志
- 冷启动期间（< 10条事件）使用宽松阈值，避免数据不足导致的误报

### 与现有框架的接口
- **输入**：events历史记录（来自进化周期的每次迭代输出）
- **输出到**：§3.7.2（告警聚合）——信号熵超标作为新的告警源
- **输出到**：§3.7.3（规则退役/表观遗传权重）——规则1的信号抑制标记影响规则的活跃/休眠状态

## §9.6 紧凑表示原则（新增设计原则）

### 定位
§9（安全边界）的最后一条设计原则。约束框架所有产出的提示词（prompt）、经验编码（experience encoding）、历史摘要（history summary）的格式和长度。

### 原则文本

**"经验的价值不在于提供更多信息，而在于以最小token消耗传递最大控制信号。"**

### 三条子规则

#### 子规则1：单一经验/信号编码上限
- 每条经验（如一个过去的实验结果、一个已知的失败模式）的注入长度 **≤ 500字符**（可配置参数 `MAX_EXPERIENCE_CHARS`）
- 如原始内容超限，必须经过压缩处理（见子规则3），不得直接截断（截断=丢弃信息，压缩=提炼信息）

#### 子规则2：结构化编码优先
- 经验/信号编码默认使用结构化JSON格式，而非自然语言叙述
- JSON schema必须包含：信号类型（`signal`）、置信度（`confidence`，0-1）、上下文（`context`）、来源（`source`）
- 示例：
  ```json
  {"signal":"factor_ic_decay","confidence":0.85,"context":"csi500_neutral","source":"reo_20260614_001"}
  ```
  而非：
  ```
  在CSI 500中性化测试中发现因子的IC有衰减趋势，置信度较高...
  ```
- **原理**：Evolver论文的4,590次对照试验表明，紧凑结构化表示（230 tokens）在Agent控制效果上优于冗长文档表示（2,500 tokens）。虽然此结论在量化研究场景尚未独立验证，但作为设计原则，结构化编码至少不会比自然语言更差，且有可解析性优势

#### 子规则3：历史压缩——滚动窗口 + 统计摘要
- 仅保留最近 **N 次事件**的完整记录（N可配置，默认20）
- 超过窗口的历史事件以统计摘要替代：
  - 信号频率分布（各类信号的出现次数）
  - 成功率趋势（最近N次的pass rate滑动平均）
  - 最近一次同类成功的完整记录（作为正样本保留）
- **原理**：Evolver论文发现"紧凑失败警告优于朴素追加"——把全部失败历史都塞进prompt反而降低性能。选择性保留比全量追加更有效

### 验证方法
在新方法论被正式接纳为框架设计原则前，应在目标场景（量化研究）做小规模A/B测试：
- **Arm A**：紧凑JSON格式编码的经验注入
- **Arm B**：等量信息的自然语言格式编码注入
- **任务**：相同的回测指令（如"设计一个CSI500中性化因子"）
- **评估指标**：AI产出因子的IC均值、IC_IR、以及生成过程的token消耗

### 反模式警示
- **不要**在未经验证的情况下直接假设"文档越详细越好"（这是Evolver论文的核心教训）
- **不要**将本原则等同于"所有prompt都要尽可能短"——关键信息不能省略，但表达方式应结构化
- **注意**：Evolver的发现限于科学代码求解场景（Gemini 3.1系列模型），对量化研究场景和不同模型的泛化性待验证

## §7 研究经验对象（Research Experience Objects, REO）

### 定位
新增独立节。在§6（工具与自动化）和§8（通信与协作）之间。REO为框架提供结构化的经验存储和检索机制——替代目前非结构化的"每次对话后写memory summary"模式。

### REO Schema

```json
{
  "reo_id": "reo_YYYYMMDD_NNN",
  "type": "gene | capsule | event",
  "hypothesis": "对研究假设的简洁陈述",
  "test": {
    "method": "测试方法",
    "parameters": {}
  },
  "result": {
    "outcome": {},
    "validation_passed": true
  },
  "validation_rule": "通过/不通过的判定条件",
  "epigenetic_mark": {
    "success": 0,
    "total": 0,
    "boost": 0.0
  },
  "signals": [],
  "parent_reo": null,
  "created_at": "ISO 8601",
  "model": "生成此REO的模型"
}
```

### 三种REO类型

#### Gene型（可复用的验证策略）
- 代表一个经过验证的、可跨项目复用的方法论规则
- 示例：*"A股因子IC均值>0.02且IC_IR>0.3才纳入正式组合，否则仅记录为探索性发现"*
- `validation_rule` 是可执行的条件判断（如 `ic_mean > 0.02 AND ic_ir > 0.3`）
- 存储在 `reo/genes/` 目录下

#### Capsule型（成功历史快照）
- 代表一个具体的、已完成的研究步骤的快照
- 包含原始数据路径、关键参数、最终结果
- 示例：*"2024-2026年CSI500成分股中，动量因子在市值中性化后IC从0.015提升到0.035"*
- 存储在 `reo/capsules/` 目录下

#### Event型（审计日志）
- 代表每次实验的输入/输出/决策的完整记录
- append-only，不可修改
- 存储在 `reo/events.jsonl`（JSONL格式，每行一个Event）

### Gene/Capsule/Event的闭环

```
Event（记录每次实验）
  ↓ 多次成功
Capsule（固化成功经验）
  ↓ 跨项目复用
Gene（提炼为可复用策略）
  ↓ 在新项目中触发
Event（新实验记录，链条延续）
```

闭环确保：(1) 方法论不是一次性文档而是活的经验对象；(2) 每次新实验自动继承相关历史经验；(3) 失败历史也被保留（作为anti-patterns），防止重复踩坑

### 与kill-test-first的协作
- kill-test-first的预注册模板**扩展**以包含 `intent_distribution`（意图分配：explore/validate/repair的比例）
- 预注册的 `kill_condition`（否决条件）直接映射到REO的 `validation_rule` 字段
- 实验完成后，预注册 + 结果 = 一个Event型REO

### 实现优先级
- **Phase 1**（最小可用）：定义JSON schema + 手动创建/读取REO
- **Phase 2**（半自动）：Python脚本自动从实验输出生成Event型REO
- **Phase 3**（全自动）：Gene自动推荐（基于epigenetic权重）+ 冲突检测（新REO是否与已有Gene矛盾）

### 存储结构
```
reo/
├── genes/
│   └── gene_*.json       # 每个Gene一个文件
├── capsules/
│   └── capsule_*.json     # 每个Capsule一个文件
├── events.jsonl           # 所有Event append-only
└── README.md              # REO使用说明
```

## §3.7.3 修改：规则退役 → 表观遗传权重

### 原内容（v1.3.1）
§3.7.3定义的是静态规则退役机制：规则超过N个周期未触发→退役。

### 修改（v1.3.2）
将静态退役替换为**基于累积性能的动态权重调整**——表观遗传权重（Epigenetic Weighting）。

### 机制
1. **每个方法论资产（因子/规则/策略/REO）维护boost分数**，初始值0.0
2. **成功时**：boost += `delta_success`（默认 +0.05）
3. **失败时**：boost -= `delta_failure`（默认 -0.15，非对称惩罚——失败的信息量大于成功）
4. **硬抑制**：boost ≤ -0.3 → 自动排除（标记为 `suppressed`）
5. **惯性抑制**：连续8次触发但无实际影响（`stable_no_error`）→ 自动排除（标记为 `inert`）
6. **Context-aware维度**：同一方法论在不同市场环境/任务类型下独立维护boost分数——A股动量因子在中证500环境下boost=+0.2，在沪深300环境下boost=-0.1，两个环境独立评估

### 与信号熵的关系
- 信号熵检测层（§3.7.0）的**规则1（频率抑制）产生的标记**直接影响boost分数的更新频率——被频率抑制的信号在抑制期内不参与boost更新
- 信号熵检测层的**规则2（修复循环检测）强制注入的exploration信号**为被抑制的资产提供"复活"机会——在被排除前至少被exploration重新评估一次

### 优势（vs 静态退役）
- 不会因为"长期未触发"退役一个在市场切换后可能又有效的策略
- 失败惩罚 > 成功奖励的非对称设计更符合风险管理逻辑
- Context-aware维度避免"在一个市场上表现差→在所有市场上被退役"的过度泛化
