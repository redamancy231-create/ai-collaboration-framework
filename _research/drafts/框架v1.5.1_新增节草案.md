## §3.7.0 事件流健康度监测（新增，置于 §3.7.1 之前）

### 定位
作为 §3.7 漂移检测层的**前置输入源**，不改变 §3.7 的现有架构。§3.7.0 只观测事件流本身的健康度（信号密度、修复频率、变更活性），输出到 §3.7.3（信号聚合与告警规则）作为若干普通监测项。**不新增独立告警等级，不自动改变 Loop/Workflow 行为。**

与 §3.7.1 的核心约束一致：**只观测，不阻断。** 所有输出均为建议性质，操作者通过现有 HG（Human Gate）机制决定是否干预。

### 三条监测规则

#### 规则1：重复信号频率监测（Frequency Monitor）
- **观测**：同一信号类型在最近 8 个迭代周期的事件日志中出现 ≥3 次
- **输出**：在监测报告中标记 `over_processed`，附带频率计数和最近出现时间
- **建议动作**：操作者考虑在未来 3 个周期内手动抑制该信号的重复注入
- **不自动执行**：不自动抑制信号注入，不修改事件日志
- **冷启动**：事件总数 < 10 时，阈值自动翻倍（≥6 次/8 周期才标记），避免数据稀疏误报
- **原理**：Evolver 项目的经验提示——频繁出现的信号往往不是"需要更多关注"，而是"已被充分处理但反馈回路卡住"。但此观察限于 Evolver 的具体场景（科学代码求解、Gemini 3.1；证据等级：[Sp]），在量化研究场景的有效性待验证

#### 规则2：修复循环监测（Repair Loop Monitor）
- **观测**：连续 ≥3 个迭代周期的 `intent = repair`
- **输出**：触发 `repair_loop_alert`，作为 HG-1 建议提交给操作者（建议注入探索性任务以打破循环）
- **不自动执行**：不自动注入 explore 信号，不改变当前迭代的 intent
- **冷启动**：前 5 个周期默认阈值翻倍（≥6 次连续 repair 才告警）
- **原理**：纯修复模式可能收敛到局部最优。周期性强制探索可防止"修好 A→B 崩→修 B→A 又崩"的死循环。此观察限于 Evolver 场景（[Sp]）

#### 规则3：空周期监测（Empty Cycle Monitor）
- **观测**：连续 ≥5 个迭代周期的 `blast_radius = 0`（无实际变更）
- **输出**：触发 `steady_state_alert`，仅记录告警（不提交 HG，因为需要人工判断"系统已收敛" vs "检测机制失效"）
- **不自动执行**：不自动干预
- **原理**：长期无变更可能是正面（收敛）或负面（失效），需要人工判断

### 事件流健康度指标

将三条规则的输出聚合为单一 `event_stream_health` 指标（0.0-1.0，越低越健康）。

**计算公式**：

```
health = clamp(0, 1, w1×freq_score + w2×repair_score + w3×empty_score)
```

其中：
- `freq_score` = 最近 8 个周期中，被规则1标记 `over_processed` 的信号类型数 / 8
- `repair_score` = min(1.0, 连续 repair 周期数 / 6)（连续 ≥6 次 repair → 1.0）
- `empty_score` = min(1.0, 连续空周期数 / 10)（连续 ≥10 次空周期 → 1.0）
- `w1 = 0.3`, `w2 = 0.5`, `w3 = 0.2` [可配置]（修复循环权重最高——修复死循环的危害 > 信号重复 > 空周期）
- `clamp(0, 1, x)` = max(0, min(1, x))

**阈值映射**：
- `health ∈ [0.0, 0.2)`：健康
- `health ∈ [0.2, 0.5)`：轻度关注 → 记录到监测日志
- `health ∈ [0.5, 0.8)`：中度关注 → 作为普通监测项输入 §3.7.3
- `health ∈ [0.8, 1.0]`：重度关注 → 作为高权重监测项输入 §3.7.3，附 HG-1 建议

注意：`event_stream_health` 不产生独立告警。它作为 §3.7.3 的输入源之一，与现有的绩效信号、发现率衰减等监测项一同参与绿/黄/橙/红四级聚合。**不新增告警颜色或等级。**

### 与现有 §3.7 的接口
- **输入**：迭代事件日志（来自 L2 Loop 的每次迭代产出，或等效的事件记录）
- **输出到 §3.7.3**：`event_stream_health` + 三条规则各自的触发状态（作为普通监测项）
- **输出到 §3.7.4**（规则退役自动化）：规则1 的 `over_processed` 标记作为规则退役的参考信号（被频繁标记的规则优先进入退役评估）
- **不输出到 §3.7.5**（宪法审计）：事件流健康度是操作层面指标，非宪法层面

### 实现
- 一个 Python 脚本 `event_health.py`，读取事件日志，输出 health 值 + 规则触发列表
- 每个迭代周期结束时运行，结果追加到监测日志
- 冷启动期间（< 10 条事件）所有阈值自动翻倍，只记录不告警
- **禁用开关**：`EVENT_HEALTH_ENABLED=0` 可完全禁用本监测层
- **只观测模式**（默认）：`EVENT_HEALTH_DRY_RUN=1`（默认开启），确保任何自动建议在手动确认前不生效
- 实施时间：2-3 小时（脚本雏形，不含事件 schema 定义和字段填充责任分配——这些需要项目级约定）

### 实施前验证
- [ ] 确认事件日志格式已稳定、字段语义已明确
- [ ] dry-run ≥ 3 个完整迭代周期，记录误报率
- [ ] 人工审查首次触发的每条规则，确认非误报
- [ ] 确认与现有 §3.7.3 的告警聚合不产生重复告警
- [ ] 数值敏感性分析：健康度阈值 ±50% 对告警频率的影响

## §3.7.4.1 自适应权重淘汰（修改 §3.7.4 的替代方案）

### 定位
对 §3.7.4（规则退役自动化）的补充替代机制。原 §3.7.4 使用"超时未触发→退役"的静态逻辑。本节提供基于累积性能的动态权重调整作为替代或并行选项。

### 机制

**每个被追踪的方法论资产（规则/因子/策略/REO）维护一个 `weight` 分数，初始值 0.0。**

| 事件 | 权重变化 | 默认值 | 说明 |
|------|---------|--------|------|
| 资产关联的任务成功 | `weight += delta_success` | `+0.05` [可配置] | 小幅正向调整 |
| 资产关联的任务失败 | `weight += delta_failure` | `-0.15` [可配置] | 非对称惩罚——失败信息量 > 成功 |
| 资产触发但无实际影响 | `weight += delta_no_impact` | `0.0` [可配置] | 不改变权重，但计入惯性计数 |

**淘汰条件**（两条独立路径，任一触发即淘汰）：
1. **硬抑制候选**：`weight ≤ -0.3` [可配置] → 标记 `suppression_candidate`，在下一 HG-1 中提交操作者确认后才移出活跃资产池。确认前资产在注入层仍正常使用，但在监测报告中标注为候选淘汰。
2. **惯性抑制候选**：连续 8 次 [可配置] 触发但 `delta_no_impact` → 标记 `inert_candidate`，在下一 HG-1 中提交操作者确认后才移出活跃资产池。确认前资产在注入层仍正常使用。

**复活机制**：被抑制的资产在以下情况可复活：
- 操作者手动解除抑制
- §3.7.0 的规则2（修复循环监测）建议注入探索性任务时，被抑制资产有机会被重新评估（操作者决定是否给予复活机会）

**Context-aware 维度**：同一资产在不同项目类型/市场环境下**独立维护权重**。例如：一个量化因子在中证 500 环境下 weight=+0.2，在沪深 300 环境下 weight=-0.1，两者独立评估、独立淘汰。

### 数值依据与敏感性

**重要声明**：`delta_success=+0.05`、`delta_failure=-0.15`、硬抑制阈值 `-0.3`、惯性阈值 `8` 均来自 Evolver 项目的设计选择。该项目的核心算法代码（`epigenetics.js`）被 javascript-obfuscator 混淆，数值的可验证性限于"从上游分析报告的转述"，非公开源码事实。且 Evolver 的 epigenetic suppression 在科学代码求解场景（Gemini 3.1）上设计，迁移到量化研究场景的有效性未经任何经验检验。

因此：
- 所有数值标注为 **[可配置]**，非硬编码
- 非对称惩罚比（0.15/0.05 = 3:1）的设计原理来自 Evolver："失败的信息量大于成功"，但这一直觉在金融策略中可能过度惩罚高方差但正期望的探索
- 调参建议：风险偏好较低 → 增大非对称比（如 5:1）；探索性研究 → 减小（如 2:1）
- 首次试跑时需做敏感性分析：主要阈值 ±50% 对淘汰决策的影响

### 与 §3.7.0 的接口
- §3.7.0 规则1 的 `over_processed` 标记：被标记的资产在抑制期内的成功率不计入 weight 更新（避免"已被充分处理的信号"继续拉低/拉高权重）
- §3.7.0 规则2 的 `repair_loop_alert`：如果修复循环涉及的资产即将达到硬抑制阈值，告警中附带"可能被淘汰"的预警

### 与静态退役（§3.7.4）的关系
- 本机制是 §3.7.4 的**补充替代**，非强制替换
- 操作者可以选择：仅用静态退役、仅用自适应权重、或两者并行（静态退役作为安全网，自适应权重作为主要机制）
- 默认建议：新项目先用静态退役，积累 ≥20 条事件后考虑启用自适应权重

## §9.7 经验注入上下文预算规则（新增）

### 定位
跨层关切——约束所有向 AI 上下文注入的经验摘要的格式和长度。只约束**注入层**（拼入 prompt 的 compact view），不约束**存储层**（完整 REO 存档、事件日志、审查报告）。

### 规则文本

**"向 AI 上下文注入的经验摘要应控制长度、使用结构化格式、避免冗余追加。"**

### 三条子规则

#### 子规则1：注入长度上限
- 单条经验摘要的注入长度 ≤ **500 字符** [可配置：`MAX_EXPERIENCE_INJECTION_CHARS`]
- 如原始经验记录的完整内容超限，必须生成 compact_view（≤500 字符），不得直接截断（截断=丢弃信息，compact_view=提炼信息）
- 存储层的完整记录**不**受此限制

#### 子规则2：结构化编码优先
- 经验摘要默认使用结构化 JSON 格式，而非自然语言叙述
- JSON 需包含：信号类型（`signal`）、置信度（`confidence`，0-1）、上下文（`context`）、来源（`source`）
- 示例（注入层 compact_view）：
  ```json
  {"signal":"factor_ic_decay","confidence":0.85,"context":"csi500_neutral_2024","source":"reo_20260614_001"}
  ```
  替代等效自然语言：
  > 在 CSI 500 中性化测试中（2024年数据），因子的 IC 有衰减趋势，置信度较高。来源：reo_20260614_001。

#### 子规则3：历史压缩——滚动窗口 + 统计摘要
- 注入上下文的经验历史仅保留最近 **N 次事件**的 compact_view（N 可配置，默认 20）
- 超过窗口的历史以统计摘要注入：
  - 各类信号的频率分布
  - 成功率滑动平均（最近 N 次的 pass rate 趋势）
  - 最近一次同类成功的完整 compact_view（作为正样本保留）
- 存储层始终保留全量历史

### 来源证据等级

本原则的思想源于 Evolver 项目（arXiv:2604.15097，beta technical report）的方法论提取。该论文报告的 retained analyses（非独立随机对照试验）暗示——在科学代码求解场景（Gemini 3.1 系列模型）下——紧凑结构化表示（~230 tokens）在 Agent 控制效果上可能优于冗长文档表示（~2,500 tokens）。

**证据限定**（以下限定均经 Qwen qwen3.7-max 和 Codex ChatGPT-5.5 两轮独立审查确认）：
1. 4,590 是 retained analyses 的累计数，非 4,590 个独立随机对照试验
2. 论文不含置信区间、p 值、效应量或多重比较校正
3. 仅测试了 Gemini 3.1 系列（Pro/Flash），未跨模型家族验证
4. 任务限于科学代码求解，未在量化研究或非代码场景测试
5. 公开仓库缺实验配置和原始数据（Evolver 项目综合评分 4.1-4.2/10）
6. Evolver 核心算法代码被混淆，不可独立验证

**按本框架 §9.6 的证据分类标准，本原则的证据等级为 `[Sp]`（推测，Speculation）**——有方向性暗示但缺乏量化研究场景的严格验证。升级到 `[J]`（判断，Judgment）需完成子规则4 的 A/B 测试并通过。

### 子规则4：本场景 A/B 验证（升级到 [J] 的前提）

在将本原则从 [Sp] 升级为正式方法论前，需在量化研究目标场景完成 A/B 测试：

| 参数 | 规格 |
|------|------|
| 任务 | ≥30 个独立因子设计任务（如"设计一个 CSI 500 动量因子"） |
| Arm A | 紧凑 JSON 格式的 compact_view 注入 |
| Arm B | 等量信息的自然语言格式注入 |
| 重复 | 每个任务 ≥3 次（控制提示词随机性） |
| 模型 | 固定模型版本和 checkpoint |
| 随机种子 | 固定并记录 |
| Prompt 控制 | 除编码格式外，Arm A/B 的信息量等同（同义改写，非截断） |
| 评分指标 | IC 均值 + IC_IR + 夏普比率（主要）+ token 消耗（次要） |
| 最小有意义效应 | IC 均值差异 ≥0.005 **且** token 节省 ≥20% |
| 失败判据 | Arm A（JSON）在 IC 均值上显著劣于 Arm B（自然语言）→ 原则降级为 [F]（证伪） |
| 部分通过 | Arm A 与 Arm B 在 IC 上差异 < 最小效应 **但** token 节省 ≥20% → 原则作为"成本优化建议"有条件采纳 |

**统计方案**（执行 A/B 测试前预注册）：
- **检验方法**：配对 Wilcoxon signed-rank test（同一任务的两 arm 结果配对），α = 0.05
- **置信区间**：Hodges-Lehmann 估计量 + bootstrap 95% CI（10,000 次重采样）
- **样本外验证**：30 个任务中随机留出 10 个（33%）作为 hold-out，仅用于最终验证
- **量化风控指标**（追加到评分指标中）：换手率、样本外 Sharpe、最大回撤、因子相关性矩阵（与已有因子的平均绝对相关性）——防止紧凑编码产生过度拟合的噪音因子
- **任务集抽样**：从不同指数（CSI 500/沪深 300/中证 1000/创业板指）× 不同因子类型（动量/波动率/流动性/价值/质量）× 不同中性化方式（市值/行业/Beta/无）的笛卡尔积中分层抽样，确保覆盖多样性
- **防数据泄漏**：每个任务的回测数据切分为训练期（60%）和验证期（40%），IC 指标仅在验证期计算
- **时间估算修正**：180 次运行在有全自动评估流水线的条件下约需 6-8 小时（非草案初估的 4-6 小时），首次搭建流水线额外需 2-3 小时

如果用户不执行 A/B 测试，本原则仍可作为**上下文预算纪律**使用（标注 [Sp]），但不应作为已验证方法论呈现。

### 不应被误解为
- ❌ 所有 prompt 都要尽可能短（关键信息不可省略）
- ❌ JSON 一定优于自然语言（结构化可能丢失语义、压缩反例、让模型机械服从错误字段——这些风险在 [Sp] 阶段未被排除）
- ❌ 500 字符是硬性截断（不是——超限须生成 compact_view，不是截断）
- ❌ 本原则约束所有文档产出（仅约束注入层——存储层的完整记录、解释文档、审查报告不受限制）

## §9.8 研究经验对象（Research Experience Objects, REO）

### 定位
跨层关切——结构化的经验存储和检索机制。REO 替代目前非结构化的"每次对话后写 memory summary"模式，为 L2-L5 各层提供统一的经验持久化格式。

REO 分为**存储层**（完整 JSON 存档）和**注入层**（≤500 字符 compact_view，受 §9.7 约束）。两层由不同的字段集支撑，通过 `compact_view` 字段连接。

### REO 三类对象

| 类型 | 用途 | 存储路径 |
|------|------|---------|
| 验证规则（Validation Rule） | 可跨项目复用的方法论规则 | `reo/rules/rule_*.json` |
| 成功快照（Success Snapshot） | 具体研究步骤的快照 | `reo/snapshots/snap_*.json` |
| 审计记录（Audit Record） | 每次实验的完整记录 | `reo/events.jsonl`（append-only） |

> **术语来源说明**（仅此一处，非运行时字段）：本框架的数据结构设计受 Evolver 项目（Gene/Capsule/Event）启发，但使用金融/工程术语替代生物学隐喻以降低认知负担。旧术语不出现在任何 schema 字段、检查清单或运行时逻辑中。

### 正式 JSON Schema

#### 验证规则（Validation Rule）

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "reo_validation_rule",
  "type": "object",
```

### 晋升规则（存储层闭环）

REO 三类对象之间的晋升需满足明确的量化条件。

#### 晋升路径

| 晋升路径 | 触发条件 | 操作 |
|---------|---------|------|
| 审计记录 → 成功快照 | 同一 hypothesis 的审计记录 ≥3 条且 outcome=success | 系统提示操作者晋升；操作者手动确认 |
| 成功快照 → 验证规则 | 同一成功快照被 ≥2 个不同项目通过 `parent_audit_ids` 引用，且 performance_track.weight ≥ 0.3 | 系统提示操作者晋升；操作者手动确认 |
| 任何对象 → anti-pattern | 审计记录 outcome=failure，且同一 hypothesis 的失败次数 ≥ 成功次数 | 记录到验证规则的 anti_patterns 数组（不自动删除原对象） |
| 验证规则冲突 | 新验证规则的 validation_rule 与已有规则的 validation_rule 逻辑矛盾 | 触发人工裁决告警（HG-2 级别） |

#### 关键机制定义

**Hypothesis Identity Resolution（假设身份匹配）**：
- 两条审计记录/成功快照属于"同一 hypothesis"，当且仅当：`hypothesis` 字段的归一化文本的编辑距离 ≤ 3 且关键实体（因子名/指数名/方法名）完全匹配
- 归一化：转小写、去标点、移除版本号后缀（如 "v2"、"修订版"）
- 匹配阈值 [可配置]：`HYPOTHESIS_MATCH_MAX_EDIT_DISTANCE = 3`

**跨项目引用计数**：
- 成功快照的 `parent_audit_ids` 字段记录了支撑它的审计记录
- 验证规则通过"被哪些成功快照引用"来追踪跨项目复用次数
- 引用计数器在每次晋升评估时自动扫描 [可配置]：`REFERENCE_SCAN_PATHS = ["reo/snapshots/", "reo/events.jsonl"]`
- 引用必须来自不同项目的 REO 目录（以不同的 `source_artifact` 根路径或显式的 `project_id` 区分）

**冲突检测口径**：
- 新验证规则的 `validation_rule` 与已有规则的 `validation_rule` 进行配对比较
- 检测方法 [可配置]：LLM-based（默认），将两条 validation_rule 同时提交给模型，问"这两条规则在逻辑上是否矛盾？如是，矛盾类型是什么（直接对立/范围冲突/时序矛盾）？"
- 人工裁决记录：冲突裁决的结果（保留新规则/保留旧规则/两者共存加注/两者均废弃）写入审计记录，作为冲突解决的历史先例

**所有晋升需操作者手动确认**——不自动晋升。冲突裁决必须人工处理。每次晋升/否决/冲突裁决均生成一条 `event_type = "decision"` 的审计记录，形成完整的晋升审计链。

### 与 kill-test-first 的协作

- kill-test-first 的预注册**是实验前的不可变承诺**
- 审计记录通过 `pre_registration_id` 引用预注册，而非复制预注册中的判定条件
- 预注册中的 `kill_condition` 作为审计记录 outcome 判断的**唯一参考源**（source of truth）
- 审计记录不应有自己的 `validation_rule` 副本——防止预注册和审计记录各自维护判定条件导致漂移
- kill-test-first 预注册模板扩展 `intent_distribution` 字段（explore/validate/repair 的预期比例）——此字段为声明性，不自动强制执行

### 与 §9.7（上下文预算规则）的接口

- 存储层：REO 完整 JSON，无字符限制
- 注入层：每个 REO 的 `compact_view` 字段（≤500 字符），受 §9.7 约束
- `compact_view` 由操作者在创建/更新 REO 时手动填写，或由 AI 从完整 REO 自动生成后人工确认
- 向 AI 上下文注入经验时，只注入 `compact_view`，不注入完整 REO
