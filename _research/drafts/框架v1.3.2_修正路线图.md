## 二、逐项修正规格

### P0：版本对齐 + 重编号

**问题**：草案声称修改"§3.7.3（规则退役）"，但主文档 v1.5 的 §3.7.3 是"信号聚合与告警规则"，规则退役在 §3.7.4。草案还新增"§9.6"，但 v1.5 已有 §9.6（证据分类与声称校准）。

**修正**：
1. 读取当前 `AI协作项目全生命周期框架.md` v1.5 的完整章节目录
2. 重编号所有新增/修改节，使其在 v1.5 的编号空间内唯一：
   - 信号熵检测 → `§3.7.0`（仍可用，v1.5 的 §3.7 从 §3.7.1 开始）
   - 表观遗传权重 → `§3.7.5.1` 或替代现有退役逻辑的 `§3.7.4.1`
   - REO → `§7`（需确认 v1.5 的 §7 是否空闲）
   - 紧凑表示原则 → `§9.7`（v1.5 的 §9.6 已被占用）
3. 明确标注"本草案基于框架 v1.5 结构编写"

### P1：边界修正——自动干预→只观测

**问题**：草案让 §3.7.0 的规则1（自动抑制）和规则2（自动注入 explore）直接改变执行策略，违反 §3.7.1 的"只观测，不阻断"核心约束。

**修正**：
1. 规则1"自动抑制"→ 改为"在监测报告中标记 `over_processed`，建议操作者手动抑制"
2. 规则2"自动注入 explore"→ 改为"触发 `repair_loop_alert`，作为 HG-1 建议提交给操作者"
3. 规则3"空周期检测"保持告警-only（本来就没自动干预，但需明确写"不自动干预"）
4. 新增的 `signal_entropy` 指标不新增独立告警等级，而是作为现有绿/黄/橙/红聚合规则的一个输入源
5. 所有自动动作保留为"操作者可启用的可选开关"，默认 OFF

### P2：去隐喻化 + 去背书

**问题**：草案声称"已去隐喻化"但保留了 Gene/Capsule/Event/epigenetic_mark/表观遗传权重等术语。草案反复引用"DeepSeek + Qwen 双重验证"制造权威感，但 Evolver 项目综合评分仅 4.1-4.2/10。

**修正——术语替换**：

| 旧术语 | 新术语 | 理由 |
|--------|--------|------|
| Gene | 验证规则（Validation Rule） | 金融/工程通用术语 |
| Capsule | 成功快照（Success Snapshot） | 直接描述功能 |
| Event | 审计记录（Audit Record） | 直接描述功能 |
| epigenetic_mark | 性能追踪（Performance Track） | 去掉生物学隐喻 |
| 表观遗传权重 | 自适应权重（Adaptive Weight） | 标准ML术语 |
| 进化周期 | 迭代周期（Iteration Cycle） | 去掉进化隐喻 |
| REO（研究经验对象） | 保留（此缩写已足够中性） | — |

**修正——来源声明降级**：
- "经 DeepSeek + Qwen 双重验证（综合评分 Δ0.1 高度收敛）" →
  "思想来源于 Evolver 项目分析。该分析经两轮独立审查（综合评分 4.1-4.2/10，表明来源项目质量较低）。Qwen 独立审查指出原分析存在确认偏误和过度归因（方法论可靠性 6.5/10）。本草案采纳的思想需在量化研究场景独立试跑验证后升级为正式方法论。"
- 按框架 v1.5 §9.6 的证据分类，所有 Evolver 来源声称初始标注为 `[Sp]`（推测，Speculation），试跑验证后升级。

### P3：符号错误修正

**问题**：`boost -= delta_failure` 且 `delta_failure = -0.15` → `boost -= (-0.15)` = `boost + 0.15`，失败反而加分。

**修正**：统一为 `boost += delta`，其中：
- `delta_success = +0.05`
- `delta_failure = -0.15`
- `delta_no_impact = 0.0`（原草案未定义，补上）

### P4：REO Schema 正式化 + 晋升规则

**问题**：草案只给了示例 JSON 对象，不是正式 JSON Schema。闭环（Event→Capsule→Gene）缺少晋升规则。

**修正**：
1. 定义正式 JSON Schema（`$schema`/required/enum/字段类型/minLength/pattern），提供 Python `jsonschema` 验证脚本
2. 定义 REO 注入视图（compact_view）：从完整 REO 自动生成 ≤500 字符的 prompt 注入文本
3. 定义晋升规则：
   - Event → Capsule：同一 hypothesis 的 Event ≥ 3 次 validation_passed=true → 自动提示晋升
   - Capsule → Gene：同一 Capsule 跨 ≥ 2 个项目被复用且保持 ≥ 0.7 成功率 → 自动提示晋升
   - 失败路径：Event validation_passed=false → 记录 anti-pattern，不删除
   - 冲突裁决：新 Gene 与已有 Gene 的 validation_rule 矛盾时 → 触发人工裁决告警
4. 补充字段：`schema_version`、`created_by`、`source_artifact`、`pre_registration_id`、`validation_status`、`compact_view`

### P5：存储层/注入层分离

**问题**：草案混淆了"完整存档"和"上下文注入"两个层级。500 字符上限应该约束后者，不应约束前者。

**修正**：
1. 明确定义两层：
   - **存储层**：REO/Event 完整 JSON，无字符上限，保留所有字段
   - **注入层**：REO 的 compact_view（≤500 字符），用于拼入 prompt 上下文
2. §9.7（原 §9.6）的 500 字符上限只约束注入层
3. §3.7.0 的事件日志（`events.jsonl`）属于存储层，不受 500 字符限制
4. 历史压缩（滚动窗口+统计摘要）是注入层操作——完整存储层始终保留全量历史

### P6：来源证据校准

**问题**：草案在 §9.7（原 §9.6）子规则2 中写"Evolver 论文的 4,590 次对照试验表明..."，但 Qwen/Codex 指出 4,590 更接近 retained analyses，非独立 RCT，且无 CI/p 值/多重比较校正。

**修正**：将 §9.7 子规则2 的原理陈述改为：
> "Evolver 论文（arXiv:2604.15097，beta technical report）报告的 retained analyses 暗示——在科学代码求解场景（Gemini 3.1 系列模型）下——紧凑结构化表示（~230 tokens）在 Agent 控制效果上可能优于冗长文档表示（~2,500 tokens）。但该发现受限于：单模型家族、非独立试验设计、缺乏统计推断（无 CI/p 值）、公开仓库缺实验配置和原始数据，以及核心代码混淆导致的不可审计性。因此本原则在量化研究场景下应视为待验证假设（框架 §9.6 证据等级：[Sp]），非已验证方法论。"

### P7：补充遗漏

**问题**：草案遗漏了成本效率、实施前验证清单、跨模型泛化警示。

**修正——新增内容**：
1. **成本效率提示**（加在 §9.7 或实施建议中）：Evolver 报告的 token 成本优势（约为基准的 3.1%）提示——紧凑编码可能有边际成本效益，但此数据同样受限于上述证据缺陷
2. **实施前验证清单**（加在草案末尾的实施建议中）：
   - [ ] 历史日志 dry-run（≥20 条事件后评估误报率）
   - [ ] 只观测模式试跑（≥3 个完整迭代周期）
   - [ ] 误报率和人工耗时统计
   - [ ] §3.7 退出条件对接测试
   - [ ] JSON Schema validator 就绪
   - [ ] 数值敏感性分析（delta_success/delta_failure ±50% 对抑制决策的影响）
   - [ ] 自动抑制/自动 explore 禁用开关就绪
   - [ ] 术语去隐喻化检查通过
3. **跨模型泛化警示**：Evolver 的发现限于 Gemini 3.1 系列。紧凑表示在不同模型家族（Claude/GPT/开源模型）上的效果未经检验。框架使用者在不同模型上应用本原则前应先做小规模测试。

### P8：A/B 测试设计完善

**问题**：草案的 A/B 测试设计只有 Arm A/B、任务、指标，缺少实验设计的基本要素。

**修正**：将 A/B 测试规格扩展为：
- 样本量：≥30 个独立任务（量化研究场景：≥30 个因子设计任务）
- 重复次数：每个任务 ≥3 次（控制提示词随机性）
- 模型版本固定：同一模型的同一 checkpoint
- 随机种子：固定并记录
- 数据切分：如任务有训练/测试概念，固定切分
- Prompt 控制：除编码格式外，Arm A/B 的 prompt 信息量等同（同义改写）
- 评分口径：IC 均值/IC_IR/夏普比率 + token 消耗
- 最小有意义效应：IC 均值差异 ≥0.005 且 token 节省 ≥20%
- 失败判据：Arm A（结构化）在 IC 上显著劣于 Arm B（自然语言）→ 原则降级

### P9：数值可配置化 + 敏感性分析

**问题**：草案给出的 `delta_success=+0.05`、`delta_failure=-0.15`、硬抑制阈值 `-0.3`、惯性抑制 `8次` 均来自混淆源码中的不可验证数值。

**修正**：
1. 所有数值标注为"可配置初值"，加 `[可配置]` 标记
2. 增加敏感性分析要求：主要阈值 ±50% 对告警频率的影响应在首次试跑中评估
3. 非对称惩罚比（-0.15/+0.05=3:1）的合理性标注为"来自 Evolver 的设计选择，在量化研究场景的有效性未经验证"
4. 给出调参建议：风险偏好较低的场景可增大非对称比（如 5:1），探索性场景可减小（如 2:1）

## 五、元级别记录

本次修正路线图本身是方法论建设过程的一个案例：

1. **来源质量不等于提取质量**：从低质量项目（Evolver 4.2/10）提取方法论是可行的，但需要比从高质量项目提取更严格的校准和限定
2. **独立审查的价值不在通过而在驳回**：三轮审查（DeepSeek→Qwen→Codex）的增量信息是递减的（Δ0.1→Δ?），但 Codex 驳回草案时的发现（符号错误、版本错位、边界违反）是前两轮无法提供的——因为这些是草案层面的新问题
3. **二次确认偏误的顽固性**：同一个偏误模式在同一个会话中三次出现（原报告→提示词设计→草案撰写），说明仅靠"意识到偏误"不足以消除偏误——需要结构性的检查清单（如强制引用负面限定、强制标注证据等级）
