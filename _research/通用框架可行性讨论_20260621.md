# 通用 AI 协作框架的可行性讨论

> **日期**: 2026-06-21  
> **来源**: 对框架定位的自我审查——方向 B（半开放方法论）vs 方向 C（通用框架）的选择  
> **状态**: 注记，不入主文档

## 问题

框架是否应该追求成为"通用 AI 协作框架"——独立于作者、可被任何人直接使用的标准化方法？

## 结论

**一个人写不出通用框架。** 这不是能力问题，是位置问题。

## 一个人写通用框架需要什么

### 经验广度

通用框架需要覆盖的领域远超出单人职业经验的合理范围：

- **量化/金融工程**（作者有深度经验）：已覆盖
- **学术写作**（作者有深度经验）：已覆盖
- **工具/方法论分析**（作者有深度经验）：已覆盖
- **纯软件工程**（持续交付/多分支协作/CI-CD）：作者经验有限
- **产品管理**（需求迭代/利益相关方沟通/roadmap）：作者经验有限或无
- **运营类项目**（监控/告警/on-call/SRE）：作者经验有限或无
- **强监管领域**（合规审计/数据隐私/SOX）：作者经验无

框架声称"通用"意味着对上述所有领域都有经过验证的规则——一个人的职业生涯能深度覆盖其中两三个已经很好了。

### 验证独立性

通用框架的每条声称需要独立于作者之外的验证。这需要：

- 能找到真正有项目要跑、有交付压力的独立用户
- 用户行为不被"作者在场"改变（Hawthorne 效应控制）
- 多轮、多用户、多项目类型的实证数据
- 本质上是一个用户研究/UX 研究的基础设施问题，不是方法论问题

### 多工具链知识

框架目前深度绑定 Claude Code 的能力模型（worktree/MCP/agent 子进程）。通用框架需要在至少 3-4 个不同的 AI 工具链上被验证：

- Claude Code（作者使用）：有深度经验
- GitHub Copilot agent 模式：经验有限
- Cursor compose：经验有限
- Windsurf cascade：经验有限或无
- 纯 API 编排（LangChain/AutoGen 等）：经验有限
- Qwen Code / Kimi Code / Codex CLI（部分用于审查）：有使用经验但非主力

### 学科背景

方法论设计本身是一门学科，有独立的研究传统：

- **HCI（人机交互）**：研究如何设计工具和流程让人和系统有效协作
- **认知科学**：研究人类如何形成、遵循、和偏离规则
- **组织行为学**：研究团队如何采纳、适应、和绕过流程
- **软件工程方法论**：研究流程改进的实证基础（如 CMMI 的验证历史）

这些领域的从业者花整个职业生涯研究"为什么有些流程有效而另一些无效"。一个独立研究者不可能同时在这些领域达到专家水平。

### 历史参照

那些看起来"通用"的方法论框架背后有远超个人的基础：

- **Scrum/Agile**：行业协会 + 认证体系 + 多组织多年验证
- **PRINCE2**：英国政府项目 + 国际标准组织
- **ITIL**：多轮迭代 + 全球采纳 + 培训生态
- **SDD（Spec-Driven Development）**：虽来自个人/小团队，但定位明确为"有约束的应用模式"而非"通用框架"

## 框架的实际定位选择

选择**方向 B：半开放方法论**。

- **对作者**：是个人方法论工具，继续随实践演化
- **对外部**：以半开放方式发布——欢迎参考、改编、和贡献反例
- **诚实声明**：框架提供的不是通用规则，是经过证据标注的个人实践模式；读者应预期需要翻译成本才能适配自己的场景

方向 C（通用框架）在一个人的条件下不现实。承认这一点不是弱点——框架的核心价值（七层结构/证据标注/被动观测/审查链）不因为定位从"通用"降为"半开放"而减损，只因为定位更诚实而更可信。
