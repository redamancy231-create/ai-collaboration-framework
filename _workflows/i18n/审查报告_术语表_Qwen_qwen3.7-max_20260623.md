## 4. 翻译策略一致性分析

### 4.1 不自译英文术语的划定标准

**现状**：不自译清单（附录A.3）分为五组——证据标注/层编号/闸门编号/M等级/文件名/项目工具名/操作术语。

**一致性问题**：

| 问题 | 具体表现 |
|------|---------|
| **"workflow" vs "Workflow"** | D类中"workflow"小写保留英文，但A类中"L3 Review（审查）/ Workflow（多任务编排）"首字母大写且有中文括注。同一术语的大小写和翻译处理不一致 |
| **"pipeline" vs "流水线（pipeline）"** | D类有"pipeline"（纯英文保留）和"流水线（pipeline）"（翻译+括注）两个条目。前者是通用保留词，后者是§6.2模式1的名称——但读者无法从术语表判断何时该翻译何时不翻译 |
| **"gate" vs "审查门""写回审查门"** | D类"gate"保留英文，但A类"审查门""写回审查门"翻译为中文。gate作为独立概念保留英文合理，但应在说明中注明"单独出现时保留英文，在复合词中翻译为'门'" |

**建议**：在附录A.3开头新增一条**总则**：
> 当一个英文术语单独出现（指框架层名或操作概念）时保留英文；当它作为复合中文术语的组成部分时（如"审查门""写回审查门"），使用中文翻译。

### 4.2 "pipeline"的处理不一致

- D类 `pipeline` → 三语均为"pipeline"（纯保留）
- D类 `流水线（pipeline）` → 正体"管線（pipeline）"（翻译+括注）

这两条实际上指向同一英文词的不同使用场景，但术语表未说明区分规则。**建议**：合并为一条，说明"作为通用技术术语时保留英文pipeline；作为§6.2模式1名称时翻译为'管線（pipeline）'"。

### 4.3 口号类（C类）翻译策略的贯彻一致性

C类翻译原则声明："自创概念保留'异质感'——不追求地道英语/台湾台湾中文"。

| 条目 | 异质感保留程度 | 评价 |
|------|:-------------:|------|
| 方向盘 > 发动机 | 高 | 保留隐喻结构，英文直译"Steering Wheel > Engine" ✅ |
| 30% 复杂度就能跑 | 高 | 保留口语风格 ✅ |
| 沉默省略仍是弱实践 | **中偏低** | "Silent Omission Is Still Weak Practice"读起来相当地道，不像"异质感"翻译 |
| 规则只增不减 | 中 | "Rules Only Increase, Never Decrease"偏地道但可接受 |
| 加复杂度比减复杂度容易 | 中 | 直译保留了原文的逻辑结构 ✅ |
| 幸存者偏差/伪模式化/… | **混合** | 7个失效模式合为一条。"Survivorship Bias"完全地道（因为它是既有学术术语），"Pseudo-Patterning""Byproduct Worship"则有异质感 |

**评价**：C类整体贯彻了异质感原则。"沉默省略仍是弱实践"的英文翻译是唯一偏地道的条目——但这也可以接受，因为该短语本身是作为原则/警示语存在，追求一定的可读性合理。**无需修改**。

## 5. JSON结构评估

### 5.1 当前结构

```json
{
  "metadata": { "version", "date", "model", "description", "total_terms", "type_counts" },
  "terms": {
    "<simplified_zh>": {
      "zh-Hant": "...",
      "en": "...",
      "type": "A|B|C|D|E",
      "note": "..."
    }
  }
}
```

### 5.2 优点
- 以简体中文为key，直观且与md源文件对齐
- metadata包含版本/日期/模型/计数，便于机器校验
- type字段支持按类过滤

### 5.3 结构性改进建议

| # | 问题 | 建议 |
|---|------|------|
| 1 | **key使用完整简体中文术语**——部分key过长（如"幸存者偏差/伪模式化/噪声误读/观测污染/成本失控/副产品崇拜/安全错觉"），且含斜杠的key可能在某些解析器中造成路径混淆 | 新增`id`字段（如`C-11`），以短ID为key或在value中加入`id`字段。当前md中的隐含编号可作为id来源 |
| 2 | **缺少`id`字段** | 建议每个条目增加`id`（如`A-01`, `B-03`, `D-22`），便于跨文档引用和机器检索 |

### 5.4 JSON数据完整性bug

**严重**：`"Protocol 3"` 在JSON中作为key出现两次（D类和E类各一次）。由于JSON规范不允许重复key，解析器行为未定义——大多数解析器会静默保留最后出现的值。E类的"Protocol 3 (文档)"条目的key实际是`"Protocol 3 (文档)"`（带" (文档)"后缀），因此本例中**不会**发生覆盖。但这是一个脆弱设计——如果后续有人简化key，就会触发bug。

**建议**：E类条目key保持"Protocol 3 (文档)"，但在id字段中显式标记为`E-10`（与D类的`D-xx`区分）。
